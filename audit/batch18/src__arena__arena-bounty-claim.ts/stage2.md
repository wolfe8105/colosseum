# Stage 2 Outputs — arena-bounty-claim.ts

## Agent 01

### getSelectedBountyId
Reads the module-level variable `_selectedBountyId` (a `string | null`) and returns it to the caller. No parameters, no DOM access, no storage or network access, no other module state read, no writes, no calls to other functions, no branching, no error handling, no async behavior. Single expression return; the returned reference reflects whatever the click-handler inside `renderBountyClaimDropdown` last assigned on a successful `selectBountyClaim` result, or `null` if never assigned or reset since.

### resetBountyClaim
Writes the module-level variables `_selectedBountyId` (to `null`) and `_attemptFeePaid` (to `false`) in that order. Takes no parameters, returns `void`, reads no DOM or storage, issues no network calls, invokes no other functions, contains no branches, no loops, no try/catch, and nothing async. Side-effect-only reset of the two closure-scoped flags the dropdown relies on to know whether a claim is locked.

### renderBountyClaimDropdown
Reads the four parameters `container`, `debateId`, `opponentId`, `opponentName`, writes `_attemptFeePaid = false`, then writes `container.innerHTML` with a templated wrapper that interpolates `opponentName.toUpperCase()` through `escapeHTML`. Queries the freshly-written subtree for `#bounty-claim-inner` using the non-null assertion, declares a local `bounties: OpponentBounty[]` defaulted to `[]`, and awaits `getOpponentBounties(opponentId)`. The await is wrapped in `try/catch`; on throw the catch writes a "Could not load bounties." message into `inner.innerHTML` and returns early without any rethrow or logging. On empty array it writes a "No open bounties on this opponent." message into `inner.innerHTML` and returns early. Otherwise it maps each `OpponentBounty` to an `<option>` string, computing `daysLeft` as `Math.max(0, Math.ceil((new Date(b.expires_at).getTime() - Date.now()) / 86_400_000))` (reads `Date.now()`), passes `b.bounty_id`, `String(b.attempt_fee)`, and `String(b.amount)` through `escapeHTML` for the attributes, and interpolates `Number(b.amount)` and `Number(b.attempt_fee)` into the label text. The joined options are written into `inner.innerHTML` alongside `#bounty-claim-select`, `#bounty-claim-preview`, `#bounty-claim-lock-btn`, `#bounty-claim-error`, and a hidden `#bounty-claim-locked` block with a `#bounty-claim-locked-detail` child.

After the markup write it queries `select`, `preview`, `lockBtn`, and `errEl` (all non-null-asserted) and attaches two listeners, then returns; the Promise resolves at this point while the listeners remain live. The `change` listener reads `select.selectedOptions[0]`, parses `dataset.fee` and `dataset.amount` via `Number.parseFloat` with a `'0'` fallback; if `select.value` is empty it blanks `preview.textContent`, sets `lockBtn.disabled = true`, and rewrites four inline styles (background, color, borderColor, cursor) to the muted/not-allowed state and returns early; otherwise it writes the preview string with `fee` and `Math.round(amt * 0.95 * 100) / 100`, enables the button, and writes the gold-active inline styles. The `click` listener is `async`; it returns early if `!select.value || _attemptFeePaid` (reads the module flag), hides `errEl`, re-reads `select.value` into `bountyId` and reparses `fee`/`amt` off `select.selectedOptions[0].dataset`, sets `lockBtn.disabled = true` and `lockBtn.textContent = '…'`, then awaits `selectBountyClaim(bountyId, debateId)` inside a `try`. On `result.success` it writes `_selectedBountyId = bountyId` and `_attemptFeePaid = true`, hides `select`, `preview`, and `lockBtn` via `style.display = 'none'`, queries `#bounty-claim-locked` and `#bounty-claim-locked-detail` (non-null-asserted), sets the locked block to `display: 'block'`, and writes the detail text with `fee` and `Math.round(amt * 0.95 * 100) / 100`. On `result.success` being falsy it restores `lockBtn.textContent = 'LOCK IN BOUNTY CLAIM'`, writes `errEl.textContent = result.error ?? 'Something went wrong.'`, and sets `errEl.style.display = 'block'`. There is no `catch` clause, so a rejection from `selectBountyClaim` would propagate as an unhandled rejection; the `finally` block unconditionally sets `lockBtn.disabled = false`, which re-enables the button even in the success branch where the button has already been hidden. The outer function does not await the click listener — each invocation is fire-and-forget from the DOM event dispatch perspective.

### renderBountyClaimDropdown
Async function, `Promise<void>`, awaited by callers that need the initial fetch settled. First writes `_attemptFeePaid = false` (note: `_selectedBountyId` is NOT reset here — uncertain whether that's intentional or a latent bug), then overwrites `container.innerHTML` with a heading that interpolates `escapeHTML(opponentName.toUpperCase())` plus a placeholder "Checking bounties…" shell containing `#bounty-claim-inner`. Grabs that inner element via `querySelector` with a non-null assertion, declares a local `bounties: OpponentBounty[] = []`, then enters a `try`/`catch` around `await getOpponentBounties(opponentId)`. On throw, the catch block rewrites `inner.innerHTML` to a "Could not load bounties." message and returns early — the thrown error itself is swallowed. On an empty array result, it writes a "No open bounties on this opponent." message into `inner` and returns early.

For the non-empty path, it maps each bounty into an `<option>` string: `daysLeft` is computed via `Math.max(0, Math.ceil((new Date(b.expires_at).getTime() - Date.now()) / 86_400_000))`, and `bounty_id`, `attempt_fee`, `amount` are each run through `escapeHTML(String(...))` before being placed into `value` and `data-*` attributes; the visible label uses `Number(b.amount)` and `Number(b.attempt_fee)` directly without escaping (safe because they're coerced to numbers). The joined options are injected into a second `inner.innerHTML` rewrite containing `#bounty-claim-select`, `#bounty-claim-preview`, `#bounty-claim-lock-btn` (initially disabled with muted styling), `#bounty-claim-error` (hidden), and a pre-hidden `#bounty-claim-locked` success panel. Four element handles are queried out with non-null assertions. A `click` listener on the lock button is async and fire-and-forget from the DOM event perspective. It guards with `if (!select.value || _attemptFeePaid) return` (so the second click is a no-op once the fee has been paid), hides the error element, re-reads `bountyId`, `fee`, `amt` from the current selection, disables the button, swaps its text to `'…'`, and enters a `try`/`finally` around `await selectBountyClaim(bountyId, debateId)`. The `finally` re-enables the button unconditionally — note this means after a successful claim the button is re-enabled even though it has been hidden via `display:none`, and in the failure path it's re-enabled with its restored label so the user can retry. There is no `catch` clause, so a thrown `selectBountyClaim` would propagate out of the handler as an unhandled promise rejection while still running the `finally`.

Module state variables touched: `_selectedBountyId` (written only on success path inside the click handler), `_attemptFeePaid` (written to `false` at function entry, to `true` on success).

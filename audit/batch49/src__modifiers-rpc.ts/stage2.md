# Stage 2 Outputs — modifiers-rpc.ts

## Agent 01

### buyModifier

`buyModifier` is an async function that accepts a single string parameter, `effectId`, and returns a plain object typed as `{ success: boolean; modifier_id?: string; cost?: number; error?: string }`. It reads no module-level state. It calls `safeRpc` once, awaiting the result, passing the RPC name `'buy_modifier'` and a single-key params object `{ p_effect_id: effectId }`. After the await, it branches on `result.error`: if that property is truthy, it returns immediately with `{ success: false, error: ... }`, where the error string is either `result.error.message` (if present) or a stringified fallback via `String(result.error)`. If `result.error` is falsy, the function reaches the second return statement and casts `result.data` directly to the declared return type without any runtime shape validation — a known landmine noted in the file's comment block (LM-MODS-002). The function writes nothing to module-level or external state beyond the `safeRpc` call itself, and its return value is the sole output.

### buyPowerup

`buyPowerup` is an async function that accepts `effectId: string` and an optional `quantity` parameter defaulting to `1`, and returns `{ success: boolean; new_quantity?: number; cost?: number; error?: string }`. It reads no module-level state. It calls `safeRpc` once, awaiting the result, with RPC name `'buy_powerup'` and a two-key params object `{ p_effect_id: effectId, p_quantity: quantity }`. Control flow is identical to `buyModifier`: if `result.error` is truthy, it returns `{ success: false, error: result.error.message ?? String(result.error) }` immediately; otherwise it casts and returns `result.data` as the declared type without runtime validation. The function writes nothing to module-level or external state, and its only output is its return value.

### socketModifier

`socketModifier` is an async function that accepts three parameters — `referenceId: string`, `socketIndex: number`, and `modifierId: string` — and returns `{ success: boolean; error?: string }`. It reads no module-level state. It calls `safeRpc` once, awaiting the result, with RPC name `'socket_modifier'` and a three-key params object `{ p_reference_id: referenceId, p_socket_index: socketIndex, p_modifier_id: modifierId }`. The comment notes that `socketIndex` is 0-based, so slot 1 maps to index 0. After the await, control branches on `result.error` in exactly the same pattern as the two functions above: truthy error yields an early return of `{ success: false, error: ... }`; falsy error yields a cast of `result.data`. No module-level state is read or written, and the return value is the sole output.

### equipPowerupForDebate

`equipPowerupForDebate` is an async function that accepts `debateId: string` and `effectId: string`, and returns `{ success: boolean; slots_used?: number; error?: string }`. It reads no module-level state. It calls `safeRpc` once, awaiting the result, with RPC name `'equip_powerup_for_debate'` and a two-key params object `{ p_debate_id: debateId, p_effect_id: effectId }`. The branching pattern on `result.error` is the same as all preceding functions: truthy error yields `{ success: false, error: result.error.message ?? String(result.error) }`; falsy error yields `result.data` cast to the return type without runtime validation. The `slots_used` field in the return type indicates the RPC response may carry how many equip slots are now occupied for the given debate, but the function itself does not inspect or act on that value — it merely passes it through via the cast. No module-level state is read or written.

### getUserInventory

`getUserInventory` is an async function that accepts an optional `debateId?: string` parameter and returns `UserInventory | null`. It reads no module-level state. It calls `safeRpc` once, awaiting the result, with RPC name `'get_user_modifier_inventory'` and a single-key params object `{ p_debate_id: debateId ?? null }` — when `debateId` is `undefined`, the nullish coalescing operator substitutes `null`, which is what gets sent to the RPC. The error branch differs from the other four functions in two ways: instead of returning an object with a `success` flag, it logs to `console.error` with the prefix `'[Modifiers] inventory fetch failed:'` and then returns `null`. On the happy path, it casts `result.data` as `UserInventory` (imported as a type from `./modifiers.ts`) without runtime shape validation, the same known landmine that applies to all functions in this file. The `UserInventory` type is import-only; no instance of it is held in module-level state. The function writes nothing beyond the `safeRpc` call and the potential `console.error` side effect.

## Agent 02

### buyModifier

`buyModifier` is async and accepts a single parameter, `effectId`, a string identifying the modifier effect to purchase. It reads no module-level state. It immediately awaits `safeRpc('buy_modifier', { p_effect_id: effectId })`, passing the effect ID as the sole RPC argument. `safeRpc` is the sole external call. After the await settles, the function inspects `result.error`. If that field is truthy, the function takes the error path: it returns an object literal with `success: false` and an `error` field populated by `result.error.message` if that property exists, or `String(result.error)` as a fallback. There is no `try/catch` block; error detection is entirely driven by the presence of `result.error` on the returned object. On the happy path — when `result.error` is absent — the function returns `result.data` cast as `{ success: boolean; modifier_id?: string; cost?: number; error?: string }` with no runtime shape validation (the file's own landmine comment, LM-MODS-002, flags this). The function writes no module-level or external state itself; all side effects (token deduction, modifier record creation) are delegated to the server-side `buy_modifier` RPC.

### buyPowerup

`buyPowerup` is async and accepts two parameters: `effectId` (string) and `quantity` (number, defaulting to `1`). It reads no module-level state. It awaits `safeRpc('buy_powerup', { p_effect_id: effectId, p_quantity: quantity })`, forwarding both parameters to the server RPC. After the await, it checks `result.error` using the same pattern as `buyModifier`: a truthy `result.error` causes an early return of `{ success: false, error: result.error.message ?? String(result.error) }`. On the happy path, it returns `result.data` cast as `{ success: boolean; new_quantity?: number; cost?: number; error?: string }`, again without runtime validation. The optional `new_quantity` field in the return type distinguishes this from `buyModifier`'s `modifier_id`, reflecting that powerups are consumables tracked by quantity. There is no `try/catch`; error propagation depends entirely on `safeRpc` placing error information in `result.error` rather than throwing. The function writes nothing outside its return value.

### socketModifier

`socketModifier` is async and accepts three parameters: `referenceId` (string), `socketIndex` (number, 0-based slot index), and `modifierId` (string identifying which owned modifier to place). It reads no module-level state. It awaits `safeRpc('socket_modifier', { p_reference_id: referenceId, p_socket_index: socketIndex, p_modifier_id: modifierId })`, passing all three arguments directly to the RPC. The error path mirrors the other functions: if `result.error` is truthy, it returns `{ success: false, error: result.error.message ?? String(result.error) }`. On the happy path it returns `result.data` cast as `{ success: boolean; error?: string }` — a narrower return shape than `buyModifier` or `buyPowerup`, containing no additional fields beyond the success flag and an optional error message. There is no `try/catch`, no looping, and no branching beyond the single error check. The function writes nothing outside its return value; the server-side RPC is responsible for binding the modifier to the specified socket slot on the reference record.

### equipPowerupForDebate

`equipPowerupForDebate` is async and accepts two parameters: `debateId` (string) and `effectId` (string). It reads no module-level state. It awaits `safeRpc('equip_powerup_for_debate', { p_debate_id: debateId, p_effect_id: effectId })`. After the await, if `result.error` is truthy it returns `{ success: false, error: result.error.message ?? String(result.error) }`. On the happy path it returns `result.data` cast as `{ success: boolean; slots_used?: number; error?: string }`. The optional `slots_used` field in the return type allows callers to know how many of the three allowed debate slots have been consumed after this call, without requiring a separate inventory fetch. There is no `try/catch`, no loop, and no branching beyond the error check. All inventory decrement and slot tracking logic is delegated to the server.

### getUserInventory

`getUserInventory` is async and accepts one optional parameter, `debateId` (string or undefined). It reads no module-level state. It awaits `safeRpc('get_user_modifier_inventory', { p_debate_id: debateId ?? null })`, substituting `null` for the RPC parameter when `debateId` is not supplied — this signals to the server that no debate-specific loadout should be included in the response. After the await, the error path diverges from the other four functions in two ways: it calls `console.error('[Modifiers] inventory fetch failed:', result.error)` before returning, and it returns `null` rather than an error-shape object. This makes the return type `UserInventory | null` rather than a union with a `{ success: false }` object, shifting error discrimination to a nullability check on the caller side. On the happy path, it returns `result.data` cast as `UserInventory` (imported as a type from `./modifiers.ts`), again without runtime shape validation per the module-level LM-MODS-002 comment. There is no `try/catch`, no loop, and no branching beyond the single error check.

# THE COLOSSEUM — DEFENSE MAP
### How Every Feature Is Protected
### Created: Session 27 (March 2, 2026)

> This document maps every defense mechanism to every feature and design principle. If a feature exists, its defense should be traceable here. If a defense is missing, it should be visible here.

# DEFENSE LAYER 1: INPUT SANITIZATION
**What it does:** Strips malicious content from anything a user types before it hits the database.
**Mechanisms:** sanitize_text(), sanitize_url()
**Status:** ✅ Built (Session 17)

**Features protected:**
1. Post a hot take (Feature #1) — sanitize_text on body
2. Send a message in text battle (Feature #25) — sanitize_text on message
3. Record a voice memo argument (Feature #26) — no text sanitization needed, but transcript if generated needs sanitize_text
4. Edit your profile (Feature #49) — sanitize_text on display name, bio, all freeform fields
5. Complete Profile Depth sections (Feature #50) — sanitize_text on all open-ended answers
6. Share a profile link (Feature #68) — sanitize_url on any user-submitted URLs
7. Use a referral/deep link (Feature #70) — sanitize_url on inbound link parameters
8. Report a user or content (Feature #81) — sanitize_text on report reason

**Design areas protected:**
- Area #4 (Never feel empty) — bot-seeded hot takes still pass through sanitize_text
- Area #10 (Rage engine) — auto-debate content generated by AI passes through sanitize_text before storage
- Area #9 (Reciprocal gating) — Profile Depth answers are user input, sanitized

**Not needed for:**
- Reactions (Feature #2) — predefined values, not freeform
- Votes (Features #29, #30) — boolean/enum, not text
- Follow/unfollow (Features #13, #14) — no text input
- Cosmetic equip (Feature #59) — selection from predefined list
- All OAuth/auth flows — handled by Supabase auth, not user text

**Gaps:**
- ⏳ Voice memo transcriptions (if/when Deepgram is added) need sanitize_text on output
- ⏳ AI Sparring responses come from Groq — currently trusted, but should sanitize before display as defense-in-depth

# DEFENSE LAYER 2: RATE LIMITING
**What it does:** Prevents spam, flooding, and abuse by capping how often actions can be performed.
**Mechanisms:** rate_limits table, middleware.js request throttling, per-action RPC limits
**Status:** ✅ Built (Session 17 + Session 16)

**Features protected:**
1. Post a hot take (Feature #1) — rate limit per user per time window
2. React to a hot take (Feature #2) — rate limit prevents reaction spam
3. Challenge someone to a debate (Feature #4) — rate limit prevents challenge harassment
4. Place a prediction (Feature #9) — rate limit prevents prediction manipulation
5. Follow a user (Feature #10) — rate limit prevents mass-follow spam
6. Declare a hated rival (Feature #16) — hard cap at 5 active (built-in rate limit via business rule)
7. Enter matchmaking queue (Feature #21) — rate limit prevents queue flooding
8. Send a message in text battle (Feature #25) — rate limit per round/per minute
9. Vote on a live debate (Feature #29) — one vote per user per debate
10. Vote on an auto-debate (Feature #30) — one vote per user per debate
11. Tip a debater (Feature #64) — rate limit prevents tip-spam abuse
12. Report a user (Feature #81) — rate limit prevents report abuse
13. Sign up (Feature #41-43) — rate limit on account creation per IP
14. Log in (Feature #44) — rate limit on login attempts (brute force prevention)
15. Reset password (Feature #47) — rate limit on reset requests

**Design areas protected:**
- Area #7 (Guest access) — middleware.js rate limits unauthenticated requests harder than authenticated
- Area #10 (Rage engine) — auto-debate voting rate limited so bots can't stuff ballots
- Area #11 (Zero founder marketing) — bot army has its own internal rate limits to avoid platform bans on Reddit/Twitter/Discord

**Infrastructure rate limits:**
- middleware.js — global request throttling per IP
- Supabase — connection pooling limits
- Vercel — serverless function invocation limits
- Stripe — webhook retry limits
- Groq — free tier API call limits

**Gaps:**
- ⏳ Voice memo uploads need size + frequency limits
- ⏳ Share card generation (Feature #69) could be abused for compute — needs throttle
- ⏳ Profile edit frequency should be capped (prevent name-change spam)

# DEFENSE LAYER 4: ROW LEVEL SECURITY (RLS)
**What it does:** Database-level enforcement. Even if someone bypasses the app, the database itself refuses unauthorized access.
**Mechanisms:** Supabase RLS policies on all 29 tables, hardened in Session 14
**Status:** ✅ Built (24 hardened RLS policies)

**Tables and what they protect:**
1. profiles — users can only read public fields, write only their own
2. profiles_private (view) — only visible to the profile owner
3. profiles_public (view) — safe subset visible to everyone
4. hot_takes — users can only delete their own, anyone can read
5. hot_take_reactions — users can only manage their own reactions
6. predictions — users can only see/manage their own predictions
7. follows — users can only manage their own follow relationships
8. rivals — users can only manage their own rival declarations
9. debates — read access based on participation/spectator status
10. debate_messages — scoped to debate participants
11. arena_debates — read for all, write scoped to participants
12. arena_votes — one vote per user per debate enforced at DB level
13. debate_queue — users can only see/manage their own queue entry
14. auto_debates — public read, no user write (system-generated)
15. auto_debate_votes — one vote per user per debate
16. payments — users can only see their own payment history
17. token_transactions — users can only see their own transactions
18. user_cosmetics — users can only see/manage their own
19. user_achievements — users can only see their own, public read for display
20. user_settings — users can only read/write their own
21. notifications — users can only see their own
22. reports — users can create, cannot see others' reports
23. rate_limits — system table, no direct user access
24. achievements — public read (catalog), no user write
25. cosmetics — public read (catalog), no user write
26. debate_votes — one vote per user enforced
27. async_debates — scoped to participants
28. profile_depth_answers — users can only see/write their own
29. bot activity table — no user access (bot army internal)

**Features with RLS as primary defense:**
- Every feature that reads or writes data. All 86 features ultimately hit a table protected by RLS.

**Design areas protected:**
- Area #1 (Defense) — RLS is the innermost ring. Even if middleware, auth, and sanitization all fail, RLS still blocks unauthorized data access.
- Area #8 (Real money) — payments and token_transactions locked to owner-only access
- Area #9 (Reciprocal gating) — profile_depth_answers private to user, can't be scraped

**Gaps:**
- None identified. All 29 tables have RLS. This is the most complete defense layer.

# DEFENSE LAYER 5: SERVER-SIDE VALIDATION (RPC FUNCTIONS)
**What it does:** All mutations go through SECURITY DEFINER functions. The client never writes directly to tables. Functions validate inputs, check permissions, and enforce business rules.
**Mechanisms:** 30+ SECURITY DEFINER functions, supabase.rpc() pattern
**Status:** ✅ Built (Session 17 migration, Session 23 +8 RPCs, Session 24 +10 RPCs)

**Feature-to-RPC mapping:**
1. Post a hot take → post_hot_take() — validates content, enforces rate limit, sanitizes
2. React to a hot take → react_hot_take() — toggle function, handles add AND remove
3. Delete hot take → delete_hot_take() — verifies ownership before delete
4. Place prediction → place_prediction() — validates debate exists, user hasn't already predicted
5. Follow user → follow_user() — prevents self-follow, validates target exists
6. Unfollow user → unfollow_user() — verifies relationship exists
7. Declare rival → declare_rival() — enforces 5-rival cap, prevents self-rival
8. Remove rival → remove_rival() — verifies ownership
9. Join queue → join_debate_queue() — validates mode, prevents duplicate queue entries
10. Leave queue → leave_debate_queue() — verifies user is in queue
11. Send debate message → send_debate_message() — validates debate is active, user is participant
12. Vote on arena debate → vote_arena_debate() — one vote per user, validates debate exists
13. Get arena feed → get_arena_feed() — safe read with UNION ALL ordering
14. Get lobby debates → get_lobby_debates() — filtered, safe read
15. Mark notification read → mark_notification_read() — verifies ownership
16. All scoring functions → Elo, XP, leveling are server-side only (SELECT reads for client)
17. Token transactions → server-side only, client cannot credit tokens
18. Stripe webhook processing → Edge Function validates webhook signature

**Design areas protected:**
- Area #8 (Real money) — all financial mutations are server-side. Client can't credit tokens or fake purchases.
- Area #5 (Spectators first) — voting RPCs enforce one-vote-per-user, preventing manipulation
- Area #9 (Reciprocal gating) — achievement unlocks are server-side, can't be faked

**Gaps:**
- ⏳ Profile edit may still use direct update in some places — verify all profile writes go through RPC
- ⏳ Cosmetic equip/purchase should be verified server-side (can't equip unowned cosmetics)

# DEFENSE LAYER 6: INFRASTRUCTURE HARDENING
**What it does:** Protects the platform at the network/deployment level before requests even reach the app.
**Mechanisms:** vercel.json (12 security headers), middleware.js, Stripe CORS, Edge Function CORS
**Status:** ✅ Built (Session 16, confirmed Session 21)

**What it covers:**
1. CSP (Content Security Policy) — prevents XSS by controlling what scripts can execute
2. HSTS (Strict Transport Security) — forces HTTPS
3. X-Frame-Options — prevents clickjacking
4. X-Content-Type-Options — prevents MIME sniffing
5. Referrer-Policy — controls what's sent in referer headers
6. Permissions-Policy — restricts browser API access
7. Plus 6 additional security headers
8. middleware.js — request-level rate limiting, CORS enforcement
9. Stripe Edge Function — CORS hardened, webhook signature verification
10. AI Sparring Edge Function — CORS headers on Groq proxy

**Features protected:**
- All 86 features — infrastructure hardening is the outermost ring, protects everything
- Specifically critical for:
  - Feature #64 (Tipping) — Stripe CORS prevents cross-origin payment hijacking
  - Feature #24 (AI Sparring) — Edge Function CORS prevents unauthorized Groq usage
  - Features #41-44 (Auth flows) — HTTPS enforcement via HSTS protects credentials in transit

**Design areas protected:**
- Area #7 (Guest access) — middleware treats unauthenticated traffic appropriately
- Area #11 (Zero founder marketing) — bot army runs on separate VPS, doesn't touch Vercel infrastructure
- Area #2 (Mobile forward) — CSP allows required CDN resources (fonts, libraries) while blocking unknown scripts

**Gaps:**
- ⏳ No WAF (Web Application Firewall) — Cloudflare free tier could add this
- ⏳ No DDoS protection beyond Vercel's built-in — Cloudflare would help here too
- ⏳ No geographic restrictions — probably not needed yet, but monitor for abuse patterns

# DEFENSE LAYER 7: FINANCIAL PROTECTION
**What it does:** Protects all money flows — subscriptions, tips, purchases.
**Mechanisms:** Stripe Checkout (server-initiated), webhook signature verification, Edge Functions, sandbox mode
**Status:** ✅ Built (Session 10), ⚠️ Still in sandbox mode

**Features protected:**
1. Purchase subscription (Feature #61) — Stripe Checkout session created server-side, not client-manipulable
2. Upgrade subscription (Feature #62) — server-side Stripe API call
3. Purchase token pack (Feature #63) — Stripe Checkout, server validates amount
4. Tip a debater (Feature #64) — amount validated server-side, platform cut calculated server-side
5. View transaction history (Feature #65) — payments table RLS, user sees only their own
6. Profile Depth discount (Feature #52) — discount calculation is server-side, can't be faked

**Financial attack vectors defended:**
- Price manipulation → Stripe Checkout creates session with server-defined price, client can't change it
- Fake purchase confirmation → webhook signature verification ensures only real Stripe events are processed
- Token inflation → token_transactions table writes are SECURITY DEFINER only, no client write
- Subscription spoofing → subscription status checked server-side via Stripe API, not client state
- Refund abuse → handled by Stripe's built-in fraud detection

**Design areas protected:**
- Area #8 (Real money) — this IS the defense for real money. Every dollar flows through Stripe, not through the app.
- Area #9 (Reciprocal gating) — subscription tier affects feature access, verified server-side

**Gaps:**
- ⏳ Stripe still in sandbox mode — no real money has ever flowed. Switch to production is a pending human action.
- ⏳ Stripe Connect for creator payouts not set up yet — needed when tipping goes live
- ⏳ No fraud monitoring dashboard — rely on Stripe's built-in for now
- ⏳ No refund policy defined
- ⏳ Tip recipient payout mechanism not built (how do debaters cash out?)

**Features protected:**

3. Place prediction (Feature #9) — one prediction per user per debate
4. View Elo rating (Feature #33) — Elo calculated server-side only, client is read-only

8. Earn achievements (Feature #54) — server-side unlock logic

# DEFENSE LAYER 11: BOT ARMY SELF-DEFENSE
**What it does:** Protects the bot army itself from getting banned, detected, or causing harm to the main platform.
**Mechanisms:** DRY_RUN mode, internal rate limits, dedicated accounts, separate VPS
**Status:** ✅ Built (code complete, not deployed)

**What it protects:**
1. Bot Reddit account — internal rate limits prevent Reddit ban for spam
2. Bot Twitter account — posting frequency tuned to avoid Twitter spam detection
3. Bot Discord account — activity pacing to avoid server bans
4. Main platform reputation — bots post from dedicated accounts, not linked to founder
5. Supabase database — bot_activity table is separate, bot writes go through dedicated schema
6. Auto-debate quality — AI generation uses Groq with tuned prompts, not random garbage

**Design areas protected:**
- Area #11 (Zero founder marketing) — if bots get banned, founder doesn't lose personal accounts
- Area #10 (Rage engine) — DRY_RUN mode lets you test rage-bait without actually posting it
- Area #4 (Never feel empty) — bot army IS the content seeding strategy, so protecting the bots protects the content pipeline

**Gaps:**
- ⏳ No automatic account rotation if a bot account gets banned
- ⏳ No monitoring alerts if bot activity drops to zero (PM2 crash unnoticed)
- ⏳ No content review pipeline — AI-generated rage-bait posts without human review (by design, but risky)
- ⏳ Reddit/Twitter TOS compliance — bot posting may violate platform rules. Accepted risk.
- ⏳ No IP rotation on VPS — single IP could get flagged

# DEFENSE LAYER 12: AUTH SYSTEM RESILIENCE
**What it does:** Ensures the auth system never blocks the app from loading or creates a bad experience.
**Mechanisms:** readyPromise pattern, 3s _checkSession timeout, 4s appInit Promise.race, graceful fallback to guest
**Status:** ✅ Built (Session 23 + Session 26)

**Features protected:**
- All 86 features — if auth hangs, the app still loads as guest mode
- Specifically critical for:
  - Feature #77 (Browse as guest) — the whole point. Auth failure = guest, not broken app.
  - Feature #78 (Spoke carousel) — home screen loads regardless of auth state
  - Features #41-44 (Auth flows) — login/signup work independently of main app init

**Design areas protected:**
- Area #7 (Guest access) — auth timeout IS the guest access defense. Supabase hangs? You're a guest. Network drops? You're a guest. Navigator lock orphaned? You're a guest.
- Area #3 (Ease of use) — user never sees an infinite spinner or unexplained failure
- Area #4 (Never feel empty) — app always loads with content visible, even if auth is broken

**Gaps:**
- ⏳ No offline mode — if Supabase is completely down, app loads but all data fetches fail
- ⏳ No service worker for caching — could serve cached content during outages
- ⏳ Auth token refresh — what happens when a session expires mid-use? Test this.

# CROSS-CUTTING: HOW DEFENSE MAPS TO AREAS #2-#12

## Area #2 (Mobile forward)
- CSP allows required CDN resources for mobile fonts/libraries
- Touch targets don't create security issues — no defense needed
- Deep links (Feature #70) — sanitize_url on inbound parameters

## Area #3 (Ease of use)
- Auth timeout prevents confusion (no infinite spinners)
- Error messages don't leak internal state or stack traces
- Feature flags hide unfinished features (prevents users hitting broken endpoints)

## Area #4 (Never feel empty)
- Auto-debate content passes through sanitize_text
- AI Sparring Edge Function has CORS protection
- Bot-seeded content goes through same RPC validation as user content
- Groq API key stored as Supabase secret, not exposed to client

## Area #5 (Spectators first)
- Vote integrity (one vote per user) protects the spectator experience
- Leaderboard data is server-calculated, can't be faked
- Public views (profiles_public) protect privacy while enabling spectator features

## Area #6 (Casual energy)
- Protected lobbies depend on accurate Elo — server-side Elo calculation prevents gaming
- Content moderation protects casual atmosphere from toxic users
- Rate limiting prevents spam in casual spaces

## Area #7 (Guest access)
- Auth gating split is the defense — clear boundary between see and act
- Middleware rate limits guests more aggressively than authenticated users
- Guest state can't write to database (no RPC calls succeed without auth)

## Area #8 (Real money)
- All financial flows through Stripe (server-initiated)
- Webhook signature verification
- Token transactions server-side only
- Payment history private to owner (RLS)

## Area #9 (Reciprocal gating)
- Achievement unlocks server-side (can't fake)
- Profile Depth completion tracked server-side
- Subscription tier verified server-side
- Cosmetic ownership verified before equip

## Area #10 (Rage engine)
- Auto-debate votes currently ungated — accepted tradeoff
- AI content generation through Groq with tuned prompts (not user-manipulable)
- Bot army posts sanitized before storage
- Rage-bait topics curated by AI, not user-submitted

## Area #11 (Zero founder marketing)
- Bot army on separate VPS (infrastructure isolation)
- Dedicated bot accounts (identity isolation)
- DRY_RUN mode (testing isolation)
- Internal rate limits (platform compliance)

## Area #12 (Ship or die)
- Defense shouldn't block shipping. Current defense posture is "good enough to launch."
- Gaps identified above are prioritized: must-fix-before-users vs fix-when-needed vs future
- No defense gap is currently blocking bot army deployment

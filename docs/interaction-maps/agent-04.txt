COMPREHENSIVE ANALYSIS OF src/arena/arena-core.ts
=================================================

This file is the orchestration hub of the arena subsystem. It serves four roles: it provides shared utility functions consumed by sibling arena modules, it manages browser history integration so the back button works correctly within the single-page arena experience, it exposes the init and destroy lifecycle hooks that start and stop the entire arena feature, and it wires up auto-initialization so the arena boots as soon as auth is ready. The file is 161 lines and was produced by splitting apart a monolith called arena.ts; a comment at line 2 acknowledges this lineage.

The import block (lines 4 through 31) pulls from nine distinct modules across two directories. From auth.ts it takes safeRpc, which is a wrapper around Supabase RPC calls that automatically retries on 401 (expired JWT) errors; getSupabaseClient, which returns the live SupabaseClient instance or null if credentials are placeholders; getCurrentUser and getCurrentProfile, which return the cached Supabase auth User and the application-level Profile row respectively; and ready, which is a Promise<void> that resolves once the auth subsystem has fully initialized (either successfully or after a 6-second timeout). From config.ts it imports isAnyPlaceholder, a boolean that is true when either the Supabase URL/key or the Stripe key is a placeholder string (meaning the app is in preview/demo mode), and FEATURES, a frozen object of feature flags including FEATURES.arena which gates whether the arena initializes at all and FEATURES.liveDebates which gates matchmaking queue entry. From webrtc.ts it imports leaveDebate, a function that tears down WebRTC peer connections, stops the worker timer, and terminates the audio pipeline. From arena-state.ts it imports a large set of mutable state variables and their corresponding setter functions; ES module live bindings mean that reading the imported variable always reflects the current value set by the owning module, but writes must go through the setter functions because imported let bindings are read-only from the importing side. The specific state variables imported are: view (the current ArenaView string, one of lobby, modeSelect, queue, matchFound, room, preDebate, postDebate, privateLobbyWaiting, modQueue, modDebatePicker, or modDebateWaiting), currentDebate (the active CurrentDebate object or null), roundTimer (a setInterval handle for the debate round countdown), _rulingCountdownTimer (a setInterval handle for the moderator ruling countdown overlay), matchFoundDebate (the CurrentDebate snapshot captured when a match is found), screenEl (the #screen-arena DOM element), queueElapsedTimer (interval tracking how long the user has been in queue), queuePollTimer (interval polling the server for match status), vmTimer (voice memo recording timer), and referencePollTimer (interval polling for new reference submissions). Each of these has a corresponding set_ prefixed function that mutates the canonical variable inside arena-state.ts. The resetState function, also from arena-state.ts, clears every single timer and resets every mutable variable to its default value in one sweep. From arena-types.ts it imports the ArenaView and CurrentDebate types purely for type annotations; no runtime code comes from this import. From arena-queue.ts it imports clearQueueTimers, which clears both queuePollTimer and queueElapsedTimer. From arena-match.ts it imports clearMatchAcceptTimers, which clears the match accept countdown timer and the match accept poll timer. From arena-room-live.ts it imports stopOpponentPoll, which clears the opponent presence polling interval and resets the elapsed counter. From arena-mod-refs.ts it imports stopReferencePoll, which clears the reference poll interval and empties the pendingReferences array. From arena-mod-queue.ts it imports stopModStatusPoll, which clears the moderator status polling interval. From arena-lobby.ts it imports renderLobby, which resets the view to lobby, clears the selected mode, and renders the lobby HTML into screenEl, and showPowerUpShop, which navigates to the power-up shop view. From arena-private-lobby.ts it imports joinWithCode, an async function that takes a 6-character join code, calls the join_private_lobby RPC, and transitions the user into the private lobby waiting state or directly into the debate room. From arena-css.ts it imports injectCSS, which appends a <style> element to the document head containing all arena-specific CSS, but only once (guarded by the cssInjected boolean in state). From voicememo.ts it imports cleanupPendingRecording, which revokes any ObjectURL created for a pending voice memo recording and nulls out the reference, preventing memory leaks. From arena-feed-room.ts it imports cleanupFeedRoom, which notifies the opponent of disconnection via the Supabase realtime channel, unsubscribes from that channel, clears the feed turn timer, and resets feed room state, and enterFeedRoomAsSpectator, which calls the get_arena_debate_spectator RPC and transitions the UI into spectator mode for a given debate.

The utility helpers section (lines 36 through 48) defines three small exported functions. isPlaceholder returns true if getSupabaseClient() returns null (meaning Supabase was never initialized) or if isAnyPlaceholder is true (meaning placeholder credentials are configured); this is used throughout the arena modules to gate server calls and show "preview mode" messages instead. formatTimer takes a number of seconds and returns a string in M:SS format, padding the seconds with a leading zero when below 10; this is used in queue elapsed displays, round countdown timers, and voice memo recording timers. randomFrom is a generic function that takes a readonly array and returns a random element using Math.floor(Math.random() * arr.length) with a non-null assertion; this is used to pick random AI debate topics and random AI response templates.

The browser history section (lines 54 through 90) implements the back-button behavior for the arena's single-page navigation. pushArenaState takes a view name string and calls history.pushState with an object containing arenaView set to that name and an empty title; this is called every time the arena transitions to a new view (entering queue, finding a match, entering a debate room, etc.) so that each transition creates a new browser history entry. The _onPopState constant is an arrow function that serves as the popstate event handler. When the user presses the browser back button, this function first removes any overlay elements that might be floating in the DOM: the rank overlay (arena-rank-overlay), the ruleset overlay (arena-ruleset-overlay), the mode overlay (arena-mode-overlay), and the moderator ruling overlay (mod-ruling-overlay). For the ruling overlay specifically, it also clears the _rulingCountdownTimer interval before removing the element, because that overlay has an active countdown that would otherwise leak. Then it checks the current view state. If the user is in the room or preDebate view, it performs a full cleanup: clearing the round timer interval, clearing the ruling countdown timer interval, stopping the reference poll, stopping the opponent presence poll, stopping the moderator status poll, and if the current debate mode is live, cleaning up the feed room (which sends a disconnect notification to the opponent) and leaving the WebRTC debate (which tears down audio). If the user is in the queue view, it clears the queue timers and, unless the app is in placeholder mode, calls the leave_debate_queue RPC to notify the server that the user has left the matchmaking queue; this RPC call is fire-and-forget with only a console.warn on failure. If the user is in the matchFound view, it clears the match accept timers and nulls out the matchFoundDebate reference. Finally, if the current view is anything other than lobby, it calls renderLobby to return the user to the arena lobby. The handler is registered at line 90 as a global popstate event listener via window.addEventListener, which is a module-level side effect that runs when the module is first imported.

The init function (lines 96 through 121) is the arena's entry point. It first checks the FEATURES.arena flag and returns immediately if the arena feature is disabled, making the entire arena a no-op. If enabled, it calls injectCSS to append the arena stylesheet to the document (idempotent due to the cssInjected guard). It then grabs the #screen-arena DOM element via getElementById and stores it via set_screenEl; if the element is not found, it logs a warning and returns, preventing any further initialization. With the screen element available, it calls renderLobby to populate the arena with the lobby UI. After rendering the lobby, it checks three URL query parameters for auto-navigation features. If shop=1 is present, it immediately calls showPowerUpShop to open the power-up shop overlay, which was presumably linked from elsewhere in the app or from an external campaign. If joinCode=XXXXXX is present (the F-39 challenge link feature), it strips the query parameter from the URL via history.replaceState to prevent re-triggering on refresh, uppercases the code, and calls joinWithCode as a void-returning fire-and-forget promise; this allows users to share challenge links that auto-join a private lobby. If spectate=<uuid> is present (Session 240 auto-spectate feature), it validates the UUID against a strict regex pattern requiring the standard 8-4-4-4-12 hex format, strips the parameter from the URL, and calls enterFeedRoomAsSpectator with the debate ID; this allows spectate links to be shared.

The public accessors section (lines 127 through 133) exports two getter functions. getView returns the current ArenaView string, providing read access to the view state variable without exposing the mutable binding. getCurrentDebate returns either a shallow copy of the currentDebate object (via spread) or null; the shallow copy prevents callers from accidentally mutating the canonical state, though it is only a one-level-deep copy so nested objects like the messages array are still shared references.

The destroy function (lines 139 through 155) is the arena's teardown path, called when the user navigates away from the arena page. It first checks if there is an active live debate and, if so, calls cleanupFeedRoom (which sends a disconnect event to the opponent and unsubscribes from the realtime channel) followed by leaveDebate (which tears down WebRTC). It then calls cleanupPendingRecording to revoke any ObjectURL from an in-progress voice memo. It calls resetState, which is the nuclear option: it clears every single interval timer in the arena state (queue poll, queue elapsed, match accept, match accept poll, round, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence, ruling countdown, voice memo, opponent poll, feed turn, challenge ruling) and resets every mutable variable to its default value. Finally, it removes the popstate event listener that was registered at module load time, preventing the arena's back-button handler from firing after the arena has been destroyed.

The auto-init section (line 161) is the final module-level side effect. It calls ready.then(() => init()).catch(() => init()), which means init runs after the auth promise resolves regardless of whether auth succeeded or failed. This ensures the arena initializes even if auth times out (after the 6-second safety timeout) or if the user is a guest. The catch branch calling init is important because it means auth failure does not prevent arena rendering; the arena will simply operate in guest/placeholder mode with server calls gated by isPlaceholder checks.

In terms of side effects that execute at import time, there are exactly two. First, at line 90, the popstate event listener is registered on the window object. This means that merely importing arena-core.ts (even without calling init) will install a back-button handler that can manipulate arena state and DOM elements. Second, at line 161, the auto-init chain is kicked off, which will eventually call init and render the lobby UI. These two side effects mean that arena-core.ts is not a passive utility module; importing it starts the arena.

From the user's perspective, this file controls the following visible behaviors. When the arena page loads, the lobby appears with mode selection cards, recent debates, and navigation options. If the URL contains shop=1, the power-up shop opens automatically on top of the lobby. If the URL contains joinCode=ABCDEF, the user is silently redirected into a private lobby to join a challenge match. If the URL contains spectate=<uuid>, the user enters spectator mode for an ongoing debate. When the user presses the browser back button from any arena sub-view (queue, match found, debate room, etc.), they are returned to the lobby with all timers, polls, and connections properly cleaned up. If the user was in an active debate, their opponent is notified of the disconnection. When the user navigates away from the arena entirely, all resources are released: WebRTC connections are torn down, realtime subscriptions are removed, voice memo ObjectURLs are revoked, and every polling interval is cleared.

There are several risks and potential bugs worth noting. First, the shallow copy in getCurrentDebate (line 132) uses object spread, which does not deep-clone the messages array or any nested metadata objects. If a caller pushes to the returned object's messages array, that push will not affect the canonical state, but if a caller mutates an existing message object within the array (e.g., changing message.text), that mutation will propagate to the canonical state because the message objects are shared references. This is a subtle shared-mutable-state risk, though in practice it may never be triggered depending on how callers use the returned object. Second, in the _onPopState handler at line 64, the code does clearInterval(_rulingCountdownTimer!) with a non-null assertion on a value that could genuinely be null (if no ruling is in progress). The clearInterval function tolerates null/undefined arguments in all browsers, so this will not throw at runtime, but the non-null assertion is technically incorrect and could mask a real null-safety issue if the surrounding code is refactored. Third, the _onPopState handler checks for view === 'room' || view === 'preDebate' to decide whether to clean up debate resources, but the destroy function checks currentDebate?.mode === 'live' instead. This means the popstate handler will clean up room/preDebate state for all modes (live, text, voicememo, ai), but will only call cleanupFeedRoom and leaveDebate for live mode debates, which is correct since those are WebRTC-specific. However, the destroy function only cleans up live-mode debates and does not stop other polls (stopOpponentPoll, stopReferencePoll, stopModStatusPoll) individually; it relies on resetState to clear those timers. If resetState ever fails to clear a timer that was started by a non-live mode, it would leak. Currently resetState is comprehensive so this is not an active bug, but the asymmetry between popstate cleanup and destroy cleanup is a maintenance risk. Fourth, the leave_debate_queue RPC call in _onPopState (line 80) uses .catch with only a console.warn, meaning a network failure will silently leave the user in the server-side queue even though the client has moved on. This could cause phantom queue entries that eventually time out server-side, but if the server does not have its own expiry logic, it could lead to ghost matches. Fifth, the auto-init at line 161 calls init in both the then and catch branches of the ready promise, but init is not idempotent in the strict sense: it calls renderLobby, which overwrites screenEl.innerHTML, and it checks URL parameters that may trigger side effects like joinWithCode. If by some extraordinary circumstance both the then and catch callbacks fired (which should not happen with a well-behaved promise, but has been observed in some edge cases with polyfills), init would run twice, potentially double-rendering the lobby or double-joining a challenge code. Sixth, the spectate UUID validation regex at line 117 uses the i flag for case-insensitivity and requires the full 8-4-4-4-12 format, which is correct, but it does not validate UUID version or variant bits. This is fine for a routing check but means any 32-hex-digit string in the right format will be accepted, even if it does not correspond to a real debate. The enterFeedRoomAsSpectator function handles this gracefully by showing a toast on error, so the user impact is minimal. Seventh, there is no cleanup of the URL for the shop=1 parameter the way there is for joinCode and spectate. This means if a user lands on the arena page with ?shop=1, refreshes the page, and the shop has been closed, the shop will reopen on every refresh. The joinCode and spectate parameters are cleaned via history.replaceState, but the shop parameter persists. This is a minor UX annoyance rather than a bug. Eighth, the popstate handler is registered unconditionally at module load time (line 90), before init is ever called and before FEATURES.arena is checked. If FEATURES.arena is false, init returns early and the arena never renders, but the popstate handler is still active on the window. It would attempt to read view (which would be 'lobby', the default), and since view is not anything other than 'lobby', the final check at line 88 would not call renderLobby, so it would effectively be a no-op. However, it still runs the overlay removal code at lines 60-64, which could interfere with other parts of the app if they happen to use element IDs like arena-rank-overlay or arena-mode-overlay. This is unlikely but represents a minor encapsulation leak. The destroy function does remove this listener, but destroy would need to be called explicitly; if the arena feature is disabled, nothing calls destroy, so the listener persists for the lifetime of the page.

This file is a comprehensive prose analysis of src/arena/arena-core.ts, covering every line, every import, every function call destination, all side effects, user-visible behavior, and bugs or risks.

The file opens with a comment declaring itself "arena-core.ts -- init, destroy, utilities, browser history" and noting it is part of a monolith split from a formerly single arena.ts file. This immediately tells us the arena subsystem was once one enormous file that got decomposed into many smaller modules, and arena-core.ts is the orchestration hub: it handles bootstrap, teardown, browser back-button behavior, and small shared utility functions that the other arena modules depend on.

The first import block pulls safeRpc, getSupabaseClient, getCurrentUser, and getCurrentProfile from ../auth.ts. safeRpc is the project's universal wrapper for Supabase RPC calls -- it automatically retries on 401 errors (expired JWTs) by refreshing the session and replaying the call. getSupabaseClient returns the initialized SupabaseClient singleton or null if credentials are placeholder values. getCurrentUser returns the Supabase auth user object (or null for anonymous visitors), and getCurrentProfile returns the locally cached profile row. In this file, safeRpc is only used inside the popstate handler to call leave_debate_queue, getSupabaseClient is used in the isPlaceholder utility, and getCurrentUser and getCurrentProfile are imported but never directly referenced in this file -- they are re-exports or carried along for module coherence (though technically they are dead imports here, which is a minor code smell but not a bug since tree-shaking will remove them in production).

The next import brings isAnyPlaceholder and FEATURES from ../config.ts. isAnyPlaceholder is a boolean that is true when either the Supabase URL/key or the Stripe key is a placeholder string (meaning the app is running in demo/preview mode without real backend credentials). FEATURES is a feature-flag object with boolean properties like arena, liveDebates, asyncDebates, etc. In this file, FEATURES.arena gates the entire init function -- if the arena feature is disabled, init returns immediately and nothing happens. isAnyPlaceholder feeds into the isPlaceholder utility function.

leaveDebate is imported from ../webrtc.ts. This function tears down a live WebRTC audio debate connection: it stops the Web Worker timer that manages turn-based speaking, terminates the worker entirely, resets ICE restart counters, closes the RTCPeerConnection, stops all local media tracks (microphone), unsubscribes from the Supabase Realtime signaling channel, and resets internal WebRTC state variables. It is called in two places in arena-core.ts: once in the popstate handler when the user navigates back while in a live debate room, and once in the destroy function for the same purpose.

ready is imported from ../auth.ts (a second import line from the same module, which is slightly unusual but valid). ready is a Promise<void> that resolves once the auth subsystem has determined the user's authentication state -- either they have a session (logged in) or they do not (anonymous guest). This promise is the sole trigger for arena initialization at the bottom of the file.

The next large import block pulls from ./arena-state.ts, which is the centralized mutable state store for the entire arena subsystem. Because ES module let bindings are read-only from importing modules (you can read them via live bindings but cannot assign to them), the state module exports both the variables themselves and setter functions for each one. This file imports view (the current ArenaView string, one of 'lobby', 'modeSelect', 'queue', 'matchFound', 'room', 'preDebate', 'postDebate', 'privateLobbyWaiting', 'modQueue', 'modDebatePicker', 'modDebateWaiting'), currentDebate (the CurrentDebate object describing the active debate the user is in, or null), roundTimer (a setInterval handle for the round countdown clock), _rulingCountdownTimer (a setInterval handle for the moderator ruling countdown overlay), matchFoundDebate (the CurrentDebate snapshot shown in the match-accept screen), screenEl (a reference to the #screen-arena DOM element that acts as the arena's rendering container), queueElapsedTimer (the interval counting seconds spent in queue), queuePollTimer (the interval polling the server for a match), vmTimer (voice memo recording timer), and referencePollTimer (the interval polling for new reference submissions/rulings). It also imports setter functions for matchFoundDebate, screenEl, queueElapsedTimer, queuePollTimer, roundTimer, vmTimer, referencePollTimer, and _rulingCountdownTimer, plus the resetState function. resetState is the nuclear option: it clears every single interval timer in the arena subsystem (there are over a dozen), nulls out every mutable reference, and resets all boolean flags and numeric counters to their defaults.

ArenaView and CurrentDebate are imported as types from ./arena-types.ts for TypeScript type annotations only; they produce no runtime code.

clearQueueTimers is imported from ./arena-queue.ts. It clears the two queue-related intervals (queuePollTimer and queueElapsedTimer) and nulls them out. clearMatchAcceptTimers from ./arena-match.ts clears the match-accept countdown timer and the match-accept poll timer. stopOpponentPoll from ./arena-room-live.ts clears the opponent connection poll interval and resets its elapsed counter. stopReferencePoll from ./arena-mod-refs.ts clears the reference poll interval and empties the pending references array. stopModStatusPoll from ./arena-mod-queue.ts clears the moderator status poll interval. Each of these is a small cleanup function that prevents interval leaks.

renderLobby from ./arena-lobby.ts rebuilds the entire lobby UI inside #screen-arena -- the mode selection cards, the live debate feed, pending challenges, the moderator toggle, etc. showPowerUpShop from the same module opens the power-up shop overlay. joinWithCode from ./arena-private-lobby.ts handles joining a private debate lobby via a 6-character alphanumeric join code. injectCSS from ./arena-css.ts creates a <style> element in the document head containing all arena-specific CSS rules; it uses a cssInjected boolean guard so it only runs once per page load. cleanupPendingRecording from ../voicememo.ts revokes any ObjectURL created for a pending voice memo recording blob and cancels any active MediaRecorder session. cleanupFeedRoom from ./arena-feed-room.ts sends a goodbye message to the opponent via the Supabase Realtime channel (so they know you left rather than waiting 30 seconds for a heartbeat timeout), clears the feed turn timer, unsubscribes from the realtime channel, and resets all feed room state. enterFeedRoomAsSpectator from the same module calls the get_arena_debate_spectator RPC to load debate data and drops the user into a read-only spectator view of an ongoing live moderated debate.

Now the actual executable code begins. The isPlaceholder function returns true if getSupabaseClient() returns null (meaning Supabase was never initialized, typically because credentials are placeholders) OR if isAnyPlaceholder is true (meaning any key is a placeholder string). This is used throughout the arena codebase as a guard: if the app is in placeholder/demo mode, all server-side operations are skipped and the user gets demo UI only. There is a subtle redundancy here -- if Supabase credentials are placeholder, getSupabaseClient() already returns null, so isAnyPlaceholder being true for the Supabase case is doubly covered. However, isAnyPlaceholder also catches the Stripe-only placeholder case where Supabase is real but Stripe is not, which would not cause getSupabaseClient() to return null. So the OR is intentional and correct.

formatTimer takes a number of seconds and returns a "M:SS" formatted string. It computes minutes via Math.floor(sec / 60) and seconds via sec % 60, padding with a leading zero if under 10. This is used across the arena for displaying round timers, queue wait times, and voice memo durations. There is no guard against negative values -- if sec is negative, it would produce something like "-1:0-5" which is nonsensical, but in practice the callers clamp to zero before display.

randomFrom is a generic function that picks a random element from a readonly array using Math.floor(Math.random() * arr.length). The non-null assertion (!) at the end assumes the array is never empty, which would cause an undefined return if violated. In practice, the only caller passes AI_TOPICS (10 items) or AI_RESPONSES sub-arrays (3 items each), so this is safe, but it is technically an unchecked precondition.

pushArenaState pushes a new entry onto the browser history stack with a state object containing { arenaView: viewName }. This enables the back button: when the user navigates from lobby to mode select to queue, each transition pushes a history entry, so pressing back triggers a popstate event that the handler below can catch. The second argument '' is the unused title parameter of history.pushState.

_onPopState is the popstate event handler, exported as a named const (arrow function) so it can be removed by reference in destroy(). When the browser's back button fires, this handler first cleans up any overlay elements that might be floating in the DOM: it removes #arena-rank-overlay, #arena-ruleset-overlay, and #arena-mode-overlay by ID if they exist. It also removes #mod-ruling-overlay, and if that overlay exists, it clears the _rulingCountdownTimer interval first (the countdown that ticks down the seconds a moderator has to submit a ruling). The null assertion on _rulingCountdownTimer (the !) is slightly risky -- if the overlay element exists but the timer was never started or was already cleared, clearInterval(null!) would be clearInterval(undefined) which is harmless in browsers (clearInterval tolerates non-numeric and undefined arguments), so this is not a real bug but is sloppy.

Next the handler checks the current view. If the user was in the 'room' or 'preDebate' view (meaning they were in an active debate or the pre-debate setup screen), it clears the roundTimer interval, clears the _rulingCountdownTimer, calls stopReferencePoll() to stop polling for reference submissions, calls stopOpponentPoll() to stop checking whether the opponent is still connected, calls stopModStatusPoll() to stop checking whether a moderator has been assigned, and then if the debate mode was 'live', it calls cleanupFeedRoom() (which sends the goodbye, clears the feed timer, and unsubscribes from realtime) and leaveDebate() (which tears down WebRTC). This is the critical cleanup path that prevents resource leaks when a user presses back during a live debate. The user-visible effect is that the audio connection drops, the opponent sees a disconnect event in their feed, and all timers stop.

If the user was in the 'queue' view, the handler calls clearQueueTimers() to stop the elapsed-time counter and the match-polling interval, then (if not in placeholder mode) fires a fire-and-forget safeRpc('leave_debate_queue') call to tell the server the user is no longer waiting for a match. The .catch() swallows errors with a console.warn, which is appropriate since the user is already leaving and there is nothing useful to show them.

If the user was in the 'matchFound' view, the handler calls clearMatchAcceptTimers() to stop the accept/decline countdown and the poll checking whether the opponent accepted, then nulls out matchFoundDebate. The user-visible effect is that the match acceptance screen disappears and the user returns to the lobby.

Finally, if the current view is anything other than 'lobby', the handler calls renderLobby() to rebuild the lobby UI. This means all back navigation converges on the lobby -- there is no multi-level back stack. Whether you were in queue, match found, room, or any other view, pressing back always takes you to the lobby. This is a deliberate design choice that simplifies navigation at the cost of not being able to go back to, say, mode selection from the queue.

Line 90 registers _onPopState as a listener on the window's popstate event. This is a module-level side effect: as soon as this module is imported and executed (which happens at page load due to the ES module import graph), the popstate listener is attached. This is important -- it means the handler is active even before init() is called, which could theoretically cause issues if a popstate fires during the auth-ready wait. In practice this is fine because view defaults to 'lobby' and the guard "if (view !== 'lobby') renderLobby()" would be a no-op.

The init function is the main entry point. It first checks FEATURES.arena and returns immediately if the arena feature is disabled, making this entire subsystem silently inactive. It then calls injectCSS() which creates a <style> element in the document head with all arena CSS rules (guarded by cssInjected so it only happens once). It calls set_screenEl(document.getElementById('screen-arena')) to grab the arena container element from the DOM and store it in the shared state. If screenEl is null after this (meaning the HTML page does not have a #screen-arena element), it logs a warning and returns -- the arena cannot render without its container. Otherwise it calls renderLobby() which populates #screen-arena with the lobby UI: mode selection cards (Live, Voice Memo, Text Battle, AI Sparring), the live debate feed showing recent/ongoing debates, pending challenge notifications, and the moderator availability toggle.

After rendering the lobby, init checks the URL query parameters for three special cases. First, if ?shop=1 is present, it calls showPowerUpShop() to immediately open the power-up shop overlay on top of the lobby. This lets external links deep-link directly to the shop. Second, if ?joinCode=XXXXXX is present, it strips the query parameter from the URL (via history.replaceState to avoid a reload), uppercases the code, and calls joinWithCode() to attempt joining a private debate lobby. joinWithCode calls the join_private_lobby RPC on the server, and if successful, transitions the user into the private lobby waiting screen or directly into the debate if the opponent is already there. The void keyword discards the promise result since the function handles its own errors internally. Third, if ?spectate=<uuid> is present and the value passes a UUID regex validation (the regex checks for the standard 8-4-4-4-12 hex format, case-insensitive), it strips the query parameter and calls enterFeedRoomAsSpectator() which loads the debate via RPC and puts the user into a read-only spectator view of the live debate feed.

There is a noteworthy risk in the spectate UUID validation: the regex uses the 'i' flag for case-insensitivity, which is correct, and it anchors with ^ and $, which is also correct. However, it does not validate the UUID version nibble (the 13th character should be 1-5 for standard UUIDs), so technically any hex string in the right format passes. This is a non-issue in practice because the server-side RPC will reject invalid debate IDs anyway.

The getView function is a simple accessor that returns the current view value from arena-state.ts. It is exported for use by other modules that need to know what screen the arena is showing without importing from arena-state directly.

getCurrentDebate (the local function, not to be confused with the auth.ts import) returns a shallow copy of the currentDebate object via spread syntax, or null if there is no active debate. The spread copy prevents callers from mutating the shared state object, which is a good defensive pattern. However, this is only a shallow copy -- if CurrentDebate had nested mutable objects (and it does: the messages array), callers could still mutate those nested structures. In practice, the messages array is only ever replaced wholesale rather than mutated in place, so this is acceptable.

The destroy function is the teardown counterpart to init. It first checks if there is an active live-mode debate and, if so, calls cleanupFeedRoom() (send goodbye, clear feed timer, unsubscribe realtime channel) and leaveDebate() (tear down WebRTC). It then calls cleanupPendingRecording() to revoke any ObjectURL for a pending voice memo recording and cancel any active MediaRecorder, preventing memory leaks from unreleased blob URLs. It then calls resetState() which is the comprehensive nuclear cleanup: it clears every single setInterval in the arena (queue poll, queue elapsed, match accept, match accept poll, round timer, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence timer, ruling countdown, voice memo timer, opponent poll, feed turn timer, challenge ruling timer) and resets all mutable state to defaults. Finally, it removes the _onPopState listener from the window's popstate event, which is essential to prevent the handler from firing after the arena module has been destroyed (e.g., if the user navigates to a completely different section of the app).

There is a subtle issue with destroy: it calls cleanupFeedRoom() and leaveDebate() conditionally (only for live mode debates), but resetState() also clears the feedTurnTimer and feedRealtimeChannel. If the debate mode was 'live', the feed room cleanup happens twice -- once explicitly here and once inside resetState(). The explicit cleanupFeedRoom() call sends the goodbye message and properly unsubscribes the realtime channel, while resetState() just nulls the timer and channel references. Since cleanupFeedRoom already clears and nulls the timer, resetState's clearInterval on the now-null timer is a harmless no-op (clearInterval(null) does nothing). The channel reference being set to null a second time is also harmless. So there is no bug here, just redundant cleanup. However, if the debate mode was NOT 'live' but there was somehow a feedRealtimeChannel still active (which should not happen but could due to a bug elsewhere), resetState would null the reference without unsubscribing, potentially leaving a dangling Supabase Realtime subscription. This is a minor leak risk but would require a prior bug to trigger.

The auto-init block at the bottom is the last line of the file and is a module-level side effect. It calls ready.then(() => init()).catch(() => init()), which means: wait for the auth ready promise to settle, then call init regardless of whether auth succeeded or failed. The catch branch calling init() means that even if auth initialization fails entirely (e.g., network error preventing session check), the arena still initializes. This is intentional because the app supports anonymous guest access -- guests should be able to see the arena lobby even without an authenticated session. The void-ness of init (it returns nothing) means the promise chain here has no meaningful result; it exists purely for the side effect of triggering initialization at the right time.

One important architectural consequence of this auto-init: because the file is imported into the module graph (presumably via the main arena barrel export or a page entry point), the popstate listener is registered AND init is scheduled to run as soon as this module is evaluated. There is no way to import this module without triggering both side effects. This is fine for the main app, but it means this module cannot be imported for testing or utility purposes without also attaching a popstate listener and attempting to render into #screen-arena.

The overall user-visible behavior produced by this file is as follows. When the user navigates to the arena section of the app, the arena initializes after auth is ready: CSS is injected, the lobby renders with debate mode cards and the live feed, and any URL-based deep links (shop, join code, spectate) are processed. While using the arena, the popstate handler ensures that pressing the browser back button cleanly tears down whatever state the user was in (active debate, queue, match acceptance) and returns them to the lobby. When the user leaves the arena section entirely, destroy tears down all active connections, timers, and media resources.

In terms of risks and bugs beyond those already noted: the popstate handler does not call safeRpc to inform the server when the user backs out of a live debate room (it calls leaveDebate which only handles the WebRTC side, and cleanupFeedRoom which sends a goodbye via realtime, but there is no explicit RPC to mark the debate as abandoned on the server). This means the server-side debate status update relies on the realtime goodbye message or a separate timeout mechanism, which could leave debates in a "live" status on the server if the realtime goodbye fails to deliver. There is also no debounce on the popstate handler, so rapid back-button presses could trigger overlapping cleanup calls, though in practice the cleanup functions are idempotent (clearing an already-null timer is harmless). The unused imports of getCurrentUser and getCurrentProfile are a minor code quality issue that would be caught by a stricter linter configuration. The new URLSearchParams(window.location.search) call is repeated three times in init rather than being computed once and reused, which is a minor inefficiency but not a bug. Finally, the three query parameter checks (shop, joinCode, spectate) are not mutually exclusive -- if a URL contained both ?joinCode=ABC123&spectate=some-uuid, both would execute, potentially causing a race condition between joinWithCode and enterFeedRoomAsSpectator as they both try to transition the UI to different states. This is extremely unlikely in practice but is a theoretical edge case worth noting.

arena-core.ts is the bootstrapping and teardown spine of the entire arena subsystem. It is 161 lines long and was extracted from a former monolithic arena.ts file. Its job is fourfold: provide small utility functions used everywhere else in the arena module tree, manage browser history integration so the back button works correctly within the single-page app's arena section, initialize the arena when auth is ready, and tear everything down cleanly when the arena is destroyed. Every other file in the arena/ directory depends on arena-core.ts for at least its utility helpers or its popstate handler; arena-core.ts in turn imports from nearly every sibling module, making it the hub of a star-shaped dependency graph.

The file opens with imports from two layers: the application-wide service modules (auth.ts, config.ts, webrtc.ts, voicememo.ts) and the arena's own split modules (arena-state.ts, arena-types.ts, arena-queue.ts, arena-match.ts, arena-room-live.ts, arena-mod-refs.ts, arena-mod-queue.ts, arena-lobby.ts, arena-private-lobby.ts, arena-css.ts, arena-feed-room.ts). From auth.ts it pulls safeRpc, which is a wrapper around Supabase RPC calls that automatically retries on 401 auth failures; getSupabaseClient, which returns the live Supabase JS client instance or null if credentials are placeholders; getCurrentUser and getCurrentProfile, which return the authenticated user object and the app-level profile row respectively; and ready, which is a Promise that resolves once the auth system's onAuthStateChange INITIAL_SESSION event has fired and the user/profile state is settled. From config.ts it pulls isAnyPlaceholder, a boolean that is true when either the Supabase or Stripe credentials are placeholder/demo values, and FEATURES, the feature-flag object whose .arena property gates whether the arena initializes at all. From webrtc.ts it pulls leaveDebate, which tears down the WebRTC peer connection, closes media streams, and unsubscribes from the Supabase Realtime signaling channel used for live audio debates. From voicememo.ts it pulls cleanupPendingRecording, which stops any active MediaRecorder, revokes any outstanding ObjectURLs created for audio blob previews, and resets the recording state machine so no dangling browser mic permissions or memory leaks persist.

From the arena's own modules the imports are extensive. From arena-state.ts it imports the live-binding variables view (the current ArenaView, a union of eleven string literals like 'lobby', 'queue', 'room', 'matchFound', etc.), currentDebate (a CurrentDebate object holding the active debate's id, topic, role, mode, round count, opponent info, messages array, and various optional metadata), roundTimer and _rulingCountdownTimer (setInterval handles for the per-round countdown and the moderator ruling countdown respectively), matchFoundDebate (the debate data shown during the match-accept screen), screenEl (the DOM element #screen-arena that all arena UI renders into), queueElapsedTimer and queuePollTimer (timers for the queue waiting screen), vmTimer (the voice memo recording duration timer), and referencePollTimer (the timer that periodically fetches new reference/evidence submissions during a debate). It also imports every setter function for these variables: set_matchFoundDebate, set_screenEl, set_queueElapsedTimer, set_queuePollTimer, set_roundTimer, set_vmTimer, set_referencePollTimer, set__rulingCountdownTimer, and the master resetState function. These setters exist because ES module live bindings of let variables are read-only from the importing module's perspective; the setter functions mutate the variable inside arena-state.ts's own scope, and the live binding mechanism propagates the new value to all importers. From arena-types.ts it imports only the type aliases ArenaView and CurrentDebate, which are used for type annotations in the public accessor functions. From arena-queue.ts it imports clearQueueTimers, which clears both queuePollTimer and queueElapsedTimer and nulls them out. From arena-match.ts it imports clearMatchAcceptTimers, which clears matchAcceptTimer and matchAcceptPollTimer. From arena-room-live.ts it imports stopOpponentPoll, which clears the opponentPollTimer that checks whether the opponent has connected. From arena-mod-refs.ts it imports stopReferencePoll, which clears the referencePollTimer that periodically fetches evidence submissions during a moderated debate. From arena-mod-queue.ts it imports stopModStatusPoll, which clears the modStatusPollTimer that checks whether a moderator has accepted the debate. From arena-lobby.ts it imports renderLobby, which wipes screenEl's innerHTML and rebuilds the full lobby UI including the mode selector cards, the recent debate feed, the power-up shop button, and the pending challenge list, plus showPowerUpShop, which renders the cosmetic/strategic item shop overlay. From arena-private-lobby.ts it imports joinWithCode, an async function that calls the join_debate_by_code RPC to enter a private lobby created by another user's challenge link. From arena-css.ts it imports injectCSS, which creates a style element and appends the entire arena CSS block into the document head (guarded by a cssInjected flag so it only happens once per page load). From arena-feed-room.ts it imports cleanupFeedRoom, which tears down the Supabase Realtime channel subscription, clears the feedTurnTimer, and resets all feed-room state, plus enterFeedRoomAsSpectator, which sets up the spectator view of a live moderated debate including subscribing to real-time feed events.

The first utility function, isPlaceholder, returns true if either getSupabaseClient() returns null (meaning the Supabase client was never initialized, which happens when credentials are missing) or isAnyPlaceholder is true (meaning the config module detected placeholder API keys). This function is exported and called throughout the arena module tree as a guard: when it returns true, all network-dependent operations are skipped or return placeholder data, allowing the app to render a demo/skeleton experience without a real backend.

formatTimer takes a number of seconds and returns a string in M:SS format. It computes minutes via Math.floor(sec / 60) and seconds via the modulo operator, prepending a leading zero to single-digit seconds. This is used by the queue elapsed counter, the round timer display, the voice memo recording timer, and the feed room turn timer. The function does not handle negative numbers or values exceeding 59 minutes gracefully; a negative input would produce a negative minute count with a positive seconds component due to how JavaScript's modulo works with negative dividends, which could display as something like "-1:50" for an input of -110, though in practice the callers never pass negative values.

randomFrom is a generic function that picks a random element from a readonly array. It indexes by Math.floor(Math.random() * arr.length) and uses the non-null assertion operator. If called with an empty array, Math.random() * 0 is 0, Math.floor(0) is 0, and arr[0] is undefined, so the non-null assertion would lie and return undefined cast to T. This is a latent bug, though in practice the only callers pass the AI_TOPICS and AI_RESPONSES arrays which are always non-empty constant arrays.

The browser history section, labeled as a Session 121 rewrite, implements back-button support for the arena's internal navigation. pushArenaState takes a view name string and calls history.pushState with an object containing an arenaView property. This means each arena screen transition (lobby to queue, queue to matchFound, matchFound to room, etc.) pushes a history entry, so the browser's back button can return to previous arena states rather than leaving the page entirely.

The popstate handler, _onPopState, is assigned as a const arrow function and is immediately registered as a window-level event listener on line 90, which is a module-level side effect: merely importing arena-core.ts causes this listener to be attached. The handler does three things. First, it removes any overlay elements that might be lingering in the DOM: the rank overlay, the ruleset overlay, the mode overlay, and the mod-ruling overlay. For the mod-ruling overlay specifically, it also clears the _rulingCountdownTimer interval before removing the element, since that overlay contains a live countdown that would otherwise continue ticking and potentially throw errors when it tries to update a removed DOM element. Second, it examines the current view state and performs mode-specific cleanup. If the view is 'room' or 'preDebate', it clears the roundTimer interval, clears the _rulingCountdownTimer interval, calls stopReferencePoll (which clears the periodic evidence fetch), calls stopOpponentPoll (which clears the opponent-connection check), and calls stopModStatusPoll (which clears the moderator-acceptance check); then if the current debate's mode is 'live', it calls cleanupFeedRoom to tear down the Realtime subscription and feed turn timer, followed by leaveDebate to close the WebRTC peer connection and media streams. If the view is 'queue', it calls clearQueueTimers and then, if the app is not in placeholder mode, fires a fire-and-forget safeRpc call to leave_debate_queue, which is a server-side function that removes the user from the matchmaking queue so they don't get matched after navigating away; the catch handler logs a warning but does not surface an error to the user, which is correct since the user is already navigating away. If the view is 'matchFound', it calls clearMatchAcceptTimers to stop the accept/decline countdown and nulls out matchFoundDebate. Third, if the current view is anything other than 'lobby', it calls renderLobby to return the user to the arena's home screen. This means all back-button presses within the arena converge to the lobby, which is a deliberate UX decision: rather than trying to reconstruct intermediate states like the mode-select screen or the config-settings screen (which would require re-fetching data and reconstructing ephemeral state), the back button always goes home.

The init function is the arena's entry point. It first checks the FEATURES.arena flag and returns immediately if the arena feature is disabled, meaning the entire arena section is dark-launchable via a config change. If enabled, it calls injectCSS, which creates a style element containing all arena-specific CSS rules and appends it to the document head; the cssInjected flag inside arena-state.ts prevents double-injection if init is called multiple times. It then calls set_screenEl with the result of document.getElementById('screen-arena'), storing a reference to the DOM container. If screenEl is null after this (meaning the current HTML page does not contain a #screen-arena element), it logs a warning and returns, which is a graceful degradation: pages that don't have the arena markup simply skip arena initialization without crashing. If the element exists, it calls renderLobby, which populates screenEl with the full lobby UI.

After rendering the lobby, init checks three URL query parameters. First, if ?shop=1 is present, it calls showPowerUpShop to auto-open the power-up shop overlay on top of the lobby; this is used for deep-linking to the shop from marketing materials or other pages. Second, if ?joinCode=XXXXXX is present, it extracts the code, calls window.history.replaceState to strip the query parameter from the URL (preventing the join from re-triggering on page refresh), uppercases the code (join codes are case-insensitive), and calls joinWithCode in a void-prefixed fire-and-forget pattern. joinWithCode is an async function that calls the join_debate_by_code RPC on the server to look up the private lobby associated with that code and join the user into it. Third, if ?spectate=<uuid> is present and the value matches the UUID regex pattern (lowercase hex with dashes, case-insensitive), it strips the parameter from the URL and calls enterFeedRoomAsSpectator with the debate ID, which sets up the spectator view of a live moderated debate. The UUID validation regex on line 117 uses the standard 8-4-4-4-12 pattern and is case-insensitive via the i flag. This is important because without the validation, a malicious or malformed spectate parameter could be passed directly to the server RPC, though since it goes through safeRpc which uses PostgREST parameterized queries, SQL injection is not a concern; the validation primarily prevents wasted RPC calls with invalid IDs.

There is a subtle ordering concern in init: if both ?joinCode and ?spectate are present in the URL simultaneously, both branches will execute. The joinWithCode call fires first and navigates the user into a private lobby waiting screen, but then enterFeedRoomAsSpectator also fires and tries to set up a spectator view, potentially causing conflicting state mutations and a garbled UI. This is an edge case that is extremely unlikely to occur naturally (a user would have to manually craft a URL with both parameters), but it is technically a bug.

The two public accessor functions, getView and getCurrentDebate, provide read-only access to arena state for external modules. getView simply returns the view variable. getCurrentDebate returns either null or a shallow copy of the currentDebate object via the spread operator. The spread-copy is a defensive pattern: it prevents external code from mutating the arena's internal state by modifying properties on the returned object. However, this defense is incomplete because the messages array inside CurrentDebate is not deep-copied; the spread creates a new top-level object but the messages property still references the same array, so external code could push into or modify that array and affect the arena's state. Whether this matters depends on whether any caller actually mutates the returned object, which they probably do not, but it is a latent risk.

The destroy function is the counterpart to init. It first checks if there is an active live debate and, if so, calls cleanupFeedRoom to tear down the Realtime channel and feed timer, then leaveDebate to close WebRTC. It then calls cleanupPendingRecording from voicememo.ts, which stops any active MediaRecorder (if the user was in the middle of recording a voice memo when the arena was destroyed), revokes any ObjectURLs that were created for audio blob previews (preventing memory leaks), and resets the recording state. Next it calls resetState from arena-state.ts, which is the nuclear option: it iterates through every single mutable variable in the arena state module, clearing every setInterval timer (queuePollTimer, queueElapsedTimer, matchAcceptTimer, matchAcceptPollTimer, roundTimer, privateLobbyPollTimer, modQueuePollTimer, modStatusPollTimer, modDebatePollTimer, referencePollTimer, silenceTimer, _rulingCountdownTimer, vmTimer, opponentPollTimer, feedTurnTimer, challengeRulingTimer) and resetting every value to its initial state (view back to 'lobby', all timers to null, all counters to 0, all Sets emptied, all arrays cleared). Finally, destroy removes the popstate event listener by calling window.removeEventListener with _onPopState, which is important because if the arena is re-initialized later (e.g., navigating away and back), the listener will be re-added by the module-level side effect; without removal, multiple listeners would accumulate.

There is a concern with the destroy function's cleanup order: it calls cleanupFeedRoom and leaveDebate only when currentDebate?.mode === 'live', but the user might be in a voice memo or text debate that also has active state. However, examining the broader codebase, voice memo and text debates do not use WebRTC or feed rooms, so the live-mode guard is correct; those modes only need their timers cleared, which resetState handles.

The final two lines of the file constitute the auto-initialization mechanism. The module imports ready from auth.ts, which is a Promise<void> that resolves when the authentication system has completed its initial session check. The expression ready.then(() => init()).catch(() => init()) means: wait for auth to be ready, then initialize the arena; if auth initialization fails (the promise rejects), initialize the arena anyway. This ensures the arena always renders regardless of whether auth succeeds, which aligns with the project's guest-access-is-default philosophy: anonymous users should see the arena lobby even if they can't perform authenticated actions like joining a debate. The catch path calling init rather than silently failing is deliberate; the arena in placeholder mode will render a demo experience. This auto-init is a module-level side effect: merely importing arena-core.ts will eventually cause init() to run, which means the import order matters. If arena-core.ts is imported before the DOM contains #screen-arena, set_screenEl will store null and init will bail with a warning. In practice, the HTML pages include their script tags at the bottom of the body or use type="module" with defer semantics, so the DOM is ready by the time the import graph is resolved.

The module-level side effects of this file are therefore two: the popstate listener is attached immediately when the module is first evaluated (line 90), and the auto-init promise chain is kicked off (line 161). Both of these are intentional and documented in comments, but they mean that importing arena-core.ts in a test environment without a DOM would cause errors (getElementById returns null, window.addEventListener exists but the handler references DOM elements). There is no test-mode guard or SSR protection.

Regarding unused imports: safeRpc is imported from auth.ts on line 5 and is used on line 80 inside the popstate handler. getCurrentUser and getCurrentProfile are imported on line 5 but never referenced anywhere in the file. These are dead imports that add no functionality but do cause the tree-shaker to potentially retain those functions in the bundle (though Vite's Rollup-based tree shaking should be able to eliminate them as long as they have no side effects). They were likely left over from a refactoring session when code was split out of the monolith.

The overall user-visible behavior produced by this file is: when a user navigates to a page containing #screen-arena and the arena feature flag is enabled, they see the arena lobby rendered automatically once auth is settled. If they arrived via a deep link with ?shop=1, the power-up shop opens. If they arrived via a challenge link with ?joinCode=XXXXXX, they are automatically joined into the private lobby for that challenge. If they arrived via a spectate link with ?spectate=<uuid>, they enter the spectator view of a live debate. When they use the browser back button at any point within the arena (from the queue screen, the debate room, the match-accept screen, etc.), all active timers, WebRTC connections, and polling intervals are cleaned up and they are returned to the lobby. When the arena is destroyed (e.g., by navigating to a different section of the app), all resources are fully released: WebRTC connections closed, voice recordings stopped, all intervals cleared, and the popstate listener removed.

Risks and potential issues beyond those already mentioned: the popstate handler on line 64 uses clearInterval(_rulingCountdownTimer!) with a non-null assertion, but _rulingCountdownTimer could genuinely be null at that point if the mod-ruling overlay exists in the DOM but the timer was never started or was already cleared. clearInterval(null) is a no-op in all browsers per the HTML spec, so the non-null assertion is technically a type lie but does not cause a runtime error. The fire-and-forget safeRpc('leave_debate_queue') on line 80 could fail silently if the network is down, leaving the user in the server-side queue as a ghost entry; however, the server presumably has its own timeout mechanism to clean up stale queue entries, so this is acceptable. The URLSearchParams constructor is called three separate times on lines 106, 110, and 116, parsing the same query string each time; consolidating these into a single variable would be a minor efficiency improvement but is not a bug. The enterFeedRoomAsSpectator call on line 119 is awaited only in the void sense (the promise is not caught), so any rejection would become an unhandled promise rejection; the same is true of joinWithCode on line 113, though both of those functions likely have internal try/catch blocks.

COMPREHENSIVE ANALYSIS OF arena-core.ts
========================================

arena-core.ts is the orchestration spine of the arena subsystem. It is responsible for four concerns: initializing the arena when the page loads, tearing it down when the user navigates away, managing browser history so the back button works correctly within the single-page arena flow, and exposing a handful of small utility functions that other arena modules depend on. It was extracted from a monolithic arena.ts file and now serves as the coordination layer that wires together roughly a dozen sibling modules in the src/arena/ directory. The file is 161 lines long and every one of them matters, so here is the full story of what they do and why.

The file opens with four import blocks. The first pulls safeRpc, getSupabaseClient, getCurrentUser, and getCurrentProfile from auth.ts. Of these, only safeRpc and getSupabaseClient are actually used in this file. safeRpc is the project's standard wrapper around supabase.rpc() that handles 401 retries and error normalization; it is called exactly once, inside the popstate handler, to inform the server that the user has left the debate queue. getSupabaseClient returns the live Supabase client instance or null if credentials are placeholder stubs. getCurrentUser and getCurrentProfile are imported but never referenced in the file body, making them dead imports. They likely survived the monolith split because the original arena.ts used them and whoever carved out arena-core.ts did not prune the import list. This is not a bug per se, but it does add unnecessary bytes to the bundle (though tree-shaking in Vite's Rollup pipeline should eliminate them in production).

The second import brings isAnyPlaceholder and FEATURES from config.ts. isAnyPlaceholder is a boolean that is true when either the Supabase or Stripe credentials are placeholder values, meaning the app is running in a preview or development mode where real backend calls would fail. FEATURES is the feature-flag object; the only flag checked in this file is FEATURES.arena, which gates whether the arena initializes at all. If that flag is false, init() returns immediately and the arena never renders.

The third import, leaveDebate from webrtc.ts, is the WebRTC teardown function. It stops the worker timer, terminates the worker, closes peer connections, and releases media streams. It is called in two places in this file: inside the popstate handler when the user presses back during a live debate, and inside destroy() when the arena is being torn down programmatically.

The fourth import is ready from auth.ts, imported on a separate line from the first auth.ts import (both reference the same module, so this is just an organizational split, not a double-import issue). ready is a Promise<void> that resolves once the auth subsystem has determined whether the user is logged in or anonymous. It is the sole trigger for auto-initialization at the bottom of the file.

Then comes a large block of imports from arena-state.ts. This module is the centralized state store for the entire arena. It exports roughly 50 mutable variables as ES module let bindings, along with setter functions for each one (because ES module bindings are read-only from the importing module's perspective -- you cannot assign to an imported let, you must call a setter that mutates it from within the declaring module). arena-core.ts imports the current values of view, currentDebate, roundTimer, _rulingCountdownTimer, matchFoundDebate, screenEl, queueElapsedTimer, queuePollTimer, vmTimer, and referencePollTimer as live bindings so it can read them in the popstate handler and destroy function. It also imports the setters set_matchFoundDebate, set_screenEl, set_queueElapsedTimer, set_queuePollTimer, set_roundTimer, set_vmTimer, set_referencePollTimer, set__rulingCountdownTimer, and the nuclear option resetState, which zeroes every piece of mutable state and clears every active interval timer. The type imports ArenaView and CurrentDebate come from arena-types.ts, which is a pure type-definition module with no runtime code.

The remaining imports pull specific cleanup and rendering functions from sibling modules. clearQueueTimers from arena-queue.ts stops the two queue-related intervals (the poll timer and the elapsed-seconds timer). clearMatchAcceptTimers from arena-match.ts stops the match-accept countdown and its poll timer. stopOpponentPoll from arena-room-live.ts stops the interval that checks whether the opponent has submitted their turn. stopReferencePoll from arena-mod-refs.ts stops the interval that polls for new reference submissions and clears the pending references array. stopModStatusPoll from arena-mod-queue.ts stops the moderator status polling interval. renderLobby from arena-lobby.ts resets the arena view to the lobby and rebuilds the lobby HTML. showPowerUpShop from arena-lobby.ts opens the power-up shop overlay. joinWithCode from arena-private-lobby.ts handles joining a private debate via a six-character challenge code. injectCSS from arena-css.ts dynamically creates a style element and appends the arena's CSS rules to the document head (it is idempotent: an internal cssInjected flag prevents double-injection). cleanupPendingRecording from voicememo.ts revokes any pending ObjectURLs from voice memo recordings and cancels any in-progress recording. cleanupFeedRoom and enterFeedRoomAsSpectator from arena-feed-room.ts handle tearing down the live feed room channel and entering one as a spectator, respectively.

After the imports, the file defines three utility functions that are exported for use across the arena subsystem. isPlaceholder() returns true if the Supabase client is null or if isAnyPlaceholder is true. This is the guard that prevents RPC calls in preview mode; many arena modules call it before attempting any server communication. formatTimer() takes a number of seconds and returns a string in "M:SS" format, used for queue elapsed time displays, round countdown timers, and voice memo duration displays throughout the arena UI. randomFrom() is a generic function that picks a random element from a readonly array; it is used primarily to select random AI debate topics and random AI response templates.

The browser history section, marked as a Session 121 rewrite, provides two exports. pushArenaState() calls history.pushState with a state object containing an arenaView property. This creates a new browser history entry each time the user transitions between arena views (lobby to mode select, mode select to queue, queue to room, etc.), so that pressing the browser's back button can reverse those transitions instead of leaving the page entirely.

The heart of the history management is _onPopState, an arrow function assigned to a named export so that destroy() can remove it later. This function is the popstate event handler, and it fires every time the user presses the back button (or calls history.back()). Its job is threefold: clean up overlays, clean up the current view's resources, and return to the lobby.

The overlay cleanup at the top of _onPopState searches the DOM for three overlay elements by ID -- arena-rank-overlay, arena-ruleset-overlay, and arena-mode-overlay -- and removes them if they exist. These are modal-like overlays that appear during the mode selection flow. The ruling overlay (mod-ruling-overlay) gets special treatment: if it exists, the _rulingCountdownTimer interval is cleared before the overlay is removed, preventing a dangling timer that would keep firing and trying to update a DOM element that no longer exists.

Next, the function checks the current view. If the user is in the debate room or pre-debate screen (view === 'room' or view === 'preDebate'), it clears the round timer interval, clears the ruling countdown timer, stops the reference poll, stops the opponent poll, and stops the mod status poll. If the debate is a live mode debate, it additionally calls cleanupFeedRoom() to tear down the Supabase Realtime channel and send a disconnect event, and then calls leaveDebate() to close WebRTC connections and release media. This ordering matters: the feed room cleanup sends a final disconnect event through the channel before the channel is destroyed, so the opponent gets notified immediately rather than waiting for a heartbeat timeout.

If the user is in the queue view, clearQueueTimers() stops the queue polling and elapsed-time timers, and then -- as long as we are not in placeholder mode -- safeRpc('leave_debate_queue') is called to notify the server that this user should be removed from the matchmaking queue. The .catch() logs warnings but does not surface errors to the user, which is correct because the user is navigating away and does not care about queue-leave failures.

If the user is on the match-found screen, clearMatchAcceptTimers() stops the countdown and poll timers, and set_matchFoundDebate(null) clears the match data from state.

Finally, if the current view is anything other than lobby, renderLobby() is called to bring the user back to the main arena screen. This means that all back-button presses within the arena funnel eventually land on the lobby, which is the correct user-visible behavior: the lobby is the arena's "home" screen.

The popstate handler is registered immediately as a module-level side effect on line 90: window.addEventListener('popstate', _onPopState). This means that as soon as this module is imported by any other module, the event listener is attached. This is a deliberate design choice that ensures the handler is always active while the arena code is loaded, but it creates a subtle coupling: if arena-core.ts is imported but init() is never called (for example, if FEATURES.arena is false), the popstate handler is still listening. In practice this is harmless because the handler checks the view state before doing anything destructive, and the view defaults to 'lobby', so the final guard (if view !== 'lobby' renderLobby()) would be a no-op.

The init() function is the arena's entry point. It first checks FEATURES.arena and returns immediately if the feature is disabled, meaning no CSS is injected, no DOM lookups happen, and no lobby renders. If the feature is enabled, injectCSS() is called to append the arena's stylesheet to the document head. Then set_screenEl(document.getElementById('screen-arena')) finds the arena's container element in the DOM and stores it in the shared state module. If the element is not found, a console warning is logged and init returns early -- this would happen if the HTML page does not include the expected screen-arena div, which would be a page-structure bug.

With the screen element found, renderLobby() is called to build and display the lobby UI. Then the function checks the URL query parameters for three special cases. If ?shop=1 is present, showPowerUpShop() is called to immediately open the power-up shop overlay on top of the lobby. This is a deep-link feature that lets external links or push notifications send users directly to the shop. If ?joinCode=XXXXXX is present, the code is extracted, the query parameter is stripped from the URL via history.replaceState (so the user does not see it in the address bar and refreshing does not re-trigger the join), and joinWithCode is called with the uppercased code. This is the challenge link flow: one user creates a private lobby, gets a code, shares a link with ?joinCode=, and the recipient's arena auto-joins. If ?spectate=<uuid> is present and the value passes a UUID regex validation, the parameter is stripped from the URL and enterFeedRoomAsSpectator is called to load that debate as a spectator. The UUID regex check is a defense against injection; it ensures only well-formed UUIDs are passed to the RPC layer.

Two public accessor functions follow. getView() returns the current view state as an ArenaView type, providing a read-only interface for external modules. getCurrentDebate() returns either a shallow copy of the currentDebate object (via spread) or null. The spread copy is defensive: it prevents external callers from mutating the shared state object. However, the spread is shallow, so the messages array inside the returned object is still a reference to the original array. If any external caller were to push into that array, it would corrupt the shared state. This is a minor risk but unlikely in practice since the only consumers of getCurrentDebate() appear to use it for reading.

The destroy() function is the inverse of init(). It checks if a live-mode debate is in progress and, if so, calls cleanupFeedRoom() and leaveDebate() to tear down the realtime channel and WebRTC connections. It then calls cleanupPendingRecording() to revoke any voice memo ObjectURLs and cancel any active recording. Finally, resetState() is called, which is the nuclear reset: it iterates through every timer handle in the state module, calls clearInterval on each one, nulls each handle, and resets every mutable variable to its default value. After resetState(), destroy removes the popstate event listener, ensuring no dangling handler fires after the arena module is unloaded.

The final two lines of the file are the auto-initialization trigger: ready.then(() => init()).catch(() => init()). The ready promise from auth.ts resolves when the authentication state is determined. The pattern of calling init() in both the then and catch branches means the arena initializes regardless of whether auth succeeded or failed. This makes sense because the arena lobby should be visible even to unauthenticated (guest) users; auth is only required when they try to take actions like joining a queue. The void return of init() is implicitly consumed by the promise chain, and since init() is synchronous and does not throw, both branches behave identically in practice.

This auto-init line is a module-level side effect. When Vite bundles this file and the browser executes the module, the ready promise is subscribed to, and init() will fire once auth resolves. This means importing arena-core.ts is sufficient to boot the entire arena -- no explicit init() call is needed from the page's entry point. This is noted in the comment as matching the old IIFE pattern from before the monolith was split into modules.

Regarding side effects, this file produces three at import time: the popstate event listener is attached, the ready promise subscription is created, and (once ready resolves) init() fires, which injects CSS, looks up a DOM element, and renders the lobby. These side effects mean that merely importing this module changes the global state of the page, which is intentional for a top-level application module but would be surprising if this module were ever imported in a test harness or secondary context.

From the user's perspective, arena-core.ts is invisible. Users never interact with it directly. What they experience is: the arena lobby appears when they navigate to the arena screen; pressing the back button during any arena flow returns them to the lobby; if they arrive via a challenge link (?joinCode=), they are automatically placed into the private lobby flow; if they arrive via a spectate link (?spectate=), they enter the debate as a spectator; and when they navigate away from the arena, all timers, WebRTC connections, realtime channels, and recordings are cleaned up so nothing leaks.

As for bugs and risks, there are several worth noting. First, the dead imports of getCurrentUser and getCurrentProfile add noise and should be removed. Second, the URLSearchParams object is constructed three separate times in init() (lines 106, 110, 117), each time parsing the same query string; a single const params = new URLSearchParams(window.location.search) at the top of the function would be cleaner and marginally more efficient. Third, the shallow copy in getCurrentDebate() does not protect the messages array from external mutation, though this is a minor concern. Fourth, the popstate handler clears _rulingCountdownTimer with clearInterval but passes it through the non-null assertion operator (!), which would throw at runtime if the timer were somehow null when the ruling overlay exists -- though in practice the timer is always set when the overlay is rendered, so this is more of a TypeScript hygiene issue than a real bug. Fifth, the popstate handler does not call cleanupPendingRecording(), meaning if a user is in the middle of recording a voice memo and presses back, the recording may not be properly canceled and its ObjectURL may leak; only the destroy() path handles this cleanup. Sixth, the auto-init at the bottom of the file means init() runs unconditionally when the module loads, which could be problematic if the DOM element screen-arena does not exist yet (for instance, if the module is loaded before the DOM is ready); however, since ready is a promise that resolves after auth init, and auth init waits for DOMContentLoaded, the DOM should be available by the time init() runs. Seventh, there is a potential race condition in the spectate flow: if both ?joinCode= and ?spectate= are present in the URL, the joinWithCode call fires first and replaces the URL state, then enterFeedRoomAsSpectator fires with the spectate ID. Both are async and void-ed, so they would run concurrently, potentially corrupting the arena state. This is an edge case that would only occur if someone manually crafted a URL with both parameters, but it is technically unhandled. Eighth, the popstate handler calls safeRpc('leave_debate_queue') to dequeue the user, but if the user's auth session has expired or they are in guest mode, safeRpc's 401 retry logic may trigger a token refresh that takes time, and the user has already navigated away; the .catch() swallows this gracefully, but the RPC may still fire after the arena is destroyed if destroy() is also called in the same navigation event, leading to a harmless but unnecessary network request.

Overall, arena-core.ts is a well-structured orchestration module that cleanly separates initialization, teardown, history management, and utilities. Its primary job is lifecycle management -- ensuring the arena boots correctly, cleans up after itself, and handles the browser back button gracefully across all possible arena states. The dead imports and repeated URLSearchParams construction are the most actionable cleanup opportunities; the shallow-copy risk and popstate-recording-cleanup gap are the most meaningful behavioral concerns.

arena-core.ts is the initialization, teardown, and glue module for the entire arena subsystem of The Moderator. It was extracted from a monolithic arena.ts file (the comment on line 2 says "Part of the arena.ts monolith split") and serves as the orchestration layer that wires up the arena feature when the page loads, tears it down when the user navigates away, manages browser history for the arena's internal view stack, and provides a small handful of pure utility functions used across the other arena-* siblings. The file is 161 lines long and has no UI rendering of its own; it delegates all DOM work to arena-lobby.ts and arena-css.ts.

The imports divide into four families. The first family comes from auth.ts: safeRpc is the project's universal wrapper around supabase.rpc() that handles 401 retry logic automatically, getSupabaseClient returns the live Supabase JS client instance (or null if credentials are placeholder stubs), getCurrentUser and getCurrentProfile return the authenticated user and their profile row respectively (these two are imported but never actually used inside arena-core.ts itself, which is a minor dead-import situation — they were likely carried over from the monolith split and never cleaned up). The second import from auth.ts is the ready promise, which is a Promise<void> that resolves once onAuthStateChange fires its INITIAL_SESSION event, meaning the Supabase auth layer has finished its cold-start handshake. The config.ts import brings in isAnyPlaceholder, a boolean that is true when either the Supabase URL or the Stripe key is a dummy/placeholder value (used in development or when credentials are not configured), and FEATURES, which is the project's feature-flag object — a frozen record of booleans like arena: true, liveDebates: true, etc.

The third family of imports comes from the arena's own sibling modules. From arena-state.ts, the file imports roughly two dozen mutable state variables and their setter functions. This unusual pattern exists because ES module let bindings are read-only from the perspective of an importing module — you can read view but you cannot write view = 'lobby' from outside arena-state.ts. So every variable has a corresponding set_view(), set_roundTimer(), etc. function that mutates the variable inside arena-state.ts, and the live binding then propagates the new value to all importers. The specific variables imported are: view (the current arena sub-view like 'lobby', 'queue', 'room', 'matchFound', etc.), currentDebate (the full debate state object containing id, topic, role, mode, round, opponent info, messages, and more), roundTimer (a setInterval handle for the in-debate round countdown), _rulingCountdownTimer (a setInterval handle for the moderator ruling countdown overlay), matchFoundDebate (the debate object when a match has been found but not yet accepted), screenEl (the #screen-arena DOM element that serves as the arena's render target), queueElapsedTimer and queuePollTimer (the queue's elapsed-time display and server-poll intervals), vmTimer (voice memo recording timer), and referencePollTimer (the interval that polls for reference/citation rulings during a debate). The resetState function is also imported — it is the nuclear option that clears every single interval timer and zeros every mutable variable back to defaults.

From arena-types.ts the file imports ArenaView and CurrentDebate as TypeScript types only (the import type syntax means they are erased at compile time and produce no runtime code). ArenaView is a union type of eleven string literals representing every possible sub-screen of the arena. CurrentDebate is a rich interface with fields for the debate id, topic, the user's role (a or b), debate mode, round number, total rounds, opponent name/id/elo, ranked flag, messages array, moderator info, staking result, ruleset, spectator flag, concession state, and more.

From arena-queue.ts, clearQueueTimers is imported — it clears the two queue-related intervals (the poll timer and the elapsed-seconds display timer). From arena-match.ts, clearMatchAcceptTimers clears the match-accept countdown timer and the match-accept poll timer. From arena-room-live.ts, stopOpponentPoll clears the interval that polls for the opponent's presence/readiness in a live debate room. From arena-mod-refs.ts, stopReferencePoll clears the interval that polls for reference challenge rulings. From arena-mod-queue.ts, stopModStatusPoll clears the moderator status polling interval. From arena-lobby.ts, renderLobby is the function that paints the full lobby UI into screenEl and showPowerUpShop opens the power-up shop overlay. From arena-private-lobby.ts, joinWithCode handles joining a private debate via a six-character join code. From arena-css.ts, injectCSS creates a <style> element and appends the arena's CSS to the document head (it guards against double-injection via a cssInjected boolean). From voicememo.ts, cleanupPendingRecording revokes any ObjectURLs from in-progress voice memo recordings and resets recording state. From arena-feed-room.ts, cleanupFeedRoom tears down the live feed room (unsubscribes from the Supabase Realtime channel, clears the feed turn timer, etc.) and enterFeedRoomAsSpectator lets a user enter a live debate as a spectator by debate ID.

The webrtc.ts import brings in leaveDebate, which is the function that tears down a WebRTC peer connection — it closes the RTCPeerConnection, stops all media tracks, unsubscribes from the Supabase Realtime signaling channel, and resets audio state. This is called whenever the user leaves or is removed from a live audio debate.

The file then defines three utility functions. isPlaceholder() returns true if the Supabase client is null or if any credentials are placeholder values; this is used as a guard throughout the arena subsystem to prevent making real RPC calls in development/demo mode. formatTimer(sec) takes an integer number of seconds and produces a "m:ss" formatted string, used for displaying queue elapsed time and round countdowns. randomFrom(arr) picks a random element from a readonly array using Math.floor(Math.random() * arr.length) and the non-null assertion operator; this is used to pick random AI topics and random AI response templates elsewhere in the arena.

The browser history section (commented as "Session 121 rewrite") has two pieces. pushArenaState(viewName) calls history.pushState with an object containing an arenaView property, which pushes a new entry onto the browser's history stack so that the Back button can navigate within the arena's internal views. The _onPopState handler is the core of back-button behavior. When the user presses Back, it first removes any overlay elements that might be floating on top of the arena: the rank overlay, the ruleset overlay, the mode overlay, and the moderator ruling overlay (and if the ruling overlay exists, it also clears the ruling countdown timer). Then it examines the current view state. If the user was in the 'room' or 'preDebate' view, it clears the round timer interval, the ruling countdown timer, stops reference polling, stops opponent polling, stops moderator status polling, and if the debate mode was 'live', it calls cleanupFeedRoom() to tear down the Realtime subscription and leaveDebate() to close the WebRTC connection. If the user was in the 'queue' view, it clears the queue timers and — if not in placeholder mode — fires a leave_debate_queue RPC to tell the server the user is no longer waiting for a match (using .catch() to swallow errors, just logging a warning). If the user was in the 'matchFound' view, it clears the match accept timers and nulls out the matchFoundDebate object. Finally, if the current view was anything other than 'lobby', it calls renderLobby() to bring the user back to the arena lobby. This means ALL back navigation within the arena collapses to the lobby — there is no intermediate back-step from room to queue or queue to mode select. The _onPopState handler is immediately registered as a global popstate listener on line 90, which is a module-level side effect: as soon as this module is imported by anyone, the listener is attached to the window.

The init() function is the arena's bootstrap. It first checks the FEATURES.arena flag and returns immediately if the arena feature is disabled, making the entire arena a no-op. It calls injectCSS() which creates and appends the arena's stylesheet to the document head (idempotent via the cssInjected guard). It grabs the #screen-arena element from the DOM and stores it in the screenEl state variable via set_screenEl. If no such element exists, it logs a warning and bails — this would happen if the HTML page does not have an arena section. It then calls renderLobby() to paint the initial lobby UI. After that it checks URL query parameters for three special entry points: if ?shop=1 is present, it opens the power-up shop overlay on top of the lobby (this is used for deep-linking directly to the shop). If ?joinCode=XXXXXX is present (the F-39 challenge link feature), it strips the query parameter from the URL using replaceState (so the code does not persist across refreshes), uppercases the code, and calls joinWithCode() which initiates the flow to join a private lobby. If ?spectate=<uuid> is present (Session 240 feature), it validates the UUID format with a regex, strips the parameter, and calls enterFeedRoomAsSpectator() to drop the user straight into spectating a specific live debate.

The public accessors section exports getView() which returns the current ArenaView value, and getCurrentDebate() which returns either a shallow copy of the currentDebate object (using the spread operator, so callers cannot mutate the internal state) or null. These are the arena's read-only public API for code outside the arena subsystem.

The destroy() function is the inverse of init(). It checks if the current debate is a live-mode debate and, if so, calls cleanupFeedRoom() followed by leaveDebate() to tear down the Realtime channel and WebRTC connection. It calls cleanupPendingRecording() to revoke any ObjectURLs from voice memo recordings. It then calls resetState(), which is the comprehensive teardown function in arena-state.ts that clears every single interval timer (queue poll, queue elapsed, match accept, match accept poll, round, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence, ruling countdown, voice memo, opponent poll, feed turn, challenge ruling) and zeros out every mutable variable. Finally, it removes the popstate listener to prevent the arena from intercepting browser Back events after the arena has been destroyed.

The auto-init block at the bottom (line 161) is the file's final side effect and the reason the arena starts up automatically. It calls ready.then(() => init()).catch(() => init()), meaning it waits for the auth system's ready promise to resolve and then calls init(). The .catch(() => init()) arm means that even if the auth system's ready promise rejects (which can happen on cold visits with no saved session, or if the 6-second safety timeout fires), the arena still initializes. This is by design because the app supports guest access — anonymous users should see the full arena lobby even without being authenticated. The implication is that init() runs exactly once, triggered automatically when this module is first imported and the auth system settles.

Side effects of importing this module are significant. Line 90 attaches a popstate listener to the window immediately at module evaluation time. Line 161 schedules init() via the ready promise. This means merely importing arena-core.ts from any other module will cause the arena to initialize and the popstate listener to be attached. The arena subsystem is designed so that arena-core.ts is the root of the import tree — the other arena-* modules import from it, and whoever imports arena-core.ts triggers the entire arena boot sequence.

From the user's perspective, what happens is: the page loads, the auth system initializes (either successfully authenticating the user or timing out), and then the arena lobby appears in the #screen-arena element showing debate mode cards, a live feed of recent debates, and action buttons. If the URL had a ?shop=1 parameter, the power-up shop modal appears. If it had a ?joinCode=ABC123 parameter, the user is whisked directly into a private lobby to join a challenge. If it had a ?spectate=<uuid> parameter, the user jumps straight into spectating a live debate. When the user navigates within the arena (entering queue, finding a match, entering a debate room), each transition pushes a history entry. Pressing the browser's Back button at any point tears down whatever the user was doing (leaving the queue, declining a match, closing timers, disconnecting WebRTC) and returns them to the lobby. When the arena is destroyed (navigating away from the arena page entirely), all resources are cleaned up: WebRTC connections closed, Realtime subscriptions ended, intervals cleared, ObjectURLs revoked.

There are several bugs and risks worth noting. First, getCurrentUser and getCurrentProfile are imported from auth.ts but never used anywhere in arena-core.ts — these are dead imports that add unnecessary bundle weight (though tree-shaking in the Vite build should eliminate them, the imports are not side-effect-free declarations, so Vite may or may not eliminate them depending on its analysis). Second, the _onPopState handler reads the view variable to decide what cleanup to perform, but view is a live binding from arena-state.ts that could theoretically be in an intermediate state if the popstate event fires during an async transition. In practice this is unlikely because popstate fires on the main thread and the view is set synchronously, but it is worth noting. Third, in the _onPopState handler, the leave_debate_queue RPC is fire-and-forget with only a console.warn on failure — if the network is down when the user presses Back, their queue slot on the server may not be released, potentially causing ghost entries in the matchmaking queue until the server's own timeout cleans them up. Fourth, the popstate handler on line 90 is attached unconditionally at module load time, even before init() runs. If the arena feature is disabled (FEATURES.arena is false), init() returns early and never renders anything, but the popstate listener is still active. This means pressing Back on a page where the arena is disabled would still run the overlay cleanup code and potentially call renderLobby() if view happened to not be 'lobby' (though since init never ran, view would still be 'lobby' from its default value, so the final if-check would prevent renderLobby from firing — this is safe by accident rather than by design). Fifth, the destroy() function only checks currentDebate?.mode === 'live' before calling cleanupFeedRoom() and leaveDebate(), but the _onPopState handler performs much more extensive cleanup (clearing round timers, stopping reference polls, stopping opponent polls, stopping mod status polls). This means if destroy() is called while the user is in a non-live debate room (e.g., text or voice memo mode), the round timer and polling intervals might not be stopped before resetState() clears them. However, resetState() does clear all these timers itself, so the practical risk is minimal — the timers will fire a few more times between the destroy() call and the resetState() call, which is harmless. Sixth, the spectate UUID validation regex on line 117 only accepts lowercase hex digits with the /i flag, which is correct, but it does not validate UUID version or variant — it accepts any 8-4-4-4-12 hex string, which is fine for this use case. Seventh, new URLSearchParams(window.location.search) is constructed three separate times in init() (lines 106, 110, 117) when it could be constructed once and reused, which is a minor efficiency issue but not a bug. Eighth, the auto-init on line 161 means the arena initializes even on pages that might not have a #screen-arena element — init() handles this gracefully by bailing after the console.warn, but the CSS injection (injectCSS) still happens before the element check, meaning arena CSS is injected into the document head on every page regardless of whether the arena UI exists. This is wasteful but not harmful.

The overall architecture is sound for a single-page-app arena subsystem. The module serves as a clean orchestration point: it boots the arena, manages the browser history stack, provides teardown, and exposes a minimal public API. The split from the monolith into arena-core, arena-state, arena-types, arena-queue, arena-match, arena-lobby, arena-room-live, arena-mod-refs, arena-mod-queue, arena-private-lobby, arena-feed-room, and arena-css follows a reasonable separation of concerns where arena-core handles lifecycle and arena-state holds all mutable state with its setter functions acting as the project's approximation of a state management pattern.

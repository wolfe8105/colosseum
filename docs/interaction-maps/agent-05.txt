arena-core.ts is the initialization, teardown, and utility hub for the entire arena subsystem. It was extracted from what the comment header calls "the arena.ts monolith split," meaning it once lived inside a single enormous arena.ts file and was broken out to hold the lifecycle bookends (init and destroy) plus a handful of pure helpers and the browser history handler. Every other arena-* module depends on it for those utilities, and it depends on nearly every other arena-* module in turn for cleanup duties. The file is 161 lines long, but its influence radiates across the entire arena feature surface.

The import block at lines 4 through 31 pulls from ten distinct modules. From auth.ts it brings safeRpc, which is the project's universal Supabase RPC wrapper that handles 401 retry logic; getSupabaseClient, which returns the live Supabase client instance or null if credentials are placeholder; getCurrentUser and getCurrentProfile, which return the cached auth user and profile objects respectively. Notably, getCurrentUser and getCurrentProfile are imported but never actually referenced anywhere in this file -- they are dead imports, likely left over from an earlier refactor when init() or destroy() contained profile-aware logic that was later moved to arena-lobby.ts or arena-room-setup.ts. This is not a bug per se since tree-shaking at build time should eliminate them, but it is unnecessary noise and a minor code-smell.

From config.ts the file imports isAnyPlaceholder, a boolean that is true whenever either the Supabase URL or the Stripe key is set to a development placeholder value, and FEATURES, the global feature-flag object whose .arena boolean gates whether the entire arena initializes at all.

From webrtc.ts it imports leaveDebate, the function that tears down the WebRTC peer connection, stops media tracks, kills the worker timer, and generally disconnects the user from a live audio debate session.

The ready import on line 9, also from auth.ts, is a Promise<void> that resolves once the auth layer has determined whether the user is logged in or anonymous. This is the gating mechanism used at the bottom of the file to defer initialization.

The large block from arena-state.ts (lines 11-19) imports a mixture of live-binding variables and setter functions. The variables -- view, currentDebate, roundTimer, _rulingCountdownTimer, matchFoundDebate, screenEl, queueElapsedTimer, queuePollTimer, vmTimer, referencePollTimer -- are ES module let bindings that are read-only from the perspective of an importing module. That is why every variable has a companion set_ function: because ES module live bindings cannot be reassigned from outside the declaring module, each setter is a function defined in arena-state.ts that mutates the variable locally, and the live binding propagates the new value to all importers. The resetState function, also from arena-state.ts, is the nuclear option -- it clears every single interval timer and resets every mutable variable back to its default, and it is called inside destroy().

The type imports on line 20 bring ArenaView (the union type of all possible arena screen states: lobby, modeSelect, queue, matchFound, room, preDebate, postDebate, and several moderator/private-lobby states) and CurrentDebate (the rich interface describing an in-progress debate including topic, role, mode, round count, opponent info, moderator info, staking results, and several flags).

From arena-queue.ts the file imports clearQueueTimers, a function that stops the two queue-related interval timers (the poll timer that checks the server for a match and the elapsed-seconds display timer). From arena-match.ts it imports clearMatchAcceptTimers, which similarly stops the match-accept countdown timer and the match-accept polling timer. From arena-room-live.ts it imports stopOpponentPoll, which clears the interval that polls for the opponent's latest debate turn every 3 seconds. From arena-mod-refs.ts it imports stopReferencePoll, which clears the interval that polls for newly submitted references during a moderated debate. From arena-mod-queue.ts it imports stopModStatusPoll, which clears the interval that checks whether a human moderator has accepted or declined a moderation request.

From arena-lobby.ts come renderLobby, the function that builds and injects the full lobby HTML (mode cards, arena feed, leaderboard snippets) into the #screen-arena element, and showPowerUpShop, which renders the power-up purchase overlay. From arena-private-lobby.ts comes joinWithCode, the async function that takes a six-character alphanumeric challenge code, calls the join_private_lobby RPC, and either enters the matched debate room or shows an error toast. From arena-css.ts comes injectCSS, which appends a <style> element containing all arena-specific CSS to the document head, guarded by a cssInjected boolean so it only runs once. From voicememo.ts comes cleanupPendingRecording, which revokes any ObjectURLs created for in-progress voice memo recordings to prevent memory leaks. From arena-feed-room.ts come cleanupFeedRoom, which tears down the Supabase Realtime channel subscription, clears the feed turn timer, and notifies the opponent of disconnection, and enterFeedRoomAsSpectator, which calls the get_arena_debate_spectator RPC and renders the spectator view of a live feed-room debate.

The utility section starting at line 36 defines three small pure functions. isPlaceholder returns true if either the Supabase client is null (meaning the SDK was never initialized because placeholder credentials were detected) or isAnyPlaceholder is true; this is used throughout the arena modules to short-circuit server calls during local development. formatTimer takes a number of seconds and returns a string in "M:SS" format, used to display countdown clocks for round timers, queue elapsed time, and match-accept countdowns. randomFrom is a generic function that picks a uniformly random element from a readonly array, used to select random AI topics and random AI response templates during AI sparring mode.

The browser history section at line 54 contains pushArenaState and the popstate handler. pushArenaState calls history.pushState with an object containing an arenaView key, which means every arena screen transition pushes a new entry onto the browser history stack. This lets the user press the browser back button to navigate backward through arena states. The _onPopState handler, defined as an arrow function and immediately registered as a window popstate listener on line 90, is the counterpart that fires when the user presses back.

When _onPopState fires, it first cleans up any overlay elements that might be floating in the DOM -- the rank selection overlay, the ruleset selection overlay, the mode selection overlay, and the moderator ruling overlay. For the ruling overlay specifically, it also clears the _rulingCountdownTimer interval, because that overlay has a live countdown that would otherwise keep ticking and potentially cause errors when it tries to update a DOM element that no longer exists.

Then it checks the current view state. If the user is in the room or preDebate view, it aggressively tears down everything related to an active debate: it clears the round timer, clears the ruling countdown timer, stops the reference poll, stops the opponent poll, stops the moderator status poll, and if the debate mode is live, it calls cleanupFeedRoom (which unsubscribes from the Realtime channel and notifies the opponent) and leaveDebate (which tears down WebRTC). If the user is in the queue view, it clears the queue timers and, as long as the app is not in placeholder mode, fires a leave_debate_queue RPC to tell the server the user is no longer waiting. That RPC call is fire-and-forget with a .catch that just logs a warning, which is appropriate because the user is navigating away and there is no good way to surface the error. If the user is in the matchFound view, it clears the match-accept timers and nulls out the matchFoundDebate state. Finally, regardless of what view the user was in, if they were not already on the lobby, it calls renderLobby to take them back to the lobby screen. This means the back button always returns to the lobby -- there is no multi-step back navigation through the arena states, which is a deliberate simplification.

The init function at line 96 is the entry point for the entire arena feature. It first checks FEATURES.arena and returns immediately if the arena feature flag is off, meaning the entire arena can be disabled by flipping a single boolean in config.ts. It then calls injectCSS to ensure the arena stylesheet is in the DOM. It grabs the #screen-arena DOM element via getElementById and stores it in the module-level screenEl variable via set_screenEl. If that element does not exist, it logs a warning and bails out, which means the arena silently fails to initialize on any page that does not have a #screen-arena div in its HTML. Assuming the element exists, it calls renderLobby to paint the initial lobby view.

After the lobby renders, init checks URL query parameters for three special behaviors. If ?shop=1 is present, it immediately opens the power-up shop overlay on top of the lobby, which is presumably used for deep-linking from marketing materials or notifications. If ?joinCode=XXXXXX is present (the F-39 challenge link feature), it strips that parameter from the URL using history.replaceState so the user does not accidentally re-trigger the join on a page refresh, then calls joinWithCode with the uppercased code. The void keyword discards the promise, so any error inside joinWithCode is handled internally by that function (it shows a toast). If ?spectate=<uuid> is present (Session 240 spectator links), it validates the UUID with a strict regex that matches the standard 8-4-4-4-12 hex format, strips the parameter from the URL, and calls enterFeedRoomAsSpectator. The UUID validation is a good security practice -- it prevents injection of arbitrary strings into the RPC call.

There is a subtle ordering issue here: the joinCode and spectate checks are not mutually exclusive. If someone crafted a URL with both ?joinCode=ABC123&spectate=some-uuid, both branches would execute. The joinWithCode call would fire first and navigate the user into a private lobby join flow, and then enterFeedRoomAsSpectator would also fire and try to navigate into a spectator view. In practice this would likely result in confusing behavior where one overwrites the other's DOM output, but it is an extremely unlikely edge case since these URLs are generated by the application, not typed by users.

The public accessors section at line 127 exports getView and getCurrentDebate. getView simply returns the current view state variable, providing read access to external modules without exposing the raw mutable binding. getCurrentDebate returns either a shallow copy of currentDebate (via the spread operator) or null. The shallow copy is a deliberate defensive pattern -- it prevents external code from mutating the canonical state object, though since CurrentDebate contains nested objects like the messages array, a deep clone would be needed for full immutability. This is a minor gap: if an external caller did currentDebate = getCurrentDebate() and then pushed to currentDebate.messages, it would mutate the original array since the spread only copies top-level properties.

The destroy function at line 139 is the full teardown path. It checks if there is a live-mode debate in progress and, if so, calls cleanupFeedRoom and leaveDebate to properly disconnect from the Realtime channel and WebRTC session. It then calls cleanupPendingRecording to revoke any blob URLs from in-progress voice memo recordings. Then it calls resetState, which is the comprehensive nuclear reset from arena-state.ts that clears every interval timer (queue poll, queue elapsed, match accept, match accept poll, round timer, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence timer, ruling countdown, voice memo timer, opponent poll, feed turn timer, challenge ruling timer) and resets every mutable variable to its default value. Finally, it removes the popstate event listener to prevent the handler from firing after the arena has been torn down, which could cause null reference errors if it tried to access DOM elements that no longer exist.

The auto-init block at line 161 is the final line of the file and it is a module-level side effect. When this module is first imported (which happens at Vite bundle evaluation time), it immediately chains off the ready promise: ready.then(() => init()).catch(() => init()). This means init() is called once the auth system has determined the user's authentication state, and crucially, init() is also called if the auth ready promise rejects (the catch branch). This is intentional -- it ensures the arena initializes even if auth initialization fails, because the arena supports guest/anonymous users. The comment "matches .js IIFE" suggests this replaced an older pattern where the original monolithic arena.js used an immediately-invoked function expression to self-initialize.

This auto-init is the most significant side effect of the entire file. Simply importing arena-core.ts anywhere in the dependency tree causes the arena to initialize as soon as auth is ready. This is a module-level side effect that cannot be tree-shaken. If any module imports anything from arena-core.ts, the init() call will execute. In the current architecture this is by design -- the arena page's entry point imports from the arena modules, and the auto-init fires. But it means you cannot import utility functions like formatTimer or isPlaceholder from this file without also triggering arena initialization, which is a coupling concern.

There is one notable risk in the popstate handler: on line 64, the code does clearInterval(_rulingCountdownTimer!) with a non-null assertion. If _rulingCountdownTimer is null at that point, clearInterval(null) is actually safe in browsers (it is a no-op), but the non-null assertion is semantically misleading. Similarly, lines 68-69 use non-null assertions on roundTimer and _rulingCountdownTimer inside the room/preDebate branch. These are safe because clearInterval tolerates null/undefined, but they suggest the author was not fully confident in the null-check logic.

Another minor risk: the popstate handler on line 80 calls safeRpc('leave_debate_queue') when exiting the queue view, but safeRpc performs a 401 retry, meaning it will attempt the RPC twice if the first call gets a 401. This is fine for idempotent operations (leaving a queue twice is harmless), but it means there is a small window where the user has already navigated away to the lobby and a retry is still in flight. Since the promise is caught and discarded, this causes no visible issue.

From a user-visible behavior standpoint, this file orchestrates the following experience: when the user navigates to the arena page, the auth system resolves, and the arena lobby renders showing mode cards and the debate feed. If the URL contains a challenge code, the user is automatically pulled into a private lobby join flow. If the URL contains a spectate parameter, the user is dropped into a spectator view of an ongoing debate. If the URL has ?shop=1, the power-up shop overlay appears. When the user presses the browser back button at any point during the arena experience -- whether they are in the queue waiting for a match, viewing a match-found screen, or in the middle of a live debate -- they are taken back to the lobby, and all active connections, timers, and server-side queue entries are cleaned up. When the user navigates away from the arena entirely (triggering destroy), everything is torn down including WebRTC connections, Realtime subscriptions, voice memo recordings, and all interval timers.

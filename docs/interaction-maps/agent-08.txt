arena-core.ts is the bootstrap, teardown, and utility hub for the entire arena subsystem. It was extracted from a monolith split of the original arena.ts and serves as the glue that wires initialization, browser history management, cleanup, and a handful of small helper functions that other arena modules depend on. The file is 161 lines long and every one of them matters, so here is what happens from top to bottom.

The file opens with four import blocks. The first pulls safeRpc, getSupabaseClient, getCurrentUser, and getCurrentProfile from auth.ts, though only safeRpc and getSupabaseClient are actually used within this file; getCurrentUser and getCurrentProfile are imported but never referenced anywhere in arena-core.ts, making them dead imports that a tree-shaker would strip but that clutter the source. safeRpc is the project's universal RPC wrapper that retries on 401 auth failures, used here only inside the popstate handler to call leave_debate_queue when the user navigates back from the queue view. getSupabaseClient returns the singleton Supabase client instance or null if auth has not initialized, and it is used in the isPlaceholder helper. The second item from that same import line, isAnyPlaceholder, comes from config.ts and is a boolean computed at module load time that is true when either the Supabase or Stripe credentials are placeholder/demo values; FEATURES also comes from config.ts and is an object of boolean feature flags, where FEATURES.arena gates whether the arena initializes at all. The third import, leaveDebate from webrtc.ts, tears down WebRTC peer connections and media streams when a live audio debate ends. The fourth import, ready from auth.ts (imported separately on line 9, which is a second import from the same module, a minor style inconsistency), is a Promise that resolves when the Supabase auth INITIAL_SESSION event fires, acting as the single canonical signal that auth is ready.

The massive import block from arena-state.ts on lines 10-19 pulls in every mutable state variable and its corresponding setter function. The arena state module uses ES module live bindings: it declares module-scoped let variables for things like view (the current arena screen), currentDebate (the active debate object), roundTimer, _rulingCountdownTimer, matchFoundDebate, screenEl, queueElapsedTimer, queuePollTimer, vmTimer, and referencePollTimer. Because ES module let bindings are read-only from importing modules, the state module also exports setter functions (set_matchFoundDebate, set_screenEl, set_queueElapsedTimer, set_queuePollTimer, set_roundTimer, set_vmTimer, set_referencePollTimer, set__rulingCountdownTimer) that mutate the variables within the state module's own scope, and live binding propagation means all other importers see the updated value. The resetState function, also imported from arena-state.ts, is a comprehensive teardown that clears every single interval timer and resets every mutable variable to its initial value.

The type import on line 20 brings in ArenaView and CurrentDebate from arena-types.ts. ArenaView is a union of string literals representing every screen the arena can show: lobby, modeSelect, queue, matchFound, room, preDebate, postDebate, privateLobbyWaiting, modQueue, modDebatePicker, and modDebateWaiting. CurrentDebate is the central interface describing an active debate, carrying the debate id, topic, the user's role (a or b), the mode (live, voicememo, text, ai), round tracking, opponent info, moderator info, messages, staking results, and various flags like spectatorView, concededBy, and _nulled.

Lines 21-28 import cleanup functions from four sibling arena modules. clearQueueTimers from arena-queue.ts (line 257 of that file) stops the queue polling interval and the elapsed-seconds counter that shows the user how long they have been waiting. clearMatchAcceptTimers from arena-match.ts (line 58) stops the match accept countdown timer and the polling timer that checks whether the opponent has accepted. stopOpponentPoll from arena-room-live.ts (line 87) stops the interval that polls for opponent presence during a live debate. stopReferencePoll from arena-mod-refs.ts (line 204) stops the interval that polls for moderator rulings on submitted references. stopModStatusPoll from arena-mod-queue.ts (line 162) stops the interval that polls whether a human moderator has been assigned to the debate.

Lines 26-27 import renderLobby and showPowerUpShop from arena-lobby.ts. renderLobby (line 31 of that file) replaces the contents of the #screen-arena element with the lobby UI, which is the default landing view for the arena. showPowerUpShop (line 302, an async function) opens the power-up shop overlay where users can equip items before a debate. Line 27 imports joinWithCode from arena-private-lobby.ts (line 202, also async), which handles joining a private debate lobby via a six-character alphanumeric code. Line 28 imports injectCSS from arena-css.ts (line 7), which dynamically creates a style element and appends the arena's CSS to the document head, but only once (guarded by the cssInjected flag in arena-state.ts). Line 29 imports cleanupPendingRecording from voicememo.ts (line 199), which stops any active MediaRecorder, revokes any Object URLs created for audio blobs, and resets recording state. Lines 30 imports cleanupFeedRoom and enterFeedRoomAsSpectator from arena-feed-room.ts. cleanupFeedRoom (line 1239) tears down the feed room's realtime subscription channel, clears the feed turn timer, and resets feed room state. enterFeedRoomAsSpectator (line 1168, async) sets up spectator mode for watching a live debate.

The first executable code is the isPlaceholder utility on line 36, which returns true if there is no Supabase client (meaning auth.ts never initialized) or if isAnyPlaceholder is true (meaning the app is running with demo credentials). This function is used throughout the arena subsystem as a guard: if isPlaceholder returns true, server-side operations are skipped and the UI shows placeholder/demo behavior instead. Notably, the function is named identically to a method on ModeratorConfig in config.ts, which could cause confusion during maintenance, though they are in different scopes.

formatTimer on line 40 takes a number of seconds and returns a string in m:ss format, padding the seconds with a leading zero when needed. It is a pure function with no side effects, used by the queue elapsed display, the round timer display, and the voice memo recording timer. There is a subtle behavior: if sec is negative (which could happen if a timer calculation goes wrong), Math.floor of a negative number and the modulo operator would produce something like "-1:0-5" because JavaScript's modulo of a negative number preserves the sign. This is not a bug per se because callers are expected to clamp, but there is no defensive guard here.

randomFrom on line 46 picks a random element from a readonly array. It is generic, typed to return T, and uses the non-null assertion operator (!) on the array access. If an empty array were passed, Math.random would produce an index of 0, the array access would return undefined, and the non-null assertion would be a lie, leading to a runtime error downstream. In practice, it is called with the AI_TOPICS and AI_RESPONSES arrays which are non-empty constants, so this is safe, but it is a latent risk if someone reuses the function with a dynamically-populated array.

The browser history section starting at line 52 was rewritten in Session 121. pushArenaState on line 54 calls history.pushState with an object containing the arenaView name, which allows the popstate handler to know which arena view was active when the user navigates back. The empty string second argument is the unused title parameter. This means every time the user enters a new arena view (queue, match found, debate room, etc.), a history entry is pushed so the browser back button works.

The popstate handler _onPopState, defined as an arrow function on line 58 and immediately registered on the window at line 90, is where the most complex cleanup logic lives. When the user presses the browser back button, the handler first removes any overlay elements that might be floating in the DOM: the rank overlay, ruleset overlay, mode overlay, and the moderator ruling overlay. For the ruling overlay, it also clears the _rulingCountdownTimer interval before removing the element, which is correct because that timer updates a visual countdown inside the overlay and leaving it running after the element is removed would cause errors or silent no-ops writing to a detached DOM node.

Next, the handler checks the current view state. If the user was in the room or preDebate view, it clears the round timer interval, the ruling countdown timer interval, stops the reference poll (which was checking for moderator rulings on submitted evidence), stops the opponent poll (which was checking whether the opponent is still connected), stops the mod status poll (which was checking for moderator assignment), and if the debate mode was live, it also calls cleanupFeedRoom to tear down the realtime subscription and leaveDebate to close WebRTC connections. This is thorough cleanup that prevents orphaned intervals and stale network connections.

If the user was in the queue view, it clears queue timers and then fires off a leave_debate_queue RPC call to the server, which removes the user from the matchmaking queue. The RPC call is fire-and-forget with a .catch that only logs a warning. The isPlaceholder guard prevents the RPC call in demo mode, which is correct. However, there is a potential issue: if the popstate fires rapidly (user clicking back multiple times), the leave_debate_queue call might race with other state transitions. The .catch ensures it does not throw, but the server might receive the leave request after the user has already been matched, which could cause a no-op or a harmless error on the server side.

If the user was in the matchFound view (the "match found, accept?" screen), it clears match accept timers and nulls out matchFoundDebate. This prevents the accept countdown from continuing to tick after the user has navigated away.

Finally, line 88 checks that the current view is not already lobby, and if it is not, calls renderLobby to return the user to the arena lobby. This is the key behavior: all browser back navigation within the arena returns to the lobby. There is no multi-level back navigation through arena sub-views; it is always a single hop back to lobby regardless of depth. This is a deliberate UX decision, not a bug.

The init function starting at line 96 is the entry point for the arena subsystem. It first checks FEATURES.arena and returns immediately if the arena feature is disabled, meaning no CSS injection, no DOM lookup, no lobby render, nothing. If the feature is enabled, it calls injectCSS, which appends the arena stylesheet to the document head (idempotently, guarded by a flag). Then it looks up the #screen-arena DOM element and stores it via set_screenEl. If the element is not found, it logs a warning and returns, which means the arena silently fails to initialize if the HTML page does not contain the expected container element. This is defensive but could be surprising in debugging because there is no user-visible error.

Assuming the element is found, renderLobby is called to populate the arena container with the lobby UI. Then init checks URL query parameters for three auto-actions. First, if ?shop=1 is in the URL, showPowerUpShop is called to open the power-up shop overlay on top of the lobby. Second, if ?joinCode=XXXXXX is present (the F-39 challenge link feature), the URL is cleaned via history.replaceState (removing the query parameter so it does not persist on refresh), and joinWithCode is called with the uppercased code. The void keyword before the call explicitly discards the promise, which is fine because joinWithCode handles its own errors internally. Third, added in Session 240, if ?spectate=<debateId> is present and the value matches the UUID regex pattern (standard 8-4-4-4-12 hex format, case insensitive), the URL is cleaned and enterFeedRoomAsSpectator is called. There is a potential issue here: if both joinCode and spectate are present in the URL, both actions will fire, which would be conflicting. This is unlikely in practice since these URLs are generated by different flows, but there is no mutual exclusion.

The URL parameter parsing creates three separate URLSearchParams instances on lines 106, 110, and 116, which is wasteful but harmless. A single parse would be more efficient but the performance impact is negligible.

The public accessors section provides getView on line 127, which returns the current ArenaView string from the state module, and getCurrentDebate on line 131, which returns either a shallow copy of the currentDebate object (via spread) or null. The shallow copy is intentional: it prevents callers from mutating the internal state object directly, but it is only a shallow defense because nested objects like the messages array are still shared references. If a caller modified currentDebate.messages, it would affect the original. In practice, this function appears to be used by external modules that only read the data, so this is a minor theoretical risk.

The destroy function on line 139 is the counterpart to init. It is called when the arena is being torn down, such as when navigating away from the arena page entirely. It first checks if a live debate is active and if so calls cleanupFeedRoom and leaveDebate to close the realtime channel and WebRTC connections. Then it calls cleanupPendingRecording to stop any voice memo recording in progress and revoke associated Object URLs. Then it calls resetState, which is the nuclear option: it clears every single interval timer in the arena state module (queue poll, queue elapsed, match accept, match accept poll, round, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence, ruling countdown, vm, opponent poll, feed turn, challenge ruling) and resets all mutable variables to their defaults. Finally, it removes the popstate event listener, which is critical because without this removal, the _onPopState handler would fire on subsequent back navigations even though the arena is no longer active, potentially calling renderLobby on a detached or repurposed DOM element.

There is a subtle issue in destroy: the currentDebate check on line 141 happens before resetState, which is correct because resetState nulls out currentDebate. But if currentDebate is non-null and its mode is not live (say it is text or voicememo), cleanupFeedRoom and leaveDebate are skipped, which is correct because those only apply to live mode. However, for voicememo or text mode, there might be other cleanup needed (like aborting an in-flight text submission), and that is not handled here. Whether that matters depends on the caller's context, but it is worth noting as a gap.

The auto-init section at the very bottom, line 161, is the most important line for understanding when the arena boots. It calls ready.then(() => init()).catch(() => init()), which means: wait for the auth ready promise to resolve, then call init; but if the ready promise rejects (which happens if auth initialization fails entirely), still call init anyway. This ensures the arena always attempts to initialize regardless of auth state, which aligns with the project's "guest access is default" philosophy where anonymous users see the full app. The catch branch calling init means that even if Supabase auth is completely broken, the arena lobby will still render (though all RPC-dependent features would fail gracefully via the isPlaceholder guard or safeRpc error handling).

This auto-init runs as a module-level side effect, meaning it fires the moment any module imports from arena-core.ts (or transitively, when the arena entry point is loaded). This is the IIFE equivalent noted in the comment. The side effect is the registration of the popstate listener on line 90, which happens unconditionally at module load time, even before init runs. This means that if the module is loaded but FEATURES.arena is false, the popstate listener is still registered but _onPopState will reference view (which is lobby by default) and call renderLobby, which would try to write to a null screenEl. The renderLobby function in arena-lobby.ts presumably guards against this, but it is an ordering concern: the listener is registered synchronously at module parse time, while init runs asynchronously after the ready promise resolves. If a popstate event fires in that window, _onPopState would run with uninitialized state. In practice this is extremely unlikely because no history entries have been pushed yet, but it is a theoretical race.

From the user's perspective, what this file orchestrates is the following experience: when the user navigates to the arena page, the auth system initializes, and once ready (or even if auth fails), the arena lobby appears. The user sees debate mode cards, their stats, and a queue button. If they arrived via a challenge link (?joinCode=), they are automatically taken into the private lobby join flow. If they arrived via a spectate link (?spectate=), they are placed directly into spectator mode for that debate. If they arrived with ?shop=1, the power-up shop overlay opens. When the user navigates through arena sub-views (selecting a mode, entering the queue, accepting a match, debating), each transition pushes a history state so the browser back button works. Pressing back from any sub-view returns them to the lobby, cleaning up all active timers, WebRTC connections, realtime subscriptions, and server-side queue entries. When the user leaves the arena entirely, destroy tears everything down comprehensively.

In terms of bugs and risks, the most notable findings are: the dead imports of getCurrentUser and getCurrentProfile which add noise; the lack of empty-array protection in randomFrom; the triple URL parameter parsing in init; the potential for both joinCode and spectate to fire simultaneously; the popstate listener being registered before init completes; and the shallow copy in getCurrentDebate that does not protect nested objects. None of these are likely to cause user-visible failures in the current codebase, but they represent maintenance risks and edge cases that could become real issues if the code evolves. The overall architecture is solid: the separation of state into arena-state.ts with setter functions is a pragmatic workaround for ES module binding rules, the comprehensive cleanup in both _onPopState and destroy prevents resource leaks, and the ready-or-fail init pattern correctly implements the guest-first access model.

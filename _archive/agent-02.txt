COMPREHENSIVE ANALYSIS OF arena-core.ts
========================================

arena-core.ts is the bootstrap, teardown, and utility hub for the entire arena subsystem of The Moderator platform. It is the file that the rest of the arena module graph depends on for initialization, destruction, browser history management, and a handful of small pure-utility functions. It was extracted from what the comment header calls "the arena.ts monolith split," meaning it once lived inside a single enormous arena.ts file and was broken apart for maintainability. Despite being only 161 lines, it is the connective tissue that wires the arena module into the page lifecycle, and understanding it requires tracing every import and every outbound call into the broader system.

The file opens with four imports from auth.ts: safeRpc, getSupabaseClient, getCurrentUser, and getCurrentProfile. Of these, only safeRpc and getSupabaseClient are actually used in this file. safeRpc is the project-wide wrapper around supabase.rpc() calls that handles 401 retry logic; it appears once, inside the popstate handler, to call the leave_debate_queue server function when a user navigates backward out of the queue view. getSupabaseClient returns the initialized Supabase client instance and is used inside the isPlaceholder utility to check whether the client exists. getCurrentUser and getCurrentProfile are imported but never referenced in this file's own code -- they are dead imports here, likely left over from the monolith split when this file once contained code that needed them, or perhaps kept for anticipated future use. This is a minor code hygiene issue but has no runtime cost because Vite's tree-shaking will eliminate unused imports during the build.

The second import line brings isAnyPlaceholder and FEATURES from config.ts. isAnyPlaceholder is a boolean computed at module load time that is true when either the Supabase or Stripe credentials are placeholder/demo values rather than real ones. It feeds into the isPlaceholder() utility function. FEATURES is the feature flags object; its arena boolean property gates whether the init() function does anything at all. If FEATURES.arena is false, the entire arena system silently refuses to start -- no CSS injection, no lobby render, no URL parameter processing.

The third import, leaveDebate from webrtc.ts, is the WebRTC teardown function. It stops the worker timer that tracks round timing, terminates the worker, resets ICE restart state, clears disconnect and setup timeouts, and presumably closes peer connections and media streams. It is called in two places in arena-core.ts: inside the popstate handler when the user navigates backward while in a live debate room, and inside destroy() when the arena module is fully torn down. Both call paths first check that the current debate's mode is 'live', because leaveDebate only applies to WebRTC-based live audio debates, not to text, voice memo, or AI sparring modes.

The fourth import is the ready promise from auth.ts. This is a Promise<void> that resolves when the authentication system has finished its INITIAL_SESSION event processing (or rejects if auth init fails, such as on timeout). The very last line of arena-core.ts uses this promise: ready.then(() => init()).catch(() => init()). This means init() will be called regardless of whether auth succeeds or fails. This is intentional and matches the project's "guest access is default" philosophy -- anonymous users should still see the arena lobby. The catch branch ensures that even if the 6-second auth safety timeout fires and rejects the promise, the arena still initializes.

The large block of imports from arena-state.ts pulls in the mutable state variables and their setter functions. ES module semantics make let bindings read-only from the importing side, so the arena-state module exports setter functions (like set_matchFoundDebate, set_screenEl, etc.) that mutate the variables locally, and the live binding mechanism propagates those mutations to all importers. The specific variables imported here are: view (the current ArenaView string, one of lobby/modeSelect/queue/matchFound/room/preDebate/postDebate/privateLobbyWaiting/modQueue/modDebatePicker/modDebateWaiting), currentDebate (a CurrentDebate object or null describing the active debate), roundTimer and _rulingCountdownTimer (setInterval handles for the round countdown and moderator ruling countdown), matchFoundDebate (the debate data held during the match-accept screen), screenEl (the #screen-arena DOM element), queueElapsedTimer and queuePollTimer (timers for the matchmaking queue), vmTimer (voice memo recording timer), and referencePollTimer (timer for polling reference/evidence submissions). Each of these is paired with a set_ function. The resetState function is also imported; it is a comprehensive cleanup that clears every single timer and resets every variable in the state module back to its default value.

The ArenaView and CurrentDebate types are imported from arena-types.ts purely for type annotations. ArenaView is a union of 11 string literals representing every possible arena screen state. CurrentDebate is an interface carrying the full debate context: id, topic, the user's role (a or b), mode, current round, total rounds, opponent info, ranked status, messages array, moderator info, staking results, ruleset, language, and various flags for spectator/mod views and concession/nullification.

From the sibling arena modules, five cleanup functions are imported. clearQueueTimers from arena-queue.ts clears the two queue-related interval timers (the poll timer that checks for matches and the elapsed-seconds display timer). clearMatchAcceptTimers from arena-match.ts clears the match-accept countdown timer and its poll timer. stopOpponentPoll from arena-room-live.ts clears the opponent poll interval and resets the elapsed counter; this poll checks whether the opponent has submitted their turn. stopReferencePoll from arena-mod-refs.ts clears the reference poll interval and empties the pending references array. stopModStatusPoll from arena-mod-queue.ts clears the moderator status poll interval. All five of these are called during the popstate handler's cleanup of the room/preDebate view, because leaving a debate room mid-stream requires tearing down all active polling.

renderLobby and showPowerUpShop are imported from arena-lobby.ts. renderLobby is the primary view renderer for the lobby screen; it resets all configuration selections (mode, moderator, ranked, ruleset, category, want-mod), clears any lingering private lobby or power-up state, removes shield and silence overlays from the DOM, replaces the browser history state, and then builds the full lobby HTML (hero section, stats row, mode cards, arena feed, action buttons). showPowerUpShop renders the power-up shop overlay and is triggered by the ?shop=1 URL parameter. joinWithCode from arena-private-lobby.ts handles joining a private debate via a 6-character alphanumeric code; it calls the join_private_lobby RPC, sets up the match-found flow, and transitions the view. injectCSS from arena-css.ts creates a <style> element containing all arena CSS rules and appends it to the document head, but only once (guarded by the cssInjected flag in state). cleanupPendingRecording from voicememo.ts revokes any ObjectURL created for a pending voice memo recording blob, cancels any in-progress recording, and revokes all fallback URLs -- this prevents memory leaks from unreleased blob URLs. cleanupFeedRoom and enterFeedRoomAsSpectator are imported from arena-feed-room.ts; cleanupFeedRoom sends a goodbye message over the realtime channel, clears the feed turn timer, unsubscribes from the Supabase Realtime channel, cleans up Deepgram transcription resources, resets all feed room state, and clears reference/challenge state. enterFeedRoomAsSpectator calls the get_arena_debate_spectator RPC to load a debate's data and enters the feed room in read-only spectator mode.

With the imports fully mapped, the file's own code begins with three utility functions. isPlaceholder() returns true if the Supabase client is null or if any credentials are placeholders; this is used throughout the arena module to gate server calls so that demo/placeholder mode does not attempt real RPC calls that would fail. formatTimer(sec) converts a number of seconds into an "M:SS" string for display in countdown timers; it uses Math.floor for minutes and a ternary for zero-padding the seconds. randomFrom(arr) picks a random element from a readonly array using Math.floor(Math.random() * arr.length) and asserts non-null with the ! operator; this is used to select random AI debate topics and AI response templates.

The browser history section, annotated as a Session 121 rewrite, contains pushArenaState and the _onPopState handler. pushArenaState calls history.pushState with an object containing arenaView set to the given view name, enabling the browser back button to navigate within the arena's internal view system rather than leaving the page entirely. The _onPopState handler is the core of the back-button experience. When the user presses back, it first removes any overlay elements that might be floating in the DOM: the rank picker overlay, the ruleset picker overlay, the mode selection overlay, and the moderator ruling overlay (clearing the ruling countdown timer before removing it). Then it examines the current view state. If the user is in the room or preDebate view, it performs a full debate-exit cleanup: clearing the round timer and ruling countdown timer, stopping the reference poll, the opponent poll, and the mod status poll, and if the debate mode is live, it also calls cleanupFeedRoom (to tear down the realtime channel and transcription) and leaveDebate (to close WebRTC connections). If the user is in the queue view, it clears the queue timers and fires a leave_debate_queue RPC call (fire-and-forget with .catch logging) to tell the server to remove the user from the matchmaking queue. If the user is in the matchFound view, it clears the match accept timers and nulls out the matchFoundDebate state. Finally, if the current view is anything other than lobby, it calls renderLobby() to return the user to the lobby. This means all back-button navigation in the arena collapses to the lobby -- there is no multi-step back traversal through the arena's view hierarchy.

The _onPopState handler is immediately registered as a window popstate listener on line 90, which is a module-level side effect. This means importing arena-core.ts causes the listener to be registered. This is significant: any code that imports arena-core.ts, even transitively, will attach this popstate listener to the window. The destroy() function later removes it, but if destroy() is never called (e.g., in a single-page app that navigates away from the arena without explicit cleanup), the listener would persist and potentially fire on unrelated popstate events. The handler does check `view` state before acting, and if the view is already 'lobby' it does nothing, so the practical risk is low, but it is architecturally impure.

The init() function is the arena's entry point. It first checks FEATURES.arena and returns immediately if the feature is disabled. It then calls injectCSS(), which creates and appends the arena stylesheet if it has not already been injected. Next, it grabs the #screen-arena DOM element via getElementById and stores it in the state module via set_screenEl. If the element is not found, it logs a warning and returns, preventing any further rendering. If the element exists, it calls renderLobby() to build the initial lobby view. Then it checks for three URL query parameters. If ?shop=1 is present, it opens the power-up shop overlay on top of the lobby. If ?joinCode=XXXXXX is present (the F-39 challenge link feature), it strips the parameter from the URL using history.replaceState to clean the address bar, uppercases the code, and calls joinWithCode asynchronously. If ?spectate=<uuid> is present (Session 240 feature), it validates the UUID format with a regex and, if valid, strips the parameter and calls enterFeedRoomAsSpectator. Note that if both joinCode and spectate are present in the URL simultaneously, both would execute, which is likely a benign edge case since a user would not realistically have both, but it could produce confusing behavior if they did -- the joinWithCode call would start processing a private lobby join while enterFeedRoomAsSpectator simultaneously tries to load a spectator view. There is no mutual exclusion between these two branches.

The two public accessor functions, getView() and getCurrentDebate(), provide read-only access to the module state. getView() simply returns the view variable. getCurrentDebate() returns a shallow copy of currentDebate via the spread operator, or null if there is no active debate. The shallow copy is a defensive pattern that prevents external callers from mutating the state object directly, though it only protects the top level -- nested objects like the messages array are still shared references. For the messages array specifically, this means an external caller could push items into it and corrupt state, though this is unlikely to happen accidentally.

The destroy() function is the symmetric counterpart to init(). It first checks if a live debate is active and calls cleanupFeedRoom and leaveDebate if so. Then it calls cleanupPendingRecording to release any voice memo resources. Then it calls resetState(), which is the nuclear option from arena-state.ts: it iterates through every single mutable variable, clears every interval timer, and resets everything to defaults. This is thorough -- it handles queue timers, match accept timers, the round timer, all configuration selections, private lobby state, moderator state, reference and power-up state, voice memo state, opponent poll state, feed room state, and challenge state. Finally, destroy() removes the popstate event listener to prevent it from firing after the arena module is no longer active. Notably, destroy() does not call renderLobby() or clear the #screen-arena element's innerHTML -- it assumes the caller will handle replacing the screen content, which makes sense in a single-page app where navigating away from the arena means a different screen module takes over.

The auto-init block at line 161 is the final piece and arguably the most important: ready.then(() => init()).catch(() => init()). This makes the arena self-initializing when the module is first imported. The ready promise from auth.ts resolves after the authentication state machine has processed its initial session event. Whether auth succeeds or fails (e.g., if the user is not logged in, or if the 6-second timeout fires), init() runs. This matches the project's philosophy that guest users should see the full app. The void return of init() means any errors thrown inside init (such as a missing DOM element) would be swallowed by the promise chain -- but init itself does not throw, it only logs warnings and returns early, so this is fine in practice.

Regarding side effects: importing this module causes two immediate side effects. First, the _onPopState listener is registered on window at module evaluation time (line 90). Second, the ready.then().catch() chain is kicked off, which will asynchronously call init() once the auth promise settles. This means the arena starts up automatically just by being imported -- there is no need (and indeed no provision) for an external caller to trigger initialization. The init() export exists but is effectively redundant with the auto-init; it is exported presumably so that tests or other code could re-initialize the arena after a destroy(), but in normal operation the auto-init handles it.

There are several risks and potential bugs worth noting. First, the unused imports of getCurrentUser and getCurrentProfile from auth.ts are dead code. They should be removed for clarity. Second, the popstate handler's call to safeRpc('leave_debate_queue') on back-navigation from the queue view is fire-and-forget. If the network request fails, the server might keep the user in the queue for some time until a server-side timeout removes them. The catch handler logs a warning but there is no retry. Third, the popstate handler uses clearInterval with the ! (non-null assertion) operator on _rulingCountdownTimer and roundTimer, which is safe because clearInterval silently ignores null/undefined in all browsers, but the ! assertion is misleading to readers and could mask a null pointer in stricter future environments. Fourth, the URLSearchParams constructor is called three separate times in init() (lines 106, 110, 116), creating three separate URLSearchParams objects from the same window.location.search string. This is wasteful; a single variable could be reused. Fifth, the UUID regex on line 117 for the spectate parameter does not anchor with ^ and $... actually, looking again, it does use ^ and $ with the 'i' flag, so it correctly validates a full UUID v4 format. That is fine. Sixth, the shallow copy in getCurrentDebate means the messages array and any nested objects are shared references, which could theoretically allow state corruption from external code. Seventh, the destroy function does not reset screenEl to null, but resetState also deliberately skips screenEl (as noted in its comments -- "screenEl intentionally NOT reset -- it's a DOM ref set once by init()"). This means after destroy(), screenEl still points to a DOM element that may or may not still be in the document. If the arena is destroyed and the #screen-arena element is removed from the DOM, screenEl becomes a dangling reference, though this is harmless because no code runs against it after destroy(). Eighth, there is a potential race condition: if a user navigates to the arena page with both ?joinCode and ?spectate parameters, both async operations fire concurrently without coordination, and both call history.replaceState to strip their respective parameter, which means only the second replaceState wins (but since both strip to the bare pathname, the result is the same either way -- the issue is that both async flows would run in parallel and compete for control of the screen).

From the user's perspective, arena-core.ts orchestrates the following visible behaviors: when they navigate to the arena page, the lobby automatically appears after authentication settles (showing mode cards, stats, and the arena feed). If they arrived via a challenge link (?joinCode=...), they are automatically pulled into the private lobby join flow. If they arrived via a spectate link (?spectate=...), they enter spectator mode for that debate. If they arrived with ?shop=1, the power-up shop overlay appears on top of the lobby. When they press the browser back button at any point within the arena, they are returned to the lobby, with all active timers, polls, WebRTC connections, realtime subscriptions, and recording state cleaned up. When they navigate away from the arena entirely, the destroy function tears down everything, preventing resource leaks from orphaned intervals, blob URLs, or WebRTC connections.

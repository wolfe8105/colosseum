# Stage 2 Outputs — reference-arsenal.forge-wiring.ts

## Agent 01

### _wireForgeSheet

_wireForgeSheet reads the parameters: a container HTMLElement, state object (ForgeFormState), isEdit boolean, editRef ArsenalReference, and two callbacks (onComplete, onCancel, onRender). It queries the DOM for elements with specific IDs (forge-title, forge-author, forge-date, forge-locator, forge-url, forge-claim) using getElementById, casting each to HTMLInputElement or HTMLTextAreaElement. For each found element, it attaches an 'input' event listener that writes the current value into the corresponding state property (state.source_title, state.source_author, etc.). When forge-claim input fires, it additionally reads state.claim_text.length and writes that to the textContent of forge-claim-count element. The function then queries all elements with classes forge-source-btn and forge-cat-btn, attaching click listeners to each; if not in edit mode (isEdit is false), source button clicks read the data-source attribute and call onRender after writing to state.source_type; category button clicks read the data-cat attribute and call onRender after writing to state.category. It queries for navigation buttons (forge-back, forge-cancel, forge-next, forge-submit) and attaches click listeners: forge-back decrements state.step and calls onRender; forge-cancel invokes the onCancel callback; forge-next calls _validateStep with state, and if it returns true, increments state.step and calls onRender; forge-submit invokes _submitForge as an async operation with the state, isEdit, editRef, and onComplete callback. The function returns void. No error handling is present; if any queried element does not exist, its listener is simply not attached.

### _validateStep

_validateStep reads the state parameter and branches on state.step. If step is 1, it reads state.source_title, state.source_author, state.source_date, and state.locator, checking that source_title.trim() has length >= 2, source_author.trim() has length >= 2, source_date is truthy, and locator.trim() has length >= 1; on each failure, it calls showToast with an error message and returns false, otherwise returns true. If step is 2, it reads state.claim_text, checks that claim_text.trim() has length >= 5, calls showToast with an error message and returns false if the check fails, otherwise returns true. If step is 3, it reads state.source_type and checks that it is truthy, calling showToast and returning false if not, otherwise returning true. If step is 4, it reads state.category and checks that it is truthy, calling showToast and returning false if not, otherwise returning true. If state.step does not match 1, 2, 3, or 4, the function returns true. All validation failures result in showToast being called before returning false.

## Agent 02

### _wireForgeSheet

The function receives an HTMLElement container, a ForgeFormState object, boolean flags for edit mode and handler callbacks. It performs one-time wiring of event listeners to DOM elements and does not return anything. For each of five input elements queried by ID (forge-title, forge-author, forge-date, forge-locator, forge-url), if found, an input event listener is registered that updates the corresponding state property with the element's current value. A textarea with id forge-claim gets an input listener that updates state.claim_text and also writes the text length to the textContent of an element with id forge-claim-count. If isEdit is false, all elements matching the selector '.forge-source-btn' get click listeners that read their data-source attribute and assign it to state.source_type, then invoke onRender(). All elements matching '.forge-cat-btn' unconditionally receive click listeners that read their data-cat attribute, assign it to state.category, and invoke onRender(). Four navigation elements are wired: forge-back (click decrements state.step and calls onRender), forge-cancel (click calls onCancel), forge-next (click calls _validateStep, and if true increments state.step and calls onRender), and forge-submit (click calls _submitForge with the state, flags, and handlers, prefixed with void to suppress the promise). All listeners use optional chaining or null checks before registration, so missing elements do not cause errors.

### _validateStep

The function reads state.step and performs branch-wise validation on the state object, returning a boolean. For step 1, it validates that source_title and source_author each have at least 2 characters after trimming, that source_date is truthy, and that locator has at least 1 character after trimming; if any check fails, showToast is called with an error message and false is returned. For step 2, claim_text must be at least 5 characters after trimming; failure returns false after showing an error toast. For step 3, source_type must be truthy; failure returns false after an error toast. For step 4, category must be truthy; failure returns false after an error toast. If state.step is none of 1–4, the function returns true without validation. All error toasts use the string 'error' as the second argument to showToast.

## Agent 03

### _wireForgeSheet

This function wires up event listeners to DOM elements in a forge form, receiving an HTMLElement container, a ForgeFormState object, edit mode flags, an optional reference for editing, and two callbacks (onComplete and onCancel). It reads all form input elements by ID and attaches 'input' listeners to title, author, date, locator, and URL fields, each writing the current value into the corresponding state property. For the claim textarea (forge-claim), it additionally updates a character counter element. It reads the isEdit flag and conditionally attaches click listeners to source type buttons only if not in edit mode (preventing source type changes on existing references per LM-ARSENAL-002). It unconditionally attaches click listeners to all category buttons, both updating state and triggering a re-render via the onRender callback. For navigation: the back button decrements state.step and renders, cancel button invokes onCancel directly, next button calls _validateStep with state and only proceeds to increment step and render if validation passes, and the submit button calls _submitForge synchronously (with void to ignore the promise). All element selections use optional chaining or null checks, so missing DOM elements cause silent no-ops rather than errors. The function returns void and has no error handling.

### _validateStep

This function validates form inputs based on the current step in state and returns a boolean (true = valid, false = invalid). It reads the state.step property and switches through four validation branches. For step 1 (source details), it checks that source_title and source_author are trimmed to at least 2 characters each, that source_date is truthy, and that locator is trimmed to at least 1 character; on any failure, it calls showToast with an error message and returns false. For step 2 (claim), it validates that claim_text is trimmed to at least 5 characters; on failure, shows an error toast and returns false. For step 3 (source type), it checks that state.source_type is truthy; on failure, shows an error toast and returns false. For step 4 (category), it validates that state.category is truthy; on failure, shows an error toast and returns false. For any step outside 1-4, the function returns true. All error states are symmetric: failed validation always shows a toast with a user-facing message and returns false.

For step 1 (source details) it requires source_title and source_author to be trimmed and at least 2 characters, source_date to be truthy, and locator to be at least 1 character (lines 82–99); step 2 (claim) requires claim_text trimmed to at least 5 characters (lines 101–106); step 3 (source type selection) requires source_type to be truthy (lines 108–113); step 4 (category selection) requires category to be truthy (lines 115–120). ## Agent 05

### _wireForgeSheet

This function sets up event listeners on DOM elements within a form sheet for creating or editing arsenal references. It accepts a container element, a mutable form state object, flags indicating edit mode and the reference being edited, and callbacks for completion, cancellation, and re-rendering. The function performs no return value beyond void. It wires five input fields from Step 1 (source title, author, date, locator, URL) with 'input' event listeners that each directly mutate the corresponding state property when the user types. Step 2 wires a claim textarea with an 'input' listener that updates the claim text and synchronously updates a character count display element. Step 3 conditionally wires source-type buttons only when not in edit mode (per LM-ARSENAL-002), using click listeners to set state.source_type from the button's data attribute and trigger a re-render. Step 4 unconditionally wires category buttons with click listeners that set state.category and trigger re-render. Navigation listeners are then attached: a back button decrements state.step and re-renders, a cancel button invokes the onCancel callback, and a next button validates the current step via _validateStep—if validation passes, it increments state.step and re-renders, otherwise validation fails silently and no step advance occurs. Finally, a submit button click asynchronously invokes _submitForge with the current state, edit flag, edit reference, and completion callback. All event listeners are attached imperatively via querySelector and addEventListener without cleanup or removal logic.

### _validateStep

This function examines the current form step in the mutable state object and returns a boolean indicating whether the step's data passes validation. For step 1 (source details), it checks that source_title and source_author each have at least 2 non-whitespace characters, that source_date is truthy (not empty), and that locator has at least 1 non-whitespace character; any failure shows an error toast and returns false. For step 2 (claim), it verifies claim_text has at least 5 non-whitespace characters, returning false and showing an error toast if not. For step 3 (source type), it checks that state.source_type is truthy, failing with a toast if not. For step 4 (category), it checks that state.category is truthy, failing with a toast if not. Any other step value falls through to return true. Each validation failure calls showToast from config.ts with an error-level message before returning false.

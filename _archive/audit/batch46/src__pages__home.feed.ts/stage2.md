# Stage 2 Outputs — home.feed.ts

## Agent 01

### fetchLiveDebates

`fetchLiveDebates` is an async function that takes no parameters and returns a `Promise<LiveDebate[]>`. It begins by calling `getSupabaseClient()` to obtain a Supabase client instance. If the client is falsy, or if `getIsPlaceholderMode()` returns true, it immediately returns an empty array without making any network request. Inside a try/catch block, it performs a PostgREST query against the `arena_debates` table, selecting columns for both debates and joined profile data for `debater_a` and `debater_b` via two foreign key relationships (`profiles!arena_debates_debater_a_fkey` and `profiles!arena_debates_debater_b_fkey`). The query filters rows to those whose `status` is one of `'live'`, `'round_break'`, or `'voting'`, orders results by `created_at` descending, and limits results to 5 rows. This query is awaited. If the response contains an error or if `data` is null or undefined, the function returns an empty array. Otherwise it maps each raw row into a `LiveDebate` object: `id`, `topic`, `category`, `status`, `mode`, and `spectator_count` (defaulting to `0`), `current_round` (defaulting to `1`), and `max_rounds` (defaulting to `5`) are taken directly from the row; `debater_a_id` and `debater_b_id` are set from `d.debater_a` and `d.debater_b` with nullish coalescing to `null`; `debater_a_name` and `debater_b_name` are resolved from the nested profile objects, preferring `display_name` over `username`, and falling back to the string literals `'Debater A'` and `'Debater B'` respectively. If any exception is thrown during the try block, the catch block logs the error to the console via `console.error` and returns an empty array.

### renderFeed

`renderFeed` is an exported async function that takes no parameters and returns `Promise<void>`. It reads the DOM immediately: it calls `document.getElementById('screen-home')` and if that element is absent it returns early with no further action. It then looks for an existing element with id `'home-feed-container'`; if none exists, it creates a `div`, assigns that id, sets its `style.cssText` to `'width:100%;'`, and appends it to the `screen-home` element. Whether the container was found or created, it sets `feedEl.innerHTML` to a centered loading spinner markup, replacing any prior content. It then awaits `fetchLiveDebates()`, which performs the network query described above and returns an array of `LiveDebate` objects assigned to `liveDebates`. Next it awaits `ModeratorAsync.fetchTakes()`, a call into the async module whose return value is discarded. After both awaits complete, `renderFeed` iterates over `liveDebates` with a `for...of` loop, building an HTML string. For each debate, it appends a card `div` to `html` containing: a category label (the `category` field uppercased, defaulting to `'General'`, passed through `escapeHTML`); two debater name spans each passed through `escapeHTML` and each suffixed with the output of `bountyDot(debater_id)`, which is called with the debater's id; a topic line passed through `escapeHTML`; a metadata line displaying `spectator_count`, `current_round`, and `max_rounds` (each with numeric defaults, written directly into a template literal without an additional `Number()` cast — uncertainty: whether the raw values could be non-numeric is not visible from this file alone); and a "Watch Live" button with `data-debate-id` set to the debate id passed through `escapeHTML`. If `liveDebates` is empty, the loop body never executes and `html` remains an empty string. After the loop, it appends the string `'<div id="hot-takes-feed"></div>'` to `html`. It then sets `feedEl.innerHTML = html`, replacing the loading spinner with the fully built markup. Finally, it calls `ModeratorAsync.loadHotTakes('all')` as a fire-and-forget call — it is not awaited and its return value is discarded. There is no try/catch in `renderFeed` itself; any exception from `fetchLiveDebates`, `ModeratorAsync.fetchTakes`, or DOM operations would propagate to the caller.

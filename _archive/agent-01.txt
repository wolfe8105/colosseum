arena-core.ts is the bootstrap, teardown, and utility hub of the arena subsystem. It was extracted from what the comment header calls "the arena.ts monolith split," meaning it once lived inside a single massive arena.ts file and was later factored into arena-core.ts alongside sibling modules like arena-state.ts, arena-queue.ts, arena-match.ts, arena-room-live.ts, arena-mod-refs.ts, arena-mod-queue.ts, arena-lobby.ts, arena-private-lobby.ts, arena-css.ts, arena-feed-room.ts, and arena-types.ts. The file is 161 lines long and its job is fourfold: provide small utility functions that many sibling modules consume, manage browser history integration so the back button works sensibly in a single-page app, run the init sequence that boots the arena when the page loads, and run the destroy sequence that tears everything down when the user navigates away.

The imports at the top of the file pull from two categories: the broader application layer and the arena submodules themselves. From ../auth.ts, it imports safeRpc, which is a wrapper around supabase.rpc() that automatically retries on 401 errors (expired JWT, session refresh race); getSupabaseClient, which returns the live Supabase client instance or null if credentials are placeholders; getCurrentUser, which returns the currently authenticated Supabase user object or null for anonymous visitors; getCurrentProfile, which returns the profile row for the logged-in user; and ready, which is a Promise<void> that resolves once the auth subsystem has determined whether the visitor is logged in or anonymous --- it is built on Supabase's onAuthStateChange INITIAL_SESSION event and has a 6-second safety timeout. From ../config.ts it imports isAnyPlaceholder, a boolean that is true when either the Supabase or Stripe credentials are placeholder/demo values (meaning the app is running in a preview or development mode without a real backend), and FEATURES, which is a FeatureFlags object containing boolean toggles like arena, liveDebates, asyncDebates, and aiSparring. From ../webrtc.ts it imports leaveDebate, a function that tears down WebRTC peer connections, stops worker timers, and terminates any active audio/video streams --- this is the live audio debate infrastructure. From ../voicememo.ts it imports cleanupPendingRecording, which revokes any ObjectURLs created during voice memo recording and resets the recording state so no browser memory leaks persist.

From the arena submodules, the file imports a large swath of shared mutable state from arena-state.ts. Because ES module let bindings are read-only from importing modules (you cannot do "import { view } from './arena-state.ts'; view = 'lobby'" --- it will throw), arena-state.ts exports both the variables themselves (for reading via live bindings) and setter functions (set_view, set_screenEl, set_matchFoundDebate, etc.) for mutation. arena-core.ts imports the current values of view, currentDebate, roundTimer, _rulingCountdownTimer, matchFoundDebate, screenEl, queueElapsedTimer, queuePollTimer, vmTimer, and referencePollTimer for reading, plus the setters set_matchFoundDebate, set_screenEl, set_queueElapsedTimer, set_queuePollTimer, set_roundTimer, set_vmTimer, set_referencePollTimer, and set__rulingCountdownTimer for mutation. It also imports resetState, which is the nuclear reset function that clears every single timer (there are over a dozen setInterval handles tracked in arena-state.ts) and resets every mutable variable to its initial value. The type imports ArenaView and CurrentDebate come from arena-types.ts and are used only for type annotations. From the sibling modules it imports clearQueueTimers (arena-queue.ts), which clears the two queue polling intervals; clearMatchAcceptTimers (arena-match.ts), which clears the match accept countdown and polling intervals; stopOpponentPoll (arena-room-live.ts), which clears the interval that polls for the opponent's latest debate turn; stopReferencePoll (arena-mod-refs.ts), which clears the interval that polls for moderator reference rulings; stopModStatusPoll (arena-mod-queue.ts), which clears the interval that polls for moderator assignment status; renderLobby and showPowerUpShop (arena-lobby.ts), where renderLobby resets the view to "lobby" and renders the full lobby UI into the #screen-arena element, and showPowerUpShop opens a modal overlay for purchasing/equipping power-ups; joinWithCode (arena-private-lobby.ts), which handles joining a private debate lobby via a 6-character alphanumeric code; injectCSS (arena-css.ts), which dynamically injects the arena's stylesheet into the document head exactly once, guarded by the cssInjected flag; and cleanupFeedRoom and enterFeedRoomAsSpectator (arena-feed-room.ts), where cleanupFeedRoom tears down the feed room's realtime channel, timers, and state, and enterFeedRoomAsSpectator calls the get_arena_debate_spectator RPC and sets up the spectator view for a given debate ID.

The utility section begins with isPlaceholder, a function that returns true if there is no Supabase client (getSupabaseClient() returns null) or if isAnyPlaceholder is true. This is used throughout the arena subsystem as a guard: when the app is in placeholder/demo mode, destructive or network-dependent operations are skipped. formatTimer takes a number of seconds and returns a string like "2:05" --- it computes minutes via integer division, seconds via modulo, and zero-pads the seconds component. It is used in queue elapsed displays and round countdown timers across the arena. randomFrom is a generic function that picks a uniformly random element from a readonly array, used primarily to select random AI debate topics and random AI response templates.

The browser history section, rewritten in Session 121 according to the comment, manages the HTML5 History API so that pressing the browser back button during an arena session does something sensible rather than navigating away from the entire page. pushArenaState takes a view name string and calls history.pushState with an object containing { arenaView: viewName }, adding a new entry to the browser's history stack. This means that entering a queue, finding a match, or entering a debate room each push a history entry, and the user can press Back to return to the previous arena view.

The _onPopState handler is the listener for the popstate event, which fires when the user presses Back or Forward. It is both exported (for testing or for destroy to remove it) and immediately registered via window.addEventListener('popstate', _onPopState) on line 90, which is a module-level side effect --- the moment this module is imported by any other module, the popstate listener is attached to the window. The handler first removes any overlay elements that might be floating in the DOM: the rank selection overlay, the ruleset selection overlay, the mode selection overlay, and the moderator ruling overlay. For the ruling overlay specifically, it also clears the _rulingCountdownTimer interval, because that overlay has a live countdown that would leak if the overlay were simply removed from the DOM without clearing its timer.

The handler then examines the current value of view (imported as a live binding from arena-state.ts) to determine what cleanup is needed. If the view is "room" or "preDebate", the user is either in an active debate or in the pre-debate setup screen, so the handler clears the round timer interval, the ruling countdown timer interval, stops the reference poll (which checks for new reference rulings from the moderator), stops the opponent poll (which checks if the opponent has submitted their turn), and stops the mod status poll (which checks whether a moderator has been assigned). If the current debate is a live mode debate, it additionally calls cleanupFeedRoom to tear down the feed room's realtime Supabase channel and UI state, then calls leaveDebate to disconnect WebRTC. This is important: pressing Back during a live audio debate properly disconnects the user from the WebRTC session and notifies the server.

If the view is "queue", the handler clears the queue timers (the poll that checks for a match and the elapsed-time display counter), and if the app is not in placeholder mode, it fires a leave_debate_queue RPC call to the server. This RPC is fire-and-forget (the promise is caught but only logs a warning on failure), which is correct because the user is navigating away and there is no UI to show errors to. The RPC tells the server to remove this user from the matchmaking queue so they are not matched with someone after they have already left.

If the view is "matchFound", the handler clears the match accept timers (the countdown that gives the user N seconds to accept or decline, and the poll that checks whether the opponent has accepted) and nulls out the matchFoundDebate state.

Finally, if the view is anything other than "lobby", the handler calls renderLobby() to re-render the lobby. If the view is already "lobby", pressing Back does nothing within the arena (the browser's default behavior of navigating to the previous page-level history entry will take effect, which is correct).

The init function is the arena's entry point. It first checks FEATURES.arena and returns immediately if the arena feature flag is disabled --- this is the global kill switch for the entire arena subsystem. It then calls injectCSS(), which appends a <style> element containing the arena's CSS rules into the document head, but only on the first call (subsequent calls are no-ops because of the cssInjected guard in arena-css.ts). Next it looks up the DOM element with id "screen-arena" via document.getElementById and stores it in the screenEl state variable using set_screenEl. If no such element exists, it logs a warning and bails out --- this would happen if the HTML page does not contain the expected container div, meaning the arena cannot render anywhere. Then it calls renderLobby() to paint the initial lobby UI into that container.

After the lobby is rendered, init checks for three URL query parameters. If ?shop=1 is present, it calls showPowerUpShop() to immediately open the power-up shop modal on top of the lobby --- this is presumably linked from other parts of the app or from external marketing. If ?joinCode=XXXXXX is present (the F-39 feature, challenge links), it strips the query parameter from the URL using history.replaceState (so a page refresh does not re-trigger the join), uppercases the code, and calls joinWithCode asynchronously. joinWithCode will call the join_private_lobby RPC on the server, and if successful, will transition the user into the private lobby waiting room. If ?spectate=<debateId> is present (Session 240 feature), it validates the debate ID against a UUID regex pattern (standard 8-4-4-4-12 hex format, case-insensitive), strips the query parameter from the URL, and calls enterFeedRoomAsSpectator asynchronously, which fetches the debate data via RPC and renders the spectator feed room UI.

The public accessors section exports two functions. getView returns the current ArenaView value, providing controlled read access to the view state for modules that do not directly import from arena-state.ts. getCurrentDebate returns either a shallow copy of the currentDebate object (via spread) or null. The shallow copy is a defensive pattern --- it prevents external callers from mutating the internal state object --- though notably it is only a shallow copy, so nested objects like the messages array or _stakingResult could still be mutated by a caller if they were careless.

The destroy function is the inverse of init. It first checks if the current debate is a live mode debate and, if so, calls cleanupFeedRoom and leaveDebate to properly disconnect from the feed room realtime channel and the WebRTC session. It then calls cleanupPendingRecording from the voicememo module, which revokes any blob ObjectURLs that were created during voice memo recording, preventing memory leaks in the browser. Then it calls resetState(), which is the comprehensive reset function in arena-state.ts that iterates through every single timer handle, calls clearInterval on each one that is non-null, and resets every mutable state variable to its default value. Finally, it removes the popstate event listener from the window, which is critical because if the user navigates away from the arena page and comes back, a stale popstate listener from a previous init cycle would cause bugs.

The auto-init block at the bottom of the file, on line 161, is the mechanism that actually starts the arena when the module is first imported. It waits for the auth subsystem's ready promise to resolve and then calls init(). The .catch(() => init()) branch is notable: if the auth ready promise rejects (which would happen if auth initialization fails entirely), the arena still initializes. This is consistent with the project's design philosophy of "guest access is default" --- anonymous users should see the full app, and auth failures should not block the UI from rendering. The arena lobby will render, and any RPC calls that require authentication will fail gracefully via safeRpc's error handling.

Regarding side effects: this module has two significant side effects that execute at import time, before any explicit function call. First, line 90 attaches the _onPopState handler to the window's popstate event. This means that merely importing arena-core.ts from any other module will install a global event listener. Second, line 161 kicks off the auto-init sequence, which will (after auth resolves) inject CSS, find the screen element, render the lobby, and potentially auto-join a debate or open the shop based on URL parameters. These side effects are intentional --- the comment says "matches .js IIFE" meaning this replaces what was previously an immediately-invoked function expression in a plain JS file.

From the user's perspective, what this file orchestrates is the following: when the user navigates to the arena page, the auth system finishes initializing (or times out after 6 seconds), then the arena lobby appears with mode selection cards, a live debate feed, and navigation options. If the URL contains a joinCode parameter, the user is automatically taken into a private lobby to face a specific challenger. If it contains a spectate parameter with a valid debate UUID, they are dropped into a spectator view of an ongoing debate. If it contains shop=1, the power-up shop opens on top of the lobby. When the user presses the browser back button at any point during the arena flow --- whether in queue, at the match found screen, in a pre-debate screen, or in the debate room itself --- the popstate handler ensures proper cleanup happens (WebRTC disconnected, server notified of queue departure, timers cleared, overlays removed) and returns the user to the lobby. When the arena module is destroyed (the user navigates to a different page within the SPA), all timers are cleared, WebRTC is disconnected, voice memo recordings are cleaned up, and the popstate listener is removed.

Now for bugs and risks. The most notable concern is the shallow copy in getCurrentDebate. The spread operator on line 132 creates a new object with the same top-level properties, but the messages array inside CurrentDebate is an array of objects, and the _stakingResult is a nested object. Any caller that mutates debate.messages.push(...) or modifies debate._stakingResult.someField would be mutating the internal state. This is a latent bug waiting to happen if any consumer of getCurrentDebate treats the returned object as safely mutable.

There is a minor race condition possibility in the _onPopState handler. Between the time view is read on line 67 and the time the cleanup functions execute, another asynchronous operation could theoretically change the view state. In practice this is unlikely because popstate fires on the main thread and JavaScript is single-threaded, but if any of the cleanup functions (like leaveDebate or cleanupFeedRoom) are themselves asynchronous and yield, other microtasks could interleave. Examining the code, leaveDebate appears synchronous (it calls stopWorkerTimer and terminateWorkerTimer), and cleanupFeedRoom likely synchronously clears timers and unsubscribes from the realtime channel, so this is more of a theoretical concern than a practical one.

The leave_debate_queue RPC call on line 80 uses .catch() to swallow errors, which is correct for a fire-and-forget cleanup call, but if the network is completely down, the user will be silently left in the server-side queue. The server presumably has its own timeout to clean up stale queue entries, but if it does not, the user might find themselves matched with someone the next time the poll runs on the server side, even though they have already navigated away. This is a resilience concern rather than a bug in this file specifically.

The UUID validation regex on line 117 for the spectate parameter is correct and comprehensive, matching the standard 8-4-4-4-12 hex UUID format with case insensitivity. However, the joinCode parameter on line 111 has no validation at all --- whatever string the URL contains is uppercased and passed directly to joinWithCode. If joinWithCode passes this to an RPC, it will be sent to the server as-is. The server-side RPC presumably validates the code format, but if it does not, arbitrary strings could be injected into the query. Since the project mandates that all mutations go through RPC calls (the Castle Defense pattern), and since the server function is SECURITY DEFINER, the risk is mitigated as long as the server-side function properly validates and parameterizes the input.

The auto-init on line 161 has no guard against being called twice. If arena-core.ts is somehow imported, then destroyed, then a new dynamic import triggers re-evaluation of the module, init would run again. In practice ES modules are cached by the runtime and only evaluated once, so this is not a real concern in standard usage, but it is worth noting that init() itself has no idempotency guard (it does not check whether the arena has already been initialized). The injectCSS call inside init is idempotent (guarded by cssInjected), and renderLobby presumably overwrites the innerHTML of screenEl, so double-init would result in a double-render but no catastrophic failure.

One subtle issue: the destroy function removes the popstate listener on line 154, but init does not re-add it. The listener is added at module evaluation time (line 90), which only runs once. So if destroy() is called and then init() is called again (for example, if the SPA re-enters the arena page), the popstate listener will be gone and the back button will no longer work properly within the arena. This is likely not a problem if the SPA navigation causes a full page reload or a re-import of the module, but if the arena is treated as a component that can be mounted and unmounted within the same page lifecycle, this is a genuine bug.

The imports of getCurrentUser and getCurrentProfile from auth.ts on line 5 are notable because they are never used within arena-core.ts itself. They are imported but not referenced in any function body in this file. This suggests they were either used in an earlier version of the code and the import was not cleaned up, or they are re-exported from this module (though the file does not contain any re-export statements). This is dead import code and a minor code quality issue --- the TypeScript compiler or a linter configured with no-unused-imports would flag it.

Finally, the formatTimer function on line 40 does not handle negative inputs or non-integer inputs. If sec is negative, Math.floor(sec / 60) would produce a negative minute value and sec % 60 would produce a negative remainder, resulting in output like "-1:-05". If sec is NaN, the output would be "NaN:0NaN". In practice the callers likely always pass non-negative integers, but defensive programming would add a Math.max(0, Math.floor(sec)) guard.

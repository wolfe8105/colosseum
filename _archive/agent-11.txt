COMPREHENSIVE ANALYSIS OF src/arena/arena-core.ts
==================================================

This file is the orchestration spine of the Arena subsystem -- the part of The Moderator that handles live debates, voice memo debates, text battles, and AI sparring. It was extracted from a monolith ("arena.ts") into a set of smaller modules, and arena-core.ts serves as the bootstrap, teardown, browser-history manager, and utility provider for the entire arena module family. Every other arena-* file depends on this one for initialization and cleanup, while this file in turn depends on nearly every sibling module to coordinate their lifecycle. The file is 161 lines long and is structured into five logical zones: imports, utility helpers, browser history management, init/destroy lifecycle, and auto-init.

The file opens with a block of imports spanning lines 4 through 31. From auth.ts it pulls safeRpc (the project's universal RPC wrapper that handles 401 retries and error normalization for all Supabase .rpc() calls), getSupabaseClient (which returns the live Supabase JS client instance or null if credentials are placeholder), getCurrentUser (returns the authenticated user object or null), and getCurrentProfile (returns the cached profile row). It also imports ready from auth.ts on a separate line -- ready is a Promise<void> that resolves once the auth system has completed its INITIAL_SESSION flow, meaning the user's JWT has been validated or the guest state has been established. From config.ts it imports isAnyPlaceholder, a boolean that is true when either the Supabase URL or the Stripe key is a placeholder string (indicating the app is running in a preview/demo mode without real backend connectivity), and FEATURES, the feature flag object where FEATURES.arena gates whether the entire arena subsystem should initialize at all.

From webrtc.ts it imports leaveDebate, which tears down a WebRTC peer connection by stopping all media tracks, terminating the worker timer, and signaling the server that this participant has left. This is the nuclear cleanup for any live audio debate. From arena-state.ts it imports a large set of mutable state variables and their corresponding setter functions. The state module uses ES module let bindings to hold all arena state -- things like the current view (lobby, queue, matchFound, room, etc.), timer handles for queue polling, match acceptance countdowns, round timers, voice memo timers, reference polling, ruling countdowns, and many more. Because ES module let bindings are read-only from the perspective of importing modules, each variable has a corresponding set_ function that mutates the variable within the state module and relies on live binding propagation so all importers see the updated value. The specific imports here are: view (the current arena screen the user is seeing), currentDebate (the active debate object or null), roundTimer (the setInterval handle for round countdown), _rulingCountdownTimer (interval handle for moderator ruling countdown), matchFoundDebate (the debate object for a pending match), screenEl (the DOM element #screen-arena), queueElapsedTimer (interval counting seconds in queue), queuePollTimer (interval polling the server for a match), vmTimer (voice memo recording timer), and referencePollTimer (interval polling for new reference submissions). The setters for all of these are also imported so arena-core can null them out during cleanup.

From arena-types.ts it imports the ArenaView and CurrentDebate types. ArenaView is a string union of all possible arena screens: 'lobby', 'modeSelect', 'queue', 'matchFound', 'room', 'preDebate', 'postDebate', 'privateLobbyWaiting', 'modQueue', 'modDebatePicker', 'modDebateWaiting'. CurrentDebate is an interface carrying everything about an active debate -- its id, topic, the user's role (a or b), mode, round number, total rounds, opponent info, moderator info, messages array, staking results, ruleset, and various flags for spectator/mod/conceded/nulled states.

From arena-queue.ts it imports clearQueueTimers, a function that clears both the queuePollTimer and queueElapsedTimer intervals and nulls them via the state setters. From arena-match.ts it imports clearMatchAcceptTimers, which clears the matchAcceptTimer (the countdown for the user to accept a found match) and matchAcceptPollTimer (polling whether the opponent has also accepted). From arena-room-live.ts it imports stopOpponentPoll, which clears the opponentPollTimer (the interval that checks whether the opponent has submitted their turn) and resets the elapsed counter. From arena-mod-refs.ts it imports stopReferencePoll, which clears the referencePollTimer (interval polling for new evidence/reference submissions from either debater) and empties the pending references array. From arena-mod-queue.ts it imports stopModStatusPoll, which clears the modStatusPollTimer (interval checking whether a human moderator has accepted the moderation request for a debate).

From arena-lobby.ts it imports renderLobby, the function that renders the main arena lobby screen (the landing page showing debate modes, recent debates, leaderboard preview, etc.), and showPowerUpShop, an async function that renders the power-up shop overlay where users can equip items before a debate. From arena-private-lobby.ts it imports joinWithCode, an async function that takes a six-character uppercase join code and attempts to join a private debate lobby by calling the appropriate RPC. From arena-css.ts it imports injectCSS, a function that creates a <style> element and appends arena-specific CSS to the document head, guarded by a cssInjected boolean so it only runs once per page load. From voicememo.ts it imports cleanupPendingRecording, which revokes any ObjectURL created for a pending voice memo recording and cancels any active MediaRecorder. From arena-feed-room.ts it imports cleanupFeedRoom (which tears down the Supabase Realtime channel for the feed room, clears the feed turn timer, and sends a disconnect notification to the opponent) and enterFeedRoomAsSpectator (which calls the get_arena_debate_spectator RPC and renders the spectator view of a live feed-room debate).

The first utility function is isPlaceholder (line 36), which returns true if getSupabaseClient() returns null (meaning no client was initialized) or if isAnyPlaceholder is true (meaning credentials are placeholder strings). This is used throughout the arena to short-circuit functionality that requires a real backend -- in placeholder mode, the UI still renders but server calls are skipped or stubbed. The function is exported so every arena submodule can call it.

formatTimer on line 40 takes a number of seconds and returns a string in M:SS format. It divides by 60, floors for minutes, takes modulo 60 for seconds, and zero-pads the seconds with a conditional '0' prefix if less than 10. This is used for queue elapsed time display, round countdown timers, voice memo recording duration, and anywhere else the user sees a ticking clock.

randomFrom on line 46 is a generic function that takes a readonly array and returns a random element using Math.random. It's used primarily for selecting random AI debate topics and random AI response templates during AI sparring mode. The non-null assertion (!) at the end is technically unsafe if an empty array is passed -- it would return undefined cast as T -- but in practice the callers always pass non-empty const arrays like AI_TOPICS.

The browser history section starting at line 54 implements SPA-style navigation for the arena. pushArenaState takes a view name string and calls history.pushState to add a new history entry with an arenaView property in the state object. This means that when the user transitions from lobby to queue to matchFound to room, each transition pushes a history entry so the browser back button can unwind through them.

The _onPopState handler (lines 58-89) is the heart of the back-button experience. When the browser fires a popstate event (user hits back, or history.back() is called), this handler runs a multi-phase cleanup. First, it removes any overlay elements that might be floating in the DOM -- the rank selection overlay, ruleset selection overlay, mode selection overlay, and the moderator ruling overlay. For the ruling overlay specifically, it also clears the _rulingCountdownTimer interval before removing the DOM element, because that overlay has a live countdown that would otherwise leak.

Next, the handler checks what view the arena is currently in and performs view-specific cleanup. If the user is in the 'room' or 'preDebate' view (meaning they are actively in a debate or about to start one), it clears the roundTimer interval, clears the _rulingCountdownTimer interval, stops the reference poll, stops the opponent poll, and stops the mod status poll. If the current debate's mode is 'live', it additionally calls cleanupFeedRoom (to tear down the Realtime channel and notify the opponent) and leaveDebate (to close the WebRTC connection). This is the critical cleanup path -- without it, leaving a debate via the back button would leave orphaned intervals polling the server, an open WebRTC connection streaming audio, and a Realtime subscription consuming resources.

If the user is in the 'queue' view, the handler clears the queue timers and, if we have a real backend (not placeholder), calls safeRpc('leave_debate_queue') to tell the server to remove this user from the matchmaking queue. The .catch() on that RPC swallows errors with a console.warn because the user is navigating away and there's nothing useful to show them if the dequeue fails.

If the user is in the 'matchFound' view (the "Match Found! Accept?" screen), the handler clears the match accept timers and nulls the matchFoundDebate state. Notably, it does NOT call an RPC to decline the match -- the server's match acceptance has its own timeout, so if the user just backs out, the server-side timeout will handle it.

Finally, if the current view is anything other than 'lobby', the handler calls renderLobby() to return the user to the lobby. The conditional prevents a redundant re-render if the user is already at the lobby and hits back (which would go to whatever page was before the arena entirely, handled by the browser's normal navigation).

Line 90 is a module-level side effect: window.addEventListener('popstate', _onPopState). This means that as soon as this module is loaded (imported), the popstate listener is registered globally. This is important because it means the listener is active for the entire lifetime of the page, not just when the arena is visible. The destroy() function on line 154 removes this listener to prevent double-handling if the arena is re-initialized.

The init function (lines 96-121) is the arena's entry point. It first checks FEATURES.arena and returns immediately if the arena feature flag is off -- this is the kill switch for disabling the entire arena in one place. It then calls injectCSS() to append arena styles to the document (idempotent, guarded internally). It grabs the #screen-arena DOM element and stores it in the state module via set_screenEl. If that element doesn't exist (meaning the HTML page doesn't have the arena container), it logs a warning and bails out -- this prevents the arena from crashing on pages that don't have the arena markup. It then calls renderLobby() to render the initial lobby view.

After rendering, init checks URL query parameters for three auto-action scenarios. First, if ?shop=1 is present, it calls showPowerUpShop() to immediately open the power-up shop overlay -- this allows deep-linking directly to the shop. Second, if ?joinCode=XXXXXX is present (the F-39 challenge link feature), it strips the query parameter from the URL using history.replaceState (to prevent re-triggering on refresh), uppercases the code, and calls joinWithCode to auto-join a private debate lobby. Third, if ?spectate=<uuid> is present (Session 240 feature), it validates the UUID format with a regex, strips the query parameter, and calls enterFeedRoomAsSpectator to drop the user directly into spectating a live debate. The UUID regex check is a security measure that prevents injection of arbitrary strings into the RPC call.

The getView and getCurrentDebate functions (lines 127-133) are public accessors that other modules can use to inspect arena state without directly importing the state module. getCurrentDebate returns a shallow copy of the currentDebate object (via spread) to prevent external mutation, or null if no debate is active. This is a defensive pattern -- callers get a snapshot, not a live reference.

The destroy function (lines 139-155) is the teardown counterpart to init. It first checks if there's an active live debate and, if so, calls cleanupFeedRoom and leaveDebate to properly exit. It then calls cleanupPendingRecording to revoke any ObjectURLs from voice memo recordings (preventing memory leaks). It then calls resetState(), which is the nuclear option defined in arena-state.ts -- it clears every single interval timer in the arena (queue poll, queue elapsed, match accept, match accept poll, round timer, private lobby poll, mod queue poll, mod status poll, mod debate poll, reference poll, silence timer, ruling countdown, vm timer, opponent poll, feed turn timer, challenge ruling timer), nulls them all out, resets all mutable variables to their defaults (view back to 'lobby', all selections cleared, all flags reset), and handles the feed room state. Finally, destroy removes the popstate listener to prevent it from firing after the arena is torn down.

The auto-init block on line 161 is the final module-level side effect: ready.then(() => init()).catch(() => init()). This means that when this module is loaded, it waits for the auth system's ready promise to resolve (meaning auth state is known) and then calls init(). The .catch(() => init()) ensures that even if the auth ready promise rejects (which could happen if auth initialization times out or errors), init still runs. This is important because the arena should still render in guest mode even if auth fails -- the lobby is viewable without authentication. The consequence of this pattern is that init() will always be called exactly once per module load, but the timing depends on how long auth takes to resolve.

Regarding side effects at module load time, there are two: the popstate listener registration on line 90 and the auto-init on line 161. The popstate listener is registered synchronously during module evaluation, meaning it's active immediately when the module is first imported. The auto-init is asynchronous, waiting for the auth ready promise. This means there's a brief window where the popstate listener is active but init hasn't run yet -- during this window, if the user somehow triggers a popstate event, the handler would check view (which defaults to 'lobby' in arena-state.ts) and find it's already 'lobby', so it would be a no-op. This is safe.

From the user's perspective, what they see when the arena initializes is: the #screen-arena element fills with the lobby UI showing debate mode cards (Live, Voice Memo, Text Battle, AI Sparring), recent debates, and navigation options. If they arrived via a ?joinCode link, they're immediately taken into a private lobby flow. If they arrived via ?spectate=uuid, they're dropped into a live spectator view. If ?shop=1 is set, the power-up shop overlay appears on top of the lobby.

When the user navigates through the arena (picking a mode, entering queue, getting matched, entering a debate room), each transition pushes a history state. When they hit the back button, the popstate handler gracefully tears down whatever state they were in -- stopping polls, leaving WebRTC, dequeuing from matchmaking -- and returns them to the lobby. When the arena is destroyed (e.g., the user navigates to a different section of the app), all timers are cleared, all connections are closed, and the popstate listener is removed.

There are several bugs and risks worth noting. First, on line 64, the handler clears _rulingCountdownTimer with clearInterval but uses a non-null assertion (!) -- if _rulingCountdownTimer is null, clearInterval(null) is harmless in browsers (it's a no-op), so this isn't a crash risk, but the assertion is misleading since the value can absolutely be null. The same pattern appears on lines 68-69 with roundTimer and _rulingCountdownTimer again.

Second, there is a potential race condition in the popstate handler's queue cleanup on line 80. The handler calls safeRpc('leave_debate_queue') fire-and-forget, but then immediately allows renderLobby() to run. If the user quickly re-enters the queue before the leave_debate_queue RPC completes, the server might see conflicting state -- the leave hasn't finished but a new join is starting. In practice this is likely mitigated by the server's queue logic (probably using upsert or checking for existing entries), but it's a theoretical race.

Third, the auto-init on line 161 calls init() in both the resolved and rejected paths of the ready promise. If init() throws (e.g., due to a DOM error), that exception will be unhandled because neither .then() nor .catch() has its own error handler. A thrown error during init would become an unhandled promise rejection, which would silently fail in most browsers but could be noisy in development.

Fourth, the getCurrentDebate accessor on line 131 returns a shallow copy via spread, but the CurrentDebate interface contains a messages array (DebateMessage[]) and potentially other nested objects. A shallow copy means the caller gets a new top-level object but shares the same messages array reference. If a caller mutates the messages array (e.g., pushing a new message), it would mutate the original state. This is a partial defense at best -- a deep clone or Object.freeze would be more robust, though in practice the callers probably don't mutate the returned object.

Fifth, the isPlaceholder function is used as a guard in the popstate handler (line 80) to skip the leave_debate_queue RPC in placeholder mode, which is correct. But if a user is in a real session and their Supabase client somehow gets nulled (e.g., due to a CDN error reloading the page), isPlaceholder would return true and the dequeue RPC would be skipped, leaving the user stuck in the server-side queue until it times out. This is an edge case but worth noting.

Sixth, there is an inconsistency in how the popstate handler handles cleanup for different views. For 'room' and 'preDebate' views, it calls five different stop functions individually. For 'queue', it calls clearQueueTimers. But for 'matchFound', it only clears match accept timers and nulls the debate -- it doesn't call an RPC to decline the match or notify the server. While the server's timeout handles this eventually, it means the opponent could be left waiting for the full acceptance timeout (12-15 seconds based on the constants in arena-types.ts) before being told the match fell through. This is a UX issue for the opponent, not a bug per se.

Seventh, the URL parameter parsing creates three separate URLSearchParams instances on lines 106, 110, and 116. This is mildly wasteful -- a single parse could be reused -- but has no functional impact.

Eighth, the import of getCurrentUser and getCurrentProfile from auth.ts on line 5 is notable because neither function is actually used anywhere in this file. They are imported but never referenced. This is dead import code that Vite's tree-shaking should eliminate during production builds, so it has no runtime cost, but it's untidy.

In summary, arena-core.ts is the lifecycle manager for the entire arena feature. It boots up on auth ready, injects CSS, renders the lobby, handles deep-link parameters, manages browser history with proper cleanup on back-navigation, and provides a comprehensive destroy function that tears down every timer, connection, and subscription the arena may have created. It coordinates across roughly ten sibling modules, each responsible for a specific arena subsystem (queue, match acceptance, live room, mod queue, mod references, feed room, private lobby, voice memo, power-ups, and CSS). The file's most critical responsibility is ensuring no resource leaks occur when the user navigates away -- orphaned intervals, open WebRTC connections, or abandoned Realtime subscriptions would degrade performance and potentially cause server-side issues. The cleanup is thorough if somewhat redundant (the popstate handler cleans up view-specific resources, then destroy also calls resetState which clears everything again), but redundancy in cleanup is a feature, not a bug.

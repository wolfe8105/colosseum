arena-core.ts is the bootstrap, teardown, and utility spine of the entire arena subsystem. It is not a large file -- roughly 160 lines -- but it is the file that everything else in the arena directory depends on, and every line has consequences that ripple across at least half a dozen sibling modules. What follows is a continuous, detailed walkthrough of everything the file does, where every call goes, what it imports, what side effects it causes, and what risks lurk inside it.

The file opens with its imports, which fall into four categories. From auth.ts it pulls safeRpc (the project-wide wrapper around supabase.rpc that retries on 401), getSupabaseClient (returns the live Supabase JS client instance or null if in placeholder mode), getCurrentUser and getCurrentProfile (return the cached auth user and profile objects respectively -- these are not used directly in arena-core.ts but are imported nonetheless, which is the first minor observation: getCurrentUser and getCurrentProfile are imported but never referenced in this file, making them dead imports that Vite's tree-shaking will eliminate in production but that add noise for anyone reading the source). From config.ts it imports isAnyPlaceholder, a boolean that is true when either the Supabase or Stripe credentials are placeholder/demo values, and FEATURES, the feature flag object that gates major subsystems. From webrtc.ts it imports leaveDebate, which tears down WebRTC peer connections, stops the worker timer that tracks debate timing offscreen, and terminates the audio pipeline -- this is the "hang up the call" function for live audio debates. From auth.ts again (a second import statement, which is legal but slightly unusual) it imports ready, a Promise<void> that resolves once the auth subsystem has determined whether the user is logged in or anonymous; this promise is the heartbeat of the entire app's initialization sequence and has a 6-second safety timeout.

From the arena-state.ts sibling module the file imports a large batch of live-binding variables and their setter functions. ES module semantics make exported let bindings read-only from importing modules, so the project uses a setter-function pattern: view (the current arena screen, typed as ArenaView -- one of lobby, modeSelect, queue, matchFound, room, preDebate, postDebate, privateLobbyWaiting, modQueue, modDebatePicker, or modDebateWaiting), currentDebate (the full state object for whatever debate is in progress, or null), roundTimer (the setInterval handle for the per-round countdown clock), _rulingCountdownTimer (the setInterval handle for the moderator-ruling countdown overlay), matchFoundDebate (the debate data object shown during the match-accept screen), screenEl (the #screen-arena DOM element that serves as the arena's root container), queueElapsedTimer and queuePollTimer (interval handles for the matchmaking queue), vmTimer (voice memo recording timer), and referencePollTimer (interval handle for polling reference/evidence status). Alongside each variable, the corresponding set_ function is imported so arena-core.ts can mutate state from outside the declaring module. The resetState function is also imported -- it is the nuclear option that clears every single interval timer and resets every mutable variable back to its default value.

From arena-types.ts the file imports the ArenaView and CurrentDebate types for type annotations. From arena-queue.ts it imports clearQueueTimers, which stops the two queue-related intervals (queuePollTimer and queueElapsedTimer) via clearInterval and nulls them out through their setters. From arena-match.ts it imports clearMatchAcceptTimers, which does the same for matchAcceptTimer and matchAcceptPollTimer -- these are the countdown and polling intervals that run during the "match found, do you accept?" screen. From arena-room-live.ts it imports stopOpponentPoll, which clears the opponentPollTimer interval that checks whether the opponent has submitted their turn in text/AI mode and resets opponentPollElapsed to zero. From arena-mod-refs.ts it imports stopReferencePoll, which clears referencePollTimer and empties the pendingReferences array. From arena-mod-queue.ts it imports stopModStatusPoll, which clears the modStatusPollTimer that checks whether a moderator has been assigned to a debate. From arena-lobby.ts it imports renderLobby (which replaces the contents of screenEl with the full lobby UI -- the hero banner, enter/private/mod buttons, the live debate feed, and pending challenge cards) and showPowerUpShop (which renders the power-up purchase interface). From arena-private-lobby.ts it imports joinWithCode, an async function that takes a six-character alphanumeric challenge code, calls the join_private_lobby RPC, and transitions the user into the matched debate room. From arena-css.ts it imports injectCSS, a function that creates a <style> element containing all arena-specific CSS rules and appends it to the document head, guarded by the cssInjected flag so it only runs once per page load. From voicememo.ts it imports cleanupPendingRecording, which revokes any ObjectURLs from in-progress voice memo recordings and cancels any active MediaRecorder, preventing memory leaks from orphaned blob URLs. From arena-feed-room.ts it imports cleanupFeedRoom, which sends a disconnect feed event to the opponent, unsubscribes the Supabase Realtime channel, clears the feedTurnTimer, and nulls out all feed-room state, and enterFeedRoomAsSpectator, which loads a debate's data via the get_arena_debate_spectator RPC and renders the spectator view of a live moderated debate.

After the imports, the file defines three small utility functions. isPlaceholder returns true if getSupabaseClient() returns null (meaning the Supabase client was never initialized, which happens in certain SSR or testing contexts) or if isAnyPlaceholder is true (meaning the credentials are demo values). This function is exported and consumed by nearly every other arena module as a gate before making RPC calls -- if you are in placeholder mode, writes are silently skipped. formatTimer takes a number of seconds, computes minutes and seconds via Math.floor and modulo, and returns a string like "2:07" with zero-padded seconds. This is used for the queue elapsed timer, the round countdown timer, and the voice memo recording timer displays. randomFrom is a generic function that takes a readonly array and returns a random element using Math.floor(Math.random() * arr.length) with a non-null assertion. It is used to pick random AI debate topics and random AI response templates.

The browser history section comes next. pushArenaState takes a view name string and calls history.pushState with an object containing an arenaView property. This creates a new browser history entry each time the user navigates deeper into the arena (from lobby to queue, from queue to match found, etc.), so that the browser back button can return them to the lobby. The _onPopState handler is an arrow function assigned to a const and exported (the underscore prefix is a convention in this codebase indicating "semi-private, exported only for testing or cross-module access"). This handler fires whenever the user presses the browser back button or calls history.back(). It first removes any overlay elements that might be floating on top of the arena -- the rank picker overlay, the ruleset overlay, the mode selection overlay, and the moderator ruling overlay. If the ruling overlay exists, it also clears the _rulingCountdownTimer interval before removing it, which prevents a dangling interval from firing into a removed DOM element. Then it checks the current view state. If the user is in the room or preDebate view, it clears the roundTimer interval, clears the _rulingCountdownTimer interval, stops the reference poll, stops the opponent poll, and stops the mod status poll. If the current debate is a live-mode debate, it additionally calls cleanupFeedRoom (which sends the disconnect event and tears down the Realtime channel) and leaveDebate (which closes WebRTC connections). If the user is in the queue view, it clears queue timers and, unless in placeholder mode, fires a fire-and-forget leave_debate_queue RPC call to remove the user from the server-side matchmaking queue, logging any failure to the console as a warning rather than surfacing it to the user. If the user is in the matchFound view, it clears the match accept timers and nulls out matchFoundDebate. Finally, if the current view is anything other than lobby, it calls renderLobby to put the user back at the arena's home screen. The handler is immediately registered on the window's popstate event via addEventListener on line 90, which is a top-level side effect -- as soon as this module is imported by any other module, the popstate listener is installed. This is intentional: the arena system needs to intercept back-button presses from the moment the arena code loads.

The init function is the arena's entry point. It first checks the FEATURES.arena flag and returns immediately if the arena feature is disabled, preventing any DOM manipulation or event wiring. If the feature is enabled, it calls injectCSS, which creates and appends a <style> tag to document.head containing all arena CSS (hover states, animations, layout rules, the frosted-glass card styling, responsive breakpoints). It then grabs the #screen-arena DOM element via getElementById and stores it via set_screenEl. If the element does not exist (which would happen if the HTML page does not have a div with that ID), it logs a warning and returns early -- the arena simply will not render. If the element is found, it calls renderLobby to populate it with the lobby UI. After rendering the lobby, init checks the URL search parameters for three special query string keys. If shop=1 is present, it calls showPowerUpShop to immediately overlay the power-up store on top of the lobby -- this is a deep-link mechanism so other parts of the app or marketing emails can link directly to the shop. If joinCode is present, it strips the query parameter from the URL using history.replaceState (to prevent the code from being re-consumed on refresh), uppercases the code, and calls joinWithCode in a void-fire-and-forget fashion to immediately attempt joining a private lobby. If spectate is present and passes a UUID regex validation (the standard 8-4-4-4-12 hex pattern with case-insensitive matching), it similarly strips the parameter and calls enterFeedRoomAsSpectator to drop the user directly into spectator mode for that debate. The UUID validation regex here is a defensive measure against injection of arbitrary strings into the RPC call.

The public accessors section exports getView and getCurrentDebate. getView simply returns the current value of the view variable from arena-state. getCurrentDebate returns either null or a shallow copy of the currentDebate object (via spread). The spread is significant -- it prevents callers from mutating the debate state through the returned reference. However, this is a shallow copy, so the messages array (an array of DebateMessage objects) inside CurrentDebate is still shared by reference. If any caller were to push into or splice the returned messages array, it would mutate the canonical state. In practice this does not appear to happen in the codebase, but it is a latent risk.

The destroy function is the inverse of init. It is called when the user navigates away from the arena page (for example, switching to the settings or leaderboard screen via the navigation system). It first checks if the current debate is a live-mode debate, and if so calls cleanupFeedRoom and leaveDebate to gracefully exit the audio/feed session. It then calls cleanupPendingRecording from the voicememo module to revoke any dangling ObjectURLs and cancel any in-progress MediaRecorder. It calls resetState, which is the comprehensive teardown function from arena-state.ts: resetState clears every single interval timer in the arena system (queuePollTimer, queueElapsedTimer, matchAcceptTimer, matchAcceptPollTimer, roundTimer, privateLobbyPollTimer, modQueuePollTimer, modStatusPollTimer, modDebatePollTimer, referencePollTimer, silenceTimer, _rulingCountdownTimer, vmTimer, opponentPollTimer, feedTurnTimer, challengeRulingTimer), nulls out all object references (currentDebate, matchFoundDebate, selectedModerator, etc.), resets all numeric counters to zero, resets booleans to false, resets selectedRuleset to 'amplified', resets selectedRounds to the default from DEBATE.defaultRounds, clears the activatedPowerUps set, empties the equippedForDebate and pendingReferences arrays, and resets challengesRemaining to 3. Notably, resetState does NOT reset screenEl or cssInjected, because the DOM element persists and the CSS only needs to be injected once per page load. Finally, destroy removes the popstate event listener by calling window.removeEventListener with the same _onPopState reference, unregistering the back-button handler.

The auto-init section at the bottom of the file is the final and arguably most important piece. The expression ready.then(() => init()).catch(() => init()) runs init() once the auth ready promise settles, regardless of whether it resolves (user is logged in or anonymous -- both are valid states) or rejects (auth initialization failed or timed out after 6 seconds). This guarantees the arena will always attempt to render, even if auth is broken. The design philosophy is "guest access is default" -- anonymous users should see the full app. This line is a top-level side effect: the moment any module imports arena-core.ts (directly or transitively), the ready promise chain is set up and init will eventually be called. This is the module-level IIFE replacement for the old pre-Vite script tag pattern.

In terms of user-visible behavior, this file orchestrates the following experience. When the user navigates to the arena page, the auth system's ready promise fires, which triggers init. The arena's CSS is injected into the page, the lobby screen appears with options to enter a debate, open the private lobby, visit the mod queue, or browse the power-up shop. If the URL contains a joinCode parameter, the user is silently redirected into a private lobby challenge flow. If the URL contains a spectate parameter with a valid UUID, the user drops directly into a spectator view. If the URL contains shop=1, the power-up shop opens immediately. When the user presses the browser back button at any point during the arena flow (while queueing, viewing a match-found prompt, in a debate room, etc.), the popstate handler fires, cleans up whatever state the user was in (stopping timers, leaving WebRTC calls, dequeuing from matchmaking), and returns them to the lobby. When the user navigates away from the arena entirely, destroy is called, which tears down everything -- WebRTC connections, MediaRecorder sessions, all interval timers, all state -- and removes the popstate listener.

Now for bugs and risks. First, the dead imports: getCurrentUser and getCurrentProfile are imported from auth.ts on line 5 but never used anywhere in arena-core.ts. This is harmless in production (tree-shaking removes them) but confusing for maintainers. Second, the shallow copy in getCurrentDebate means the messages array is shared by reference; a caller that mutates the returned messages array would corrupt state. Third, the _onPopState handler on line 64 calls clearInterval(_rulingCountdownTimer!) with a non-null assertion. If _rulingCountdownTimer happens to be null at that moment (which is possible if the timer was already cleared by another code path), the non-null assertion is a lie, but clearInterval(null) is actually a no-op in all browsers, so this is a stylistic concern rather than a runtime bug. Fourth, the popstate handler on line 80 calls safeRpc('leave_debate_queue') in a fire-and-forget fashion with only a console.warn on failure. If the network is down, the user will appear to still be in the queue on the server side until the server's own timeout kicks them out. This is a known tradeoff documented in the codebase's queue design, but it means the user could accidentally end up matched in a debate they already abandoned. Fifth, the init function creates URLSearchParams(window.location.search) three separate times on lines 106, 110, and 116; this is a minor inefficiency that could be consolidated into a single parse. Sixth, the joinCode handling on line 113 fires void joinWithCode without any error handling or toast notification if the join fails -- the joinWithCode function itself handles errors internally with showToast, so this is acceptable, but it means the user sees no loading indicator while the RPC is in flight. Seventh, the spectate UUID regex on line 117 is a standard UUID v4 pattern but does not enforce UUID version -- it accepts any hex string in 8-4-4-4-12 format, which is fine for validation purposes but technically admits non-v4 UUIDs. This is not a real risk because the server-side RPC will reject invalid debate IDs regardless. Eighth, the auto-init pattern on line 161 calls init() in both the resolve and reject paths of the ready promise. If ready rejects (auth timed out), init will still run, which means renderLobby will execute with getCurrentProfile() and getCurrentUser() potentially returning null. The lobby rendering code in arena-lobby.ts must therefore be null-safe for the auth profile, which it is (it gates auth-requiring actions behind user checks), but this is a fragile contract -- any future lobby code that assumes a profile exists would break on auth-timeout scenarios. Ninth, the destroy function does not call clearQueueTimers, clearMatchAcceptTimers, stopOpponentPoll, stopReferencePoll, or stopModStatusPoll individually -- it relies entirely on resetState to clear those timers. This is correct because resetState does clear all of them, but it means destroy does not call the leave_debate_queue RPC that the popstate handler calls on line 80. If the user navigates away from the arena while in the queue (via the app's navigation system rather than the browser back button), they will not be dequeued on the server. This is a genuine bug: the popstate handler dequeues, but destroy does not. The user would remain in the server-side queue until the server's own stale-queue cleanup runs. Tenth, the window.addEventListener('popstate', _onPopState) on line 90 runs at module load time as a top-level side effect, but destroy removes this listener. If init is called again after destroy (which could happen if the user navigates back to the arena page), the listener will not be re-registered because init does not re-add it. This means the back button will stop working for arena navigation after a destroy/re-init cycle. The listener registration should either be inside init or init should re-add it. This is another genuine bug in the lifecycle management.

In summary, arena-core.ts serves as the lifecycle controller for the arena subsystem. It is responsible for initializing the DOM, injecting CSS, handling deep-link query parameters, managing browser history for back-button support, providing utility functions consumed by all sibling modules, exposing safe accessors for the current state, and orchestrating a thorough teardown of all resources (timers, WebRTC, MediaRecorder, Realtime channels, state variables) when the user leaves the arena. Its most significant side effects are the popstate listener registration at module load time and the auto-init chain on the auth ready promise. The two most actionable bugs are the missing leave_debate_queue call in destroy and the non-re-registration of the popstate listener after a destroy/re-init cycle.
